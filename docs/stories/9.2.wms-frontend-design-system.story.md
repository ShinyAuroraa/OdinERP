# Story 9.2: Design System & Componentes Base

## Status

Done

## Story

**As a** desenvolvedor frontend WMS,
**I want** um conjunto de componentes base reutilizáveis construídos sobre shadcn/ui + Radix UI + Tailwind CSS,
**so that** todas as stories de feature (Waves 2–7) possam ser implementadas consistentemente — sem reescrever DataTable, formulários, modais de confirmação ou estados de loading em cada página.

## Complexity

**Estimativa:** M (Medium) · ~2-3 dias de desenvolvimento
[Fonte: EPIC-FE-001 Wave 1]

## Dependencies

**Pré-requisitos:**
- Story 9.1 Done — `apps/wms-web/` existe com Next.js 15.3.9, TypeScript strict, Tailwind CSS, shadcn/ui base (button, badge, avatar, separator, skeleton, sheet)
- `@tanstack/react-table@^8`, `react-hook-form`, `@hookform/resolvers`, `zod` instalados

**Stories desbloqueadas após esta:**
- Story 10.1 — Gestão de Armazéns (usa DataTable + FormField + PageHeader)
- Todas as stories Wave 2–7 dependem deste design system

## Out of Scope

- ❌ Componentes de domínio WMS específicos (armazéns, produtos, etc.) → Stories Wave 2+
- ❌ Storybook → opcional, fora do MVP
- ❌ Temas dinâmicos / modo escuro → Story 15.1
- ❌ i18n completa (EN) → apenas PT-BR
- ❌ Charts/Recharts → Story 15.1

## Business Value

- **Velocidade de desenvolvimento:** Cada story Wave 2–7 aproveita DataTable, FormField, PageHeader prontos — elimina 30-40% de código repetitivo por página
- **Consistência UI:** StatusBadge com cores padronizadas por status WMS em todas as telas
- **UX defensiva:** ConfirmDialog previne ações destrutivas acidentais em operações WMS (cancelar ordens, deletar localizações)

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| TanStack Table v8 breaking changes de v7 | Baixa | Médio | Usar v8 desde o início; documentar padrão de uso |
| shadcn/ui add interativo em CI | Alta | Baixo | Configurar componentes manualmente (já estabelecido na Story 9.1) |
| Zod schema validation com TypeScript strict | Baixa | Baixo | Usar `z.infer<typeof schema>` para type inference automático |

## Acceptance Criteria

1. **shadcn/ui componentes adicionais instalados** — Além dos já existentes (button, badge, avatar, separator, skeleton, sheet), adicionar manualmente: `input`, `label`, `select`, `checkbox`, `dialog`, `alert-dialog`, `table`, `form`, `card`, `tabs`, `pagination`, `toast` (Sonner), `popover`, `command`, `tooltip`, `dropdown-menu`, `textarea`. Todos em `components/ui/`.

2. **DataTable componente** — `components/shared/DataTable.tsx` usando TanStack Table v8:
   - Props: `columns: ColumnDef<T>[]`, `data: T[]`, `loading?: boolean`, `pagination?: boolean`, `searchable?: boolean`
   - Suporte a: ordenação por coluna, paginação client-side (10/25/50 por página), campo de busca global
   - Estado vazio com mensagem configurável, skeleton loading durante `loading=true`
   - TypeScript genérico `<TData>` estritamente tipado

3. **FormField componente** — `components/shared/FormField.tsx` wrapper para React Hook Form + shadcn/ui:
   - Props: `name`, `label`, `description?`, `required?` + children (input/select/textarea/etc.)
   - Exibe mensagem de erro do React Hook Form automaticamente
   - Integra com `<Form>` do shadcn/ui (react-hook-form context)

4. **PageHeader componente** — `components/shared/PageHeader.tsx`:
   - Props: `title: string`, `description?: string`, `actions?: React.ReactNode`
   - Layout: título H1 + descrição, slot de ações à direita
   - Breadcrumbs automáticos (integra com `Breadcrumbs` existente de Story 9.1)

5. **StatusBadge componente** — `components/shared/StatusBadge.tsx`:
   - Props: `status: string`, `variant?: 'default' | 'outline'`
   - Mapeamento de cores por status WMS:
     - ACTIVE/CONFIRMED/COMPLETED/DONE → verde (success)
     - PENDING/IN_PROGRESS/PICKING → amarelo (warning)
     - INACTIVE/CANCELLED/REJECTED → vermelho (destructive)
     - QUARANTINE/BLOCKED → laranja (warning/destructive)
     - DRAFT/SCHEDULED → cinza (secondary)
   - Fallback: badge cinza para status desconhecidos

6. **ConfirmDialog componente** — `components/shared/ConfirmDialog.tsx`:
   - Props: `open`, `onOpenChange`, `title`, `description`, `onConfirm`, `onCancel?`, `confirmLabel?`, `cancelLabel?`, `variant?: 'default' | 'destructive'`
   - Usa `AlertDialog` do shadcn/ui
   - Variante destructive: botão confirmar vermelho com ícone de alerta

7. **LoadingState componente** — `components/shared/LoadingState.tsx`:
   - Props: `variant?: 'spinner' | 'skeleton' | 'page'`, `rows?: number`
   - `spinner`: ícone Loader2 centralizado com animação
   - `skeleton`: grade de `Skeleton` rows (configurável)
   - `page`: skeleton de página completa (header + tabela)

8. **ErrorBoundary componente** — `components/shared/ErrorBoundary.tsx`:
   - Class component React (ErrorBoundary é class-based por natureza)
   - Props: `fallback?: React.ReactNode`, `onError?: (error: Error) => void`
   - Exibe mensagem de erro amigável com botão "Tentar novamente" (chama `reset()`)
   - `useErrorBoundary()` hook para trigger programático

9. **Barrel exports** — `components/shared/index.ts` exporta todos os 6 componentes shared.

10. **Testes passando** — Suite de testes com ≥ 12 testes:
    - `DataTable.test.tsx`: renderiza dados, paginação, busca, estado vazio, loading
    - `FormField.test.tsx`: renderiza label + erro, integra com React Hook Form
    - `PageHeader.test.tsx`: renderiza título, descrição, slot de ações
    - `StatusBadge.test.tsx`: cores corretas para todos os status WMS
    - `ConfirmDialog.test.tsx`: abre/fecha, chama onConfirm, variante destructive
    - `LoadingState.test.tsx`: renderiza spinner, skeleton, page variant

11. **Linting e TypeScript passando** — `npm run lint` + `npm run typecheck` sem erros após adição de novos componentes.

## Dev Notes

### TanStack Table v8 Pattern

```typescript
// Em DataTable.tsx:
import {
  useReactTable,
  getCoreRowModel,
  getFilteredRowModel,
  getPaginationRowModel,
  getSortedRowModel,
  flexRender,
  type ColumnDef,
} from '@tanstack/react-table'
```

### React Hook Form + Zod Pattern

```typescript
// Uso esperado nas stories de feature:
const schema = z.object({ name: z.string().min(1) })
type FormData = z.infer<typeof schema>
const form = useForm<FormData>({ resolver: zodResolver(schema) })

<Form {...form}>
  <FormField name="name" label="Nome" required>
    <Input {...form.register('name')} />
  </FormField>
</Form>
```

### StatusBadge Color Mapping

```typescript
const STATUS_COLORS = {
  // Green (success)
  ACTIVE: 'success', CONFIRMED: 'success', COMPLETED: 'success', DONE: 'success',
  RECEIVED: 'success', AVAILABLE: 'success', SHIPPED: 'success',
  // Yellow (warning)
  PENDING: 'warning', IN_PROGRESS: 'warning', PICKING: 'warning',
  PICKING_PENDING: 'warning', PROCESSING: 'warning',
  // Red (destructive)
  INACTIVE: 'destructive', CANCELLED: 'destructive', REJECTED: 'destructive',
  FAILED: 'destructive',
  // Orange
  QUARANTINE: 'orange', BLOCKED: 'orange', STOCK_SHORTAGE: 'orange',

}
```
