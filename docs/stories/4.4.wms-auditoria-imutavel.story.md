# Story 4.4: Auditoria Imutável & Log de Movimentações

## Status

Ready for Review

## Story

**As a** administrador de armazém (WMS Admin) e auditor regulatório,
**I want** que todas as movimentações de estoque — recebimento, putaway, quarentena, transferência, ajuste e inventário — sejam registradas em um log imutável (`audit_log`) com imutabilidade garantida por trigger PostgreSQL, indexadas no Elasticsearch para consulta rápida, exportáveis para auditoria regulatória via REST, e com retenção configurável por tenant (compliance LGPD),
**so that** o histórico completo de movimentações seja preservado, auditável e exportável para órgãos regulatórios, com garantia técnica de que nenhum registro pode ser alterado ou excluído após a inserção.

## Complexity

**Estimativa:** M (Medium) · ~2-3 dias de desenvolvimento

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Trigger PostgreSQL bloqueia INSERT se código inválido | Baixa | Alto | Trigger é `BEFORE DELETE OR UPDATE` — não afeta INSERT; testar com Testcontainers |
| Performance de export degradada por grandes ranges de data | Média | Médio | Limitar export a 90 dias por request; índice `idx_audit_log_created_at` já existe |
| Elasticsearch indisponível derruba endpoint `/audit/log` | Baixa | Alto | Fallback para PostgreSQL apenas (ES é opcional para `/audit/log`; indexação é async) |
| `InventoryService` falhando ao salvar `AuditLog` rollback StockMovement | Baixa | Alto | `auditLogRepository.save()` em mesmo `@Transactional` — falha é intencional (atomicidade) |
| Retenção configurável com valor muito baixo (<12 meses) | Baixa | Alto | `CHECK constraint`: `retention_months >= 12`; default 84 (7 anos = LGPD mínimo) |

## Acceptance Criteria

### AC1 — V8 migration: trigger de imutabilidade + tabela de retenção

**Given** o banco de dados PostgreSQL com a tabela `audit_log` existente,
**When** a migration `V8__audit_immutability_retention.sql` é aplicada,
**Then**:
- `CREATE TABLE tenant_retention_config` com colunas: `id UUID PK`, `tenant_id UUID UNIQUE NOT NULL`, `retention_months INT NOT NULL DEFAULT 84`, `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `CHECK (retention_months >= 12)`
- `CREATE OR REPLACE FUNCTION prevent_audit_log_modification()` que executa `RAISE EXCEPTION 'audit_log é imutável — UPDATE/DELETE proibidos'`
- `CREATE TRIGGER trg_audit_log_immutable BEFORE UPDATE OR DELETE ON audit_log FOR EACH ROW EXECUTE FUNCTION prevent_audit_log_modification()`
- Índice `idx_audit_log_tenant_created` em `(tenant_id, created_at DESC)` já existe (V2) — NÃO recriar
- Índice `idx_audit_log_tenant_action` em `(tenant_id, action, created_at DESC)` → **novo**
- `COMMENT ON TABLE tenant_retention_config IS 'Configuração de retenção LGPD por tenant. Default: 84 meses (7 anos).'`

### AC2 — TenantRetentionConfig entity + repository

**Given** a migration V8 aplicada com a tabela `tenant_retention_config`,
**When** o @data-engineer implementa a camada JPA,
**Then**:
- `TenantRetentionConfig` entity: `@SuperBuilder @NoArgsConstructor @Entity @Table("tenant_retention_config")` — estende `BaseEntity` (tem `updatedAt`, retenção pode ser atualizada)
- Campos: `tenantId UUID unique`, `retentionMonths int` (default 84)
- `TenantRetentionConfigRepository` extends `JpaRepository<TenantRetentionConfig, UUID>`
- Método: `Optional<TenantRetentionConfig> findByTenantId(UUID tenantId)`

### AC3 — AuditLogRepository: queries para export

**Given** o `AuditLogRepository` existente com 3 métodos,
**When** são adicionados métodos de suporte ao export regulatório,
**Then** os seguintes métodos adicionais estão presentes:
```java
// Listagem geral por tenant (sem filtro de data — para GET /audit/log sem parâmetros)
Page<AuditLog> findByTenantId(UUID tenantId, Pageable pageable);

// Export por range de data (max 90 dias — validado no service)
Page<AuditLog> findByTenantIdAndCreatedAtBetween(
    UUID tenantId, Instant from, Instant to, Pageable pageable);

// Filtro combinado: tipo de entidade + range de data
Page<AuditLog> findByTenantIdAndEntityTypeAndCreatedAtBetween(
    UUID tenantId, String entityType, Instant from, Instant to, Pageable pageable);

// Contagem por tenant (para verificação de integridade)
long countByTenantId(UUID tenantId);
```

### AC4 — AuditLogDocument + AuditLogEsRepository + AuditLogIndexer

**Given** o padrão de `TraceabilityIndexer` (Story 4.2) para indexação ES assíncrona,
**When** o @data-engineer implementa o indexador de AuditLog,
**Then**:
- `AuditLogDocument` (Elasticsearch document): `@Document(indexName = "wms-audit-log")` com campos: `id`, `tenantId`, `entityType`, `entityId`, `action`, `actorId`, `actorName`, `actorRole`, `correlationId`, `createdAt`
- `AuditLogEsRepository extends ElasticsearchRepository<AuditLogDocument, String>`
- `AuditLogIndexer @Component @Slf4j` com método `@Async void indexAuditLogAsync(AuditLog entry)`:
  - Nunca propaga exceção (degradação graciosa — mesmo padrão de `TraceabilityIndexer`)
  - Loga `WARN` em caso de falha no ES
- `AuditLogIndexer` é chamado em `InventoryService.applyStockAdjustment()` após salvar o `AuditLog`

### AC5 — InventoryService: auditoria de ajustes de inventário

**Given** o `InventoryService.applyStockAdjustment()` da Story 4.3 que cria `StockMovement` mas NÃO chama `auditLogRepository`,
**When** o @data-engineer adiciona a chamada de auditoria,
**Then**:
- `applyStockAdjustment()` chama `auditLogRepository.save(AuditLog.builder()...)` com:
  - `entityType("STOCK_ITEM")`, `entityId(stockItem.getId())`
  - `action(AuditAction.MOVEMENT)`, `actorId(operatorId)` (usando `SYSTEM_OPERATOR_ID` se `operatorId == null`)
  - `newValue(JSON com: movementType=INVENTORY_ADJUSTMENT, quantity, countId)` como String JSON
- `auditLogIndexer.indexAuditLogAsync(savedAuditLog)` chamado após o save
- Tudo dentro da mesma `@Transactional` já existente

### AC6 — GET /audit/log (listagem paginada)

**Given** um usuário autenticado com role `WMS_ADMIN` e `tenant_id` no JWT,
**When** chama `GET /audit/log?entityType=STOCK_ITEM&from=2026-01-01T00:00:00Z&to=2026-02-01T00:00:00Z&page=0&size=20`,
**Then** retorna `200 OK` com:
```json
{
  "content": [
    {
      "id": "uuid",
      "tenantId": "uuid",
      "entityType": "STOCK_ITEM",
      "entityId": "uuid",
      "action": "MOVEMENT",
      "actorId": "uuid",
      "actorName": "string or null",
      "actorRole": "string or null",
      "correlationId": "string or null",
      "createdAt": "2026-01-15T10:30:00Z"
    }
  ],
  "totalElements": 42,
  "totalPages": 3,
  "size": 20,
  "number": 0
}
```
- Filtros disponíveis: `entityType` (optional), `action` (optional), `from` + `to` (ambos obrigatórios se um for informado)
- Range máximo de `from`→`to`: 90 dias — retorna `400 Bad Request` se excedido
- Sem `WMS_ADMIN` role → `403 Forbidden`

### AC7 — GET /audit/log/{id}

**Given** um usuário autenticado com role `WMS_ADMIN`,
**When** chama `GET /audit/log/{id}` com um UUID válido do mesmo tenant,
**Then**:
- Retorna `200 OK` com o `AuditLogEntryResponse` completo (incluindo `oldValue`, `newValue`, `ipAddress`, `userAgent`)
- UUID de outro tenant → `403 Forbidden` (não `404` — não vazar existência)
- UUID inexistente → `404 Not Found`

### AC8 — GET /audit/log/export

**Given** um usuário autenticado com role `WMS_ADMIN`,
**When** chama `GET /audit/log/export?from=2026-01-01T00:00:00Z&to=2026-01-31T23:59:59Z`,
**Then**:
- Retorna `200 OK` com `Content-Type: application/json` e header `Content-Disposition: attachment; filename="audit-log-{tenantId}-{date}.json"`
- Body: array JSON paginado (máx 10.000 registros por export — retorna `400` se range > 30 dias)
- Range `from`→`to` obrigatório (sem default para forçar intencionalidade)
- `from` e `to` no mesmo range de 30 dias máximo; range > 30 dias → `400 Bad Request`

### AC9 — GET /audit/retention-config + PUT /audit/retention-config

**Given** um usuário autenticado com role `WMS_ADMIN`,
**When** chama `GET /audit/retention-config`,
**Then** retorna `200 OK`:
```json
{
  "tenantId": "uuid",
  "retentionMonths": 84,
  "retentionDescription": "7 anos (padrão LGPD)"
}
```
- Se não configurado → retorna default `{ "retentionMonths": 84 }` (não cria registro)

**When** chama `PUT /audit/retention-config` com `{ "retentionMonths": 120 }`,
**Then** retorna `200 OK` com o objeto atualizado.
- `retentionMonths < 12` → `400 Bad Request` com mensagem `"Retenção mínima: 12 meses (compliance LGPD)"`

### AC10 — Imutabilidade verificada por teste de integração

**Given** o trigger `trg_audit_log_immutable` aplicado via V8 migration,
**When** o teste de integração tenta executar `UPDATE audit_log SET action = 'DELETE' WHERE id = ?`,
**Then**:
- A operação lança `DataIntegrityViolationException` (ou subclasse de `JpaSystemException`)
- O teste passa, confirmando que o trigger está ativo

### AC11 — Segurança: apenas WMS_ADMIN acessa /audit/**

**Given** endpoints `GET /audit/log`, `GET /audit/log/{id}`, `GET /audit/log/export`, `GET /audit/retention-config`, `PUT /audit/retention-config`,
**When** chamados com role `WMS_OPERATOR`, `WMS_SUPERVISOR` ou sem autenticação,
**Then** todos retornam `403 Forbidden` (ou `401 Unauthorized` para sem autenticação).

## Dev Notes

### Contexto Arquitetural Crítico

**`audit_log` já existe desde V2** — NÃO recriar a tabela. A V8 migration adiciona apenas:
1. Tabela `tenant_retention_config` (nova)
2. Trigger de imutabilidade
3. Índice adicional `idx_audit_log_tenant_action`

**Padrão `BaseAppendOnlyEntity`** (usado por `AuditLog` e `StockMovement`):
- `@SuperBuilder @NoArgsConstructor` — obrigatório
- `id UUID @GeneratedValue(UUID)`, `tenantId UUID`, `createdAt @CreatedDate` — sem `updatedAt`

**`TenantRetentionConfig`** deve usar `BaseEntity` (não `BaseAppendOnlyEntity`) pois pode ser atualizado.

### AuditLogDocument — Padrão de Getters/Setters Explícitos

**CRÍTICO:** `AuditLogDocument` deve usar **getters e setters explícitos** (sem `@Getter`/`@Setter` Lombok), seguindo o mesmo padrão de `TraceabilityDocument`. Razão: compatibilidade com Spring Data Elasticsearch. Se Lombok for usado, pode funcionar mas quebra a consistência com o padrão estabelecido no projeto.

```java
// CORRETO (seguir TraceabilityDocument):
public String getId() { return id; }
public void setId(String id) { this.id = id; }
// ... demais campos

// ERRADO:
// @Getter @Setter  ← NÃO usar em documentos ES
```

### Padrão de IndexadorES (copiar de TraceabilityIndexer)

```java
@Async
public void indexAuditLogAsync(AuditLog entry) {
    try {
        AuditLogDocument doc = toDocument(entry);
        auditLogEsRepository.save(doc);
        log.debug("AuditLog {} indexado no ES (tenant={})", entry.getId(), entry.getTenantId());
    } catch (Exception e) {
        log.warn("Falha ao indexar AuditLog {} no ES (não-crítico): {}", entry.getId(), e.getMessage());
        // Nunca propaga — ES failure é não-crítica
    }
}
```

### Integração InventoryService (AC5)

`InventoryService.applyStockAdjustment()` usa `SYSTEM_OPERATOR_ID` como fallback:
```java
private static final UUID SYSTEM_OPERATOR_ID = UUID.fromString("00000000-0000-0000-0000-000000000001");
```
O campo `actorId` do `AuditLog` deve usar o `operatorId` do `StockMovement` salvo.

### JSON para new_value (AC5)

Usar JSON simples como String (sem serialização de entidades completas):
```java
String newValue = String.format(
    "{\"movementType\":\"INVENTORY_ADJUSTMENT\",\"quantity\":%d,\"inventoryCountId\":\"%s\"}",
    movement.getQuantity(), movement.getReferenceId()
);
```

### Export (AC8)

Export retorna `List<AuditLogEntryResponse>` serializado como JSON. Não usar streaming/CSV nesta story.
`Content-Disposition: attachment; filename="audit-log-{tenantId}-{yyyyMMdd}.json"`

### Segurança (AC6, AC7, AC8, AC9, AC11)

Todos os endpoints de `/audit/**` requerem role `WMS_ADMIN`. Usar a anotação `@PreAuthorize` já configurada no `SecurityConfig`:
```java
@PreAuthorize("hasRole('WMS_ADMIN')")
```
Verificar que `tenantId` do AuditLog bate com `tenantId` do JWT antes de retornar (AC7).

### Testes — Padrões Estabelecidos

- **Reflexão para setar IDs**: quando necessário setar `id` de `BaseAppendOnlyEntity` em unit tests, usar o padrão:
  ```java
  private void setId(Object entity, UUID id, String className) throws Exception {
      Field field = Class.forName(className).getDeclaredField("id");
      field.setAccessible(true);
      field.set(entity, id);
  }
  // Usar: setId(auditLog, uuid, "com.odin.wms.domain.entity.base.BaseAppendOnlyEntity")
  ```
- **MockBean ES**: `@MockBean AuditLogEsRepository` nos testes de integração para isolar ES
- **Testcontainers**: `@SpringBootTest` + `@AutoConfigureTestDatabase` usa o padrão de `PostgreSQLContainer` dos testes da Story 4.3
