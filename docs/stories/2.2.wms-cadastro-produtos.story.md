# Story 2.2: Cadastro de Produtos no WMS (SKU + Atributos de Armazenagem)

## Status

Ready for Review

## Story

**As a** administrador de armazém (WMS Admin),
**I want** cadastrar e gerenciar produtos no contexto WMS via API REST — com atributos específicos de armazenagem (dimensões, peso, tipo de armazenagem, controle de lote/série/validade, códigos GS1 e capacidade por localização),
**so that** o sistema tenha os dados necessários para executar operações de recebimento, putaway e picking nas Waves 3-5, com rastreabilidade completa por produto, lote e série.

## Business Value

**Por que esta story é prioritária na Wave 2:**

- **Pré-requisito das Waves 3-5:** Todas as operações de recebimento, picking e expedição referenciam `product_id`. Sem produtos cadastrados, nenhuma movimentação de estoque é possível.
- **Rastreabilidade configurável por produto:** Flags `controlsLot`, `controlsSerial` e `controlsExpiry` determinam o rigor de rastreamento — essencial para indústria farmacêutica, alimentícia e eletrônica.
- **Compatibilidade GS1:** Suporte a EAN-13, GS1-128 e QR Code permite integração com scanners de armazém sem transformações adicionais.
- **Capacidade de localização:** `unitsPerLocation` define o limite físico de produtos por prateleira, base para algoritmo de putaway (Story 3.2).

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| SKU duplicado por tenant sem índice único no DB | Baixa | Alto | `ProductWmsRepository.existsByTenantIdAndSku()` já existe; validar antes do INSERT |
| EAN13 duplicado sem método no repository | Média | Médio | Adicionar `existsByTenantIdAndEan13()` ao `ProductWmsRepository` |

## Acceptance Criteria

1. **ProductWms CREATE** — `POST /api/v1/products`:
   - Recebe `CreateProductWmsRequest` com `productId` (UUID), `sku`, `name`, `storageType` obrigatórios
   - Retorna 201 Created com `ProductWmsResponse` completo
   - SKU único por tenant: se duplicado → 422 com mensagem `"SKU '{sku}' already exists for this tenant"`
   - EAN13 único por tenant (se informado): se duplicado → 422 com mensagem `"EAN-13 '{ean13}' already registered"`
   - Role requerida: `WMS_ADMIN`

2. **ProductWms GET BY ID** — `GET /api/v1/products/{id}`:
   - Retorna 200 com `ProductWmsResponse` do tenant corrente
   - Produto de outro tenant → 404 (isolamento multi-tenant)
   - Produto inexistente → 404 `ResourceNotFoundException`
   - Role requerida: `WMS_OPERATOR` ou `WMS_ADMIN`

3. **ProductWms LIST** — `GET /api/v1/products`:
   - Retorna 200 com lista de produtos do tenant corrente
   - Suporta filtro `?storageType=DRY|REFRIGERATED|FROZEN|HAZARDOUS`
   - Suporta filtro `?active=true|false`
   - Lista vazia retorna `[]` (não 404)
   - Role requerida: `WMS_OPERATOR` ou `WMS_ADMIN`

4. **ProductWms UPDATE** — `PUT /api/v1/products/{id}`:
   - Atualiza `name`, `storageType`, `controlsLot`, `controlsSerial`, `controlsExpiry`, dimensões, `unitWeightKg`, `unitsPerLocation`
   - `sku` e `productId` são imutáveis → ignorados no update (não retornar erro, apenas ignorar)
   - `gs1128` e `qrCode` atualizáveis; `ean13` atualizável se não causar duplicata por tenant
   - Retorna 200 com `ProductWmsResponse` atualizado
   - Role requerida: `WMS_ADMIN`

5. **ProductWms DEACTIVATE** — `DELETE /api/v1/products/{id}`:
   - Soft delete: seta `active = false`
   - Retorna 204 No Content
   - Idempotente: produto já inativo → 204 (sem erro)
   - Role requerida: `WMS_ADMIN`

6. **Segurança e isolamento multi-tenant:**
   - Toda operação usa `TenantContextHolder.getTenantId()` para filtrar dados
   - JWT ausente ou inválido → 401
   - Role insuficiente (OPERATOR tentando POST/PUT/DELETE) → 403
   - Tenant A nunca acessa produtos do Tenant B

7. **Testes obrigatórios:**
   - Unit tests em `ProductWmsServiceTest`: create (sucesso), SKU duplicado (BusinessException), EAN duplicado (BusinessException), findById (não encontrado), update (sucesso), deactivate
   - Integration tests em `ProductWmsControllerIntegrationTest`: create 201, duplicate SKU 422, get 200, get cross-tenant 404, list com filtro storageType, deactivate 204
   - `mvn verify` deve passar com 0 falhas

### ProductWmsService — lógica de negócio

```java
// create — validações antes do save
if (repo.existsByTenantIdAndSku(tenantId, request.sku())) {
    throw new BusinessException("SKU '" + request.sku() + "' already exists for this tenant");
}
if (request.ean13() != null && repo.existsByTenantIdAndEan13(tenantId, request.ean13())) {
    throw new BusinessException("EAN-13 '" + request.ean13() + "' already registered");
}

// findById — validação de tenant (padrão da Story 2.1)
var product = repo.findById(id)
    .filter(p -> p.getTenantId().equals(tenantId))
    .orElseThrow(() -> new ResourceNotFoundException("Product not found: " + id));

// deactivate — idempotente (sem erro se já inativo)
product.setActive(false);
repo.save(product);
```
