# Story 3.2: Putaway (Alocação Inteligente de Estoque)

## Status

Done

## Story

**As a** operador de armazém (WMS Operator),
**I want** receber sugestões automáticas de localização de armazenagem para cada `StockItem` criado na doca de recebimento — aplicando FIFO para produtos sem controle de validade e FEFO para perecíveis — e confirmar ou sobrescrever a localização via API,
**so that** cada mercadoria recebida seja alocada na posição correta do armazém com rastreabilidade completa (movimento `PUTAWAY` registrado), liberando a doca de recebimento para novos recebimentos e habilitando o fluxo de picking (Story 5.1).

## Business Value

**Por que esta story é crítica na Wave 3:**

- **Libera a doca de recebimento:** Enquanto StockItems ficam na `RECEIVING_DOCK`, novas entregas não podem ocupar o espaço. O putaway move os itens para o armazenamento definitivo.
- **Rastreabilidade completa:** O `StockMovement(PUTAWAY)` registra a cadeia INBOUND→PUTAWAY, permitindo auditar onde cada lote/série foi alocado desde o momento do recebimento.
- **Otimização de ocupação:** FIFO/FEFO garante que os itens mais antigos (ou de validade mais próxima) sejam alocados com prioridade, evitando itens presos em dock e vencimento de mercadorias.
- **Pré-requisito de picking:** Nenhum picking pode ocorrer sem que os StockItems estejam em locations do tipo `STORAGE` ou `PICKING`.

## Acceptance Criteria

1. Dado uma `ReceivingNote` com status `COMPLETED` ou `COMPLETED_WITH_DIVERGENCE`, ao chamar `POST /api/v1/receiving-notes/{id}/putaway-tasks`, o sistema cria um `PutawayTask` com status `PENDING` para cada `ReceivingNoteItem` cujo `stockItem.location.type == RECEIVING_DOCK`, retornando `201 Created` com a lista de tasks geradas.

2. Para produtos com `product.controlsExpiry = true`, a `suggestedLocation` é determinada pelo algoritmo FEFO: entre as locations disponíveis do warehouse, prefere a que já contém o mesmo produto com o lote de validade mais próxima (`lot.expiryDate ASC`); se não houver, seleciona qualquer location disponível.

3. Para produtos com `product.controlsExpiry = false`, a `suggestedLocation` é determinada pelo algoritmo FIFO: a location é sugerida baseada na disponibilidade de capacidade, priorizando co-localização com mesmo produto; o `stockItem.receivedAt ASC` é registrado como chave do algoritmo aplicado.

4. Ao chamar `POST /api/v1/putaway-tasks/{id}/start`, o status da task muda de `PENDING` para `IN_PROGRESS`; retorna 200 com task atualizada.

5. Ao chamar `POST /api/v1/putaway-tasks/{id}/confirm` com `{ "confirmedLocationId": "<uuid>" }`:
   - O sistema valida que a location pertence ao mesmo warehouse e tenant, tem `type IN (STORAGE, PICKING)` e está `active = true`
   - O sistema valida que a capacidade da location permite o item (`capacityUnits IS NULL OR countOfItemsInLocation < capacityUnits`)
   - O `stockItem.location` é atualizado para `confirmedLocation`
   - Um `StockMovement(type=PUTAWAY, sourceLocation=dockLocation, destinationLocation=confirmedLocation)` é criado
   - Um `AuditLog(action=MOVEMENT, entityType="PutawayTask")` é registrado com o operador e a location de destino
   - O `PutawayTask.status` muda para `CONFIRMED`, `confirmedLocation` é preenchido
   - Retorna 200 com task atualizada

6. Ao chamar `POST /api/v1/putaway-tasks/{id}/cancel`, o status muda para `CANCELLED`, o `StockItem` permanece na doca (nenhum movimento criado); retorna 200. Requer role `WMS_SUPERVISOR` ou `WMS_ADMIN`.

7. Se a `ReceivingNote` não estiver em status `COMPLETED` ou `COMPLETED_WITH_DIVERGENCE`, `POST /api/v1/receiving-notes/{id}/putaway-tasks` retorna `422 Unprocessable Entity` com mensagem de erro descritiva.

8. Se já existirem `PutawayTask`s geradas para a mesma `ReceivingNote`, o endpoint retorna `409 Conflict` (idempotência por `receivingNoteId`).

9. Se a `confirmedLocationId` na request de `confirm` não pertencer ao mesmo warehouse/tenant, não tiver tipo compatível ou não tiver capacidade disponível, o endpoint retorna `422` com mensagem específica.

10. `GET /api/v1/putaway-tasks` lista todas as tasks do tenant com filtros opcionais por `status` e `warehouseId`; `GET /api/v1/putaway-tasks/{id}` retorna uma task pelo ID.

11. Todos os endpoints aplicam isolamento multi-tenant via `TenantContextHolder.getTenantId()`; tentativa de acesso a task de outro tenant retorna `404 Not Found`.

12. Roles exigidas: `generate`, `start`, `confirm` requerem `WMS_OPERATOR`, `WMS_SUPERVISOR` ou `WMS_ADMIN`; `cancel` requer `WMS_SUPERVISOR` ou `WMS_ADMIN`.

### Hierarquia de Location para Warehouse

Para verificar se uma location pertence ao mesmo warehouse que a doca:

// Traversal lazy — cuidado com N+1
location.getShelf().getAisle().getZone().getWarehouse().getId()

Prefira JPQL join para evitar N+1 (ver `LocationRepository.findReceivingDockByIdAndWarehouse` como exemplo em Story 3.1).

### Algoritmo FIFO/FEFO — Estratégia de Implementação

```
suggestLocation(StockItem item, UUID tenantId):
  1. warehouseId = item.location.shelf.aisle.zone.warehouse.id
  2. locations = locationRepository.findStorageOrPickingByWarehouseWithCapacity(warehouseId, tenantId)
     → type IN (STORAGE, PICKING), active=true, mesmo tenant/warehouse
  3. Para cada location, calcular occupancy:
     occupancy = stockItemRepository.countByTenantIdAndLocationId(tenantId, location.id)
     available = (location.capacityUnits == null) ? true : occupancy < location.capacityUnits
  4. Filter: available == true
  5. if FEFO:
       preferredLocations = locations onde já existe StockItem do mesmo produto,
                            ordenados por MIN(lot.expiryDate) ASC
       return preferredLocations.first() OR any available location
  6. if FIFO:
       preferredLocations = locations onde já existe StockItem do mesmo produto (consolidação)
       return preferredLocations.first() OR any available location
  7. if empty: throw BusinessException("Nenhuma location STORAGE disponível no warehouse")
```

**ATENÇÃO N+1**: A query de `countByTenantIdAndLocationId` para cada location pode gerar N queries. Alternativa: buscar todos os counts em uma query só (GROUP BY location_id) — implementação sugerida:
```java
@Query("SELECT s.location.id, COUNT(s) FROM StockItem s WHERE s.tenantId = :tenantId GROUP BY s.location.id")
List<Object[]> countByTenantIdGroupByLocationId(@Param("tenantId") UUID tenantId);
```

### Validações de `confirm()` — Checklist

```
1. task existe e pertence ao tenant → 404 se não
2. task.status IN (PENDING, IN_PROGRESS) → 422 se CONFIRMED/CANCELLED
3. confirmedLocation existe e pertence ao tenant → 404 se não
4. confirmedLocation.type IN (STORAGE, PICKING) → 422 se dock, quarantine, etc.
5. confirmedLocation.active == true → 422 se inativa
6. confirmedLocation.shelf.aisle.zone.warehouse == task.sourceLocation.shelf.aisle.zone.warehouse → 422 se cross-warehouse
7. capacityUnits: se não null → count < capacityUnits → 422 se cheio
```
