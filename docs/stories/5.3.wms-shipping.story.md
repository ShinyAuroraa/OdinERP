# Story 5.3: Shipping — Expedição & Despacho de Volumes

## Status

Draft

## Story

**As a** supervisor de expedição (WMS Supervisor),
**I want** processar as ordens de packing concluídas na estação de expedição — confirmando o carregamento item a item via scanner, registrando dados da transportadora e veículo, gerando o manifesto de carga e despachando os volumes —
**so that** cada envio seja rastreável do armazém até a entrega, integrado ao CRM e SCM via evento Kafka de conclusão (`wms.shipping.completed`), garantindo a cadeia outbound completa com registro imutável de todas as saídas de estoque.

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| `wms.packing.completed` duplicado (retry Kafka) | Alta | Médio | `UNIQUE (tenant_id, packing_order_id)` + `DataIntegrityViolationException` catch |
| Despacho sem todos os itens carregados | Média | Alto | AC8 valida `shipping_items.loaded = true` para todos antes de `DISPATCHED` → 422 |
| Despacho sem dados de transportadora | Média | Médio | AC8 valida `carrier_name NOT NULL` antes de `DISPATCHED` → 422 |
| Concorrência: dois operadores carregam o mesmo item | Baixa | Médio | `@Version` em `ShippingItem` → HTTP 409 Conflict |
| StockMovement OUTBOUND com referência incorreta | Baixa | Alto | AC8 cria `StockMovement` com `referenceType=SHIPPING_ORDER` e `referenceId=shippingOrderId` |
| PackingCompletedEvent sem itens (packing vazio) | Baixa | Baixo | Consumer verifica `event.items().isEmpty()` → loga warning, cria ShippingOrder sem itens |

### AC3 — Kafka Consumer: `wms.packing.completed` → criação de ShippingOrder PENDING

**Given** o tópico Kafka `wms.packing.completed` recebe um `PackingCompletedEvent` (record já existente em `messaging/event/PackingCompletedEvent.java`) com:
```json
{
  "eventType": "PACKING_ORDER_COMPLETED",
  "tenantId": "uuid-tenant",
  "packingOrderId": "uuid-packing-order",
  "pickingOrderId": "uuid-picking-order",
  "crmOrderId": "uuid-crm-order",
  "warehouseId": "uuid-warehouse",
  "operatorId": "uuid-operator",
  "items": [
    {
      "packingItemId": "uuid-packing-item",
      "productId": "uuid-product",
      "lotId": "uuid-lot-or-null",
      "locationId": "uuid-location",
      "quantityPacked": 3
    }
  ],
  "completedAt": "2026-02-23T10:00:00Z"
}
```
**When** `ShippingEventConsumer.handlePackingCompleted()` processa o evento,
**Then**:
- Verifica idempotência: `shippingOrderRepository.findByTenantIdAndPackingOrderId(tenantId, packingOrderId)` — se já existe, loga e retorna sem criar duplicata
- Cria `ShippingOrder` com:
  - `status = PENDING`
  - `packingOrderId`, `pickingOrderId`, `crmOrderId`, `warehouseId` preenchidos a partir do evento
  - `tenantId` do evento
- Para cada item do evento, cria `ShippingItem` com:
  - `packingItemId` = `item.packingItemId()`
  - `productId` = `item.productId()`
  - `lotId` = `item.lotId()` *(nullable)*
  - `quantityShipped` = `item.quantityPacked()`
  - `loaded = false`
- Consumer usa `@KafkaListener(topics = "${wms.kafka.topics.packing-completed}", groupId = "wms-shipping-consumer", containerFactory = "shippingKafkaListenerContainerFactory")`
- Bean `shippingKafkaErrorHandler` configurado em `KafkaConfig.java` (mesmo padrão de `packingKafkaErrorHandler`)
- Erros logados; consumer **não propaga exceções** para não bloquear o offset Kafka
- Configurar `wms.kafka.topics.shipping-completed` em `application.yml`

### AC5 — `PUT /shipping-orders/{id}/load-item/{itemId}` — confirmar carregamento de item (scanner)

**Given** um usuário autenticado com role `WMS_OPERATOR` ou superior,
**When** chama `PUT /shipping-orders/{id}/load-item/{itemId}` com:
```json
{ "barcode": "SKU-PRODUTO-001" }
```
**Then**:
- ShippingOrder em status `IN_PROGRESS` → processada; outros status → `409 Conflict`
- ShippingItem pertencente a outra ShippingOrder ou tenant → `404 Not Found`
- Validação de barcode (case-insensitive):
  - Compara `barcode` com `productWms.sku` (via `productId` do item) **OU** com `lot.lotNumber` (via `lotId` do item, se `lotId != null`)
  - Se não coincide → `400 Bad Request` com `"Barcode {barcode} não corresponde ao produto ou lote esperado para o item {itemId}"`
- Item já com `loaded = true` → `400 Bad Request` com `"Item {itemId} já foi carregado"`
- `loaded = true`, `loadedAt = Instant.now()`, `loadedBy` = UUID do usuário autenticado
- Retorna `200 OK` com `ShippingItemResponse` atualizado

### AC8 — `PUT /shipping-orders/{id}/dispatch` — despachar envio (SUPERVISOR/ADMIN)

**Given** um usuário autenticado com role `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `PUT /shipping-orders/{id}/dispatch`,
**Then**:
- ShippingOrder em status `IN_PROGRESS` → processada; outros status → `409 Conflict`
- ShippingOrder de outro tenant → `403 Forbidden`
- **Validação — todos itens carregados**: verifica `shippingItems.stream().allMatch(i -> i.isLoaded())` → se algum item com `loaded = false` → `422 Unprocessable Entity` com `"Existem itens não carregados: [itemId1, itemId2]"`
- **Validação — dados de transportadora**: verifica `carrier_name != null` → se nulo → `422 Unprocessable Entity` com `"Dados da transportadora não informados. Chame set-carrier-details antes de despachar."`
- Dentro de `@Transactional`:
  - `status` → `DISPATCHED`, `dispatched_at = Instant.now()`
  - Cria `AuditLog` via `auditLogRepository.save()` com `action = "SHIPPING_DISPATCHED"`, `entityType = "SHIPPING_ORDER"`, `entityId = shippingOrderId`, `tenantId`, `userId` do operador autenticado
  - Chama `auditLogIndexer.indexAuditLogAsync(savedLog)` (mesma chamada que demais services)
  - Para cada `ShippingItem`, cria `StockMovement` com:
    - `type = MovementType.OUTBOUND`
    - `referenceType = ReferenceType.SHIPPING_ORDER`
    - `referenceId = shippingOrderId`
    - `productId`, `lotId`, `quantity = quantityShipped`, `tenantId`, `warehouseId`
  - Publica `ShippingCompletedEvent` via `shippingCompletedEventPublisher.publishShippingCompleted(event)` (degradação graciosa)
- Retorna `200 OK` com `ShippingOrderResponse` (status = DISPATCHED, dispatched_at preenchido)
- `ShippingCompletedEvent` publicado no tópico `wms.shipping.completed`:
  ```json
  {
    "eventType": "SHIPPING_ORDER_DISPATCHED",
    "tenantId": "uuid",

    "trackingNumber": "string-or-null",
  ```

## Dev Notes

### Contexto de Implementação

**Padrão de Consumer Kafka (seguir EXATAMENTE como PackingEventConsumer da Story 5.2):**
- `ShippingEventConsumer` em `messaging/consumer/ShippingEventConsumer.java`
- Anotação: `@KafkaListener(topics = "${wms.kafka.topics.packing-completed}", groupId = "wms-shipping-consumer", containerFactory = "shippingKafkaListenerContainerFactory")`
- `KafkaConfig.java` — adicionar `shippingOrderConsumerFactory` + `shippingKafkaListenerContainerFactory` usando `PackingCompletedEvent.class` como tipo de valor (mesmo consumidor do packing)
- Idempotência: `findByTenantIdAndPackingOrderId()` antes de qualquer save

**PackingCompletedEvent já existe** em `messaging/event/PackingCompletedEvent.java` — **não criar duplicata**. O consumer de shipping simplesmente reutiliza o mesmo record.

**Manifesto JSON — usar Jackson ObjectMapper:**
- `objectMapper.writeValueAsString(manifestDto)` para serializar
- `objectMapper.readValue(manifestJson, ShippingManifestDto.class)` para desserializar (se necessário)
- `ObjectMapper` é injetável via `@Autowired` (já configurado no contexto Spring)

**StockMovement OUTBOUND no dispatch:**
- Verificar que `MovementType.OUTBOUND` existe no enum (introduzido em Story 4.1 ou 4.5)
- Um `StockMovement` por `ShippingItem` (não uma única movimentação agregada)
- `quantity` é positivo mesmo para OUTBOUND (a direção é determinada pelo `MovementType`)

**ShippingItem.@Builder.Default:**
- `@Builder.Default boolean loaded = false` — obrigatório com `@SuperBuilder`
