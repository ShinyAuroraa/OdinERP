# Story 1.2: WMS Schema & Domain Model

## Status

Ready for Review

## Story

**As a** engenheiro de dados,
**I want** criar o schema PostgreSQL completo do WMS com todas as tabelas de domínio, entidades JPA e repositórios Spring Data,
**so that** as stories operacionais (Wave 2+) tenham uma base de dados sólida, rastreável, multi-tenant e performática para implementar toda a lógica de negócio do WMS.

## Complexity

**Estimativa:** L (Large) · ~8-13 dias de desenvolvimento
[Fonte: EPIC-WMS-001 Story Summary]

## Dependencies

**Pré-requisitos para iniciar esta story:**
- Story 1.1 concluída — projeto Spring Boot 3 com Flyway habilitado e `V1__init.sql` de baseline já existente
- PostgreSQL 17 configurado e acessível (instância wms-service)
- Testcontainers configurado (AbstractIntegrationTest.java disponível)

**Stories desbloqueadas após esta:**

- Story 2.2 — Cadastro de Produtos WMS (usa `products_wms`)
- Story 4.4 — Auditoria Imutável (usa `audit_log` e `stock_movements`)

## Out of Scope

- ❌ Lógica de negócio (services, controllers) → Wave 2+

## Business Value

**Por que esta story é crítica para o negócio:**

- **Fundação bloqueante:** Nenhuma das 22 stories operacionais restantes do EPIC-WMS-001 pode ser iniciada sem o schema de dados definido. Esta story desbloqueia Wave 2 inteira.
- **Integridade dos dados:** Schema bem projetado com constraints (FK, CHECK, UNIQUE por tenant) garante que dados inválidos jamais entrem no banco, reduzindo custos de correção em produção.
- **Multi-tenant desde o início:** `tenant_id` em todas as tabelas e constraints de isolamento garantem que dados de diferentes clientes nunca se misturem — requisito não-negociável para um ERP SaaS.
- **Rastreabilidade nativa:** `stock_movements` e `audit_log` append-only integrados ao schema desde o início eliminam a necessidade de retrofitting futuro — garantia de compliance (LGPD, Receita Federal, ANVISA).
- **FIFO/FEFO by design:** `received_at` em `stock_items` e `expiry_date` em `lots` garantem que o algoritmo de separação respeite as regras operacionais sem lógica adicional de compensação.

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Schema muito grande dificulta migrations futuras | Média | Médio | Criar migration única coesa para o schema base; stories futuras criam V3+, V4+, etc. |
| Otimistic lock collision em stock_items em testes | Baixa | Baixo | Usar @Version do JPA; testes com dados isolados |
| UUID performance no PostgreSQL | Baixa | Médio | Usar `gen_random_uuid()` nativo do PG17; índices B-tree adequados |
| FK constraints impedirem testes de integração | Baixa | Baixo | `@DirtiesContext` ou dados de teste respeitando ordem de inserção |
| Tabelas sem `updated_at` (append-only) confundirem @dev | Baixa | Baixo | Documentar claramente `stock_movements` e `audit_log` como append-only |

## Acceptance Criteria

1. **Migration Flyway V2 criada** — Arquivo `V2__wms_schema.sql` criado em `src/main/resources/db/migration/` com todas as tabelas definidas abaixo, sem erros de sintaxe PostgreSQL 17.

2. **11 tabelas implementadas** — As seguintes tabelas existem após executar `mvn flyway:migrate`:
   - `warehouses`, `zones`, `aisles`, `shelves`, `locations`
   - `products_wms`, `lots`, `serial_numbers`
   - `stock_items`, `stock_movements`, `audit_log`

3. **Multi-tenant em todas as tabelas** — Todas as 11 tabelas possuem coluna `tenant_id UUID NOT NULL`. Constraint `UNIQUE (tenant_id, ...)` aplicada nos campos de código/identificação natural.

4. **FK constraints e CHECK constraints** — Relacionamentos entre tabelas declarados com `REFERENCES` + `ON DELETE RESTRICT`. `stock_items` com `CHECK (quantity_available >= 0 AND quantity_reserved >= 0 AND quantity_quarantine >= 0 AND quantity_damaged >= 0)`.

5. **Índices de performance criados** — Mínimo: índice em `tenant_id` em todas as tabelas; índices compostos em campos de consulta frequente: `(tenant_id, code)` em `locations`, `(tenant_id, sku)` em `products_wms`, `(tenant_id, product_id)` em `stock_items`, `(expiry_date)` em `lots` (para FEFO), `(created_at DESC)` em `stock_movements` e `audit_log`.

6. **Suporte FIFO/FEFO** — `stock_items` possui `received_at TIMESTAMPTZ NOT NULL DEFAULT now()` para FIFO. `lots` possui `expiry_date DATE` para FEFO. `serial_numbers` referencia `lot_id` opcionalmente.

7. **Campos GS1** — `products_wms` contém: `ean13 VARCHAR(13)`, `gs1_128 VARCHAR(255)`, `qr_code TEXT`.

8. **Append-only para auditoria** — `stock_movements` e `audit_log` **não possuem** coluna `updated_at` (imutáveis por design). `audit_log` não é referenciado por FK de outras tabelas (apenas registra fatos).

9. **Optimistic locking** — `stock_items` possui `version BIGINT NOT NULL DEFAULT 0` mapeado com `@Version` no JPA para controle de concorrência sem lock pessimista.

10. **Entidades JPA + Repositórios** — Para cada tabela: uma `@Entity` Java com mapeamento correto + uma `@Repository` Spring Data JPA. `BaseEntity` abstrata com `id`, `tenantId`, `createdAt`, `updatedAt` via `@EntityListeners(AuditingEntityListener.class)`. `StockMovement` e `AuditLog` estendem `BaseAppendOnlyEntity` (apenas `id`, `tenantId`, `createdAt`).

11. **Testes de migração e repositórios** — `SchemaIntegrationTest.java` com Testcontainers verificando: migration executa sem erros, todas as tabelas existem, inserção básica em todas as tabelas funciona, FK constraints estão ativas, restrição de `quantity >= 0` é aplicada.

### Schema Overview — Hierarquia de Endereçamento

```
Warehouse (armazém)
  └── Zone (zona: STORAGE, QUARANTINE, RECEIVING_DOCK, etc.)
        └── Aisle (corredor)
              └── Shelf (prateleira / nível)
                    └── Location (posição — leaf node com endereço completo)
```

### Schema Overview — Estoque

```
ProductWms (produto no WMS com atributos de armazenagem + GS1)
  └── Lot (lote: lot_number + expiry_date para FEFO)
        └── SerialNumber (número de série individual)
StockItem (saldo: location + product + lot → quantities + received_at para FIFO)
StockMovement (append-only: registro de todo movimento de estoque)
AuditLog (append-only: registro de toda ação no sistema)
```

### Padrão BaseEntity

```java
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @Column(name = "tenant_id", nullable = false, updatable = false)
    private UUID tenantId;

    @CreatedDate
    @Column(name = "created_at", nullable = false, updatable = false)
    private Instant createdAt;

    @LastModifiedDate
    @Column(name = "updated_at", nullable = false)
    private Instant updatedAt;
}
```

### Padrão BaseAppendOnlyEntity (StockMovement, AuditLog)

```java
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseAppendOnlyEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @Column(name = "tenant_id", nullable = false, updatable = false)
    private UUID tenantId;

    @CreatedDate
    @Column(name = "created_at", nullable = false, updatable = false)

}
```

### Padrão StockItem (Optimistic Lock + FIFO)

```java
@Entity
@Table(name = "stock_items")
public class StockItem extends BaseEntity {

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "location_id")
    private Location location;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "product_id")
    private ProductWms product;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "lot_id")
    private Lot lot; // nullable — produtos sem controle de lote

    private int quantityAvailable;
    private int quantityReserved;
    private int quantityQuarantine;
    private int quantityDamaged;

    @Column(name = "received_at", nullable = false, updatable = false)
    private Instant receivedAt = Instant.now(); // FIFO key

    @Version
    private Long version; // Optimistic locking
}
```

### Observações Técnicas Importantes

1. **UUID como PK:** PostgreSQL 17 tem boa performance com UUID v4 via `gen_random_uuid()`. Usar `@GeneratedValue(strategy = GenerationType.UUID)` no JPA.
2. **`@EnableJpaAuditing`** já está em `WmsServiceApplication.java` (feito na Story 1.1) — `@CreatedDate` e `@LastModifiedDate` funcionam out-of-the-box.
3. **Enums no JPA:** Usar `@Enumerated(EnumType.STRING)` em todos os campos de enum para legibilidade no banco.
4. **FetchType:** Usar `LAZY` em todos os `@ManyToOne` para evitar N+1. Stories que precisarem de JOIN devem usar `@EntityGraph` ou `JOIN FETCH` explícito.
5. **`audit_log.ip_address`** é do tipo `INET` no PostgreSQL — mapear como `String` no Java (não há suporte nativo JPA para INET; usar `@Column(columnDefinition = "INET")`).
6. **`audit_log.old_value` e `new_value`** são `JSONB` — mapear como `String` no Java (ou usar um `@Convert` customizado). Stories futuras podem adicionar tipo mais específico.
