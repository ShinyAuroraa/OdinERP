# Story 8.1: App Android WMS — Fundação (Kotlin + Jetpack Compose)

## Status

Done

## Story

**As a** operador de armazém,
**I want** um aplicativo Android nativo que me permita autenticar com meu usuário Keycloak e acessar as operações do WMS através de um painel centralizado com leitor de código de barras integrado,
**so that** eu possa executar todas as operações de armazém (recebimento, picking, inventário, transferências) diretamente no dispositivo móvel com scanner, sem depender de papeladas ou terminais fixos.

## Complexity

**Estimativa:** L (Large) · ~4-5 dias de desenvolvimento

[Fonte: EPIC-WMS-001 — Story 8.1 é a fundação da Wave 8. Cria o projeto Android do zero: Kotlin + Jetpack Compose + MVVM + Clean Architecture, configuração de autenticação Keycloak OAuth2 PKCE, integração com APIs WMS (Retrofit + OkHttp), leitor de câmera (ML Kit), suporte offline base (Room DB), navegação Compose Navigation. As telas desta story são: Login, Dashboard do Operador, Menu de Operações. Operações específicas (recebimento, picking, etc.) vêm nas stories 8.2–8.4. Requer novo módulo: `apps/wms-android/`.]

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| OAuth2 PKCE no Android com Keycloak: redirect URI e browser tabs customizadas | Média | Alto | Usar `AppAuth-Android` (biblioteca referência do OAuth2 para Android) em vez de implementação manual |
| ML Kit scanner: permissão de câmera negada / dispositivos sem câmera traseira adequada | Baixa | Médio | Graceful fallback para entrada manual de código; testar em emulador com câmera virtual |
| Conectividade intermitente em armazéns (WiFi morto, zonas cegas) | Alta | Alto | Room DB para cache local + WorkManager para sync; operações críticas funcionam offline |
| Compose + MVVM boilerplate setup demorado para ambiente novo | Média | Baixo | Usar template Android com Hilt + Navigation já configurados; não customizar build tool |

### AC2 — Autenticação Keycloak OAuth2 PKCE

**Given** usuário com credenciais válidas no tenant Keycloak,
**When** abre o app e insere login/senha na tela de Login,
**Then**:
- Fluxo OAuth2 PKCE executado via `AppAuth-Android`
- Token JWT armazenado de forma segura (`EncryptedSharedPreferences` / Android Keystore)
- Claims extraídas do JWT: `tenant_id`, `sub`, `realm_access.roles`
- Token refresh automático quando expirado (sem pedir login novamente)
- Logout limpa token e volta para tela de Login
- Credenciais inválidas → mensagem de erro clara (não expõe detalhes do servidor)

---

### AC3 — Tela de Login

**Given** app aberto sem sessão ativa,
**When** o usuário vê a tela de Login,
**Then**:
- Campo de usuário, campo de senha (toggle visibilidade), botão "Entrar"
- Logo Odin ERP + "WMS Operações" como título
- Loading indicator durante autenticação
- Erro de rede → mensagem "Sem conexão — verifique o WiFi"
- Erro de credenciais → mensagem "Usuário ou senha inválidos"
- Após login bem-sucedido → navega para Dashboard

---

### AC4 — Dashboard do Operador

**Given** usuário autenticado com role `WMS_OPERATOR` (ou superior),
**When** acessa o Dashboard,
**Then**:
- Exibe: nome do usuário, role, nome do armazém padrão do tenant
- Card de resumo de estoque: total de itens disponíveis (via `GET /api/v1/stock/summary`)
- Cards de acesso rápido: Recebimento, Picking, Inventário, Transferências (com ícones)
- Cada card mostra badge com quantidade de tarefas pendentes (ex: "3 ordens de picking")
- Pull-to-refresh atualiza os dados
- Dados cacheados no Room DB → exibidos offline com indicador "Dados offline - atualizado em {time}"

---

### AC5 — Menu de Operações e Navegação

**Given** usuário no Dashboard,
**When** navega pelo app,
**Then**:
- Compose Navigation como sistema de navegação
- Bottom navigation bar: Dashboard, Operações, Scanner, Perfil
- Tela "Operações": lista de operações disponíveis com ícone, título e descrição
- Operações visíveis dependem da role: `WMS_OPERATOR` vê Picking/Recebimento/Inventário/Transferência; `WMS_SUPERVISOR` vê todas + Relatórios
- Tela "Perfil": nome, role, tenant, botão Logout, versão do app
- Back navigation consistente (sem sair do app acidentalmente)

---

### AC6 — Leitor de Barcode/QR Code (ML Kit)

**Given** usuário em qualquer tela de operação que exige scan,
**When** aciona o leitor,
**Then**:
- Câmera abre com overlay de enquadramento
- Formatos suportados: EAN-13, EAN-8, Code-128 (GS1-128), QR Code, Data Matrix, SSCC
- Decodificação em tempo real via `ML Kit Barcode Scanning` (on-device, sem internet)
- Vibração + som de confirmação ao ler código válido
- Resultado retornado via callback/Flow para a tela chamadora
- Permissão de câmera não concedida → dialog explicativo + botão "Conceder Permissão"
- Fallback: campo de entrada manual do código ativável por botão no overlay

---

### AC7 — Suporte Offline Base

**Given** dispositivo sem conectividade,
**When** usuário abre o app,
**Then**:
- App inicia normalmente (token ainda válido no Keystore)
- Dashboard exibe dados cacheados no Room DB com banner amarelo "Modo Offline"
- Operações de leitura (saldo, listas de picking) funcionam com dados locais
- Operações de escrita (confirmar picking, etc.) enfileiradas no Room DB com status `PENDING`
- WorkManager sincroniza automaticamente ao recuperar conexão
- Conflito de sync → log de erro; operação marcada como `SYNC_FAILED` visível no Perfil

---

### AC8 — Segurança & Multi-tenant

**Given** qualquer requisição HTTP do app,
**Then**:
- Header `Authorization: Bearer {jwt}` incluído em todas as chamadas via OkHttp Interceptor
- `tenant_id` extraído do JWT e incluído nas queries Room (sem misturar dados de tenants)
- Token expirado → refresh automático transparente; se refresh falhar → logout forçado
- Certificate Pinning configurado para o domínio da API WMS (evitar MITM)
- ProGuard/R8 habilitado no build `release` (ofuscação de bytecode)
- Nenhuma credencial hardcoded (URLs via `BuildConfig` injetadas no build)

### Fluxo OAuth2 PKCE com AppAuth

```kotlin
// AuthRepository.kt
class AuthRepository(context: Context) {
    private val authService = AuthorizationService(context)

    fun buildAuthRequest(): AuthorizationRequest {
        val config = AuthorizationServiceConfiguration(
            Uri.parse("${BuildConfig.KEYCLOAK_URL}/realms/${BuildConfig.KEYCLOAK_REALM}/protocol/openid-connect/auth"),
            Uri.parse("${BuildConfig.KEYCLOAK_URL}/realms/${BuildConfig.KEYCLOAK_REALM}/protocol/openid-connect/token")
        )
        return AuthorizationRequest.Builder(
            config,
            BuildConfig.KEYCLOAK_CLIENT_ID,
            ResponseTypeValues.CODE,
            Uri.parse("com.odin.wms.android:/oauth2redirect")
        ).setScope("openid profile").build()
    }
}
```

### OkHttp Interceptor para Bearer Token

```kotlin
class AuthInterceptor(private val tokenProvider: TokenProvider) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val token = tokenProvider.getAccessToken()
        val request = chain.request().newBuilder()
            .addHeader("Authorization", "Bearer $token")
            .build()
        return chain.proceed(request)
    }
}
```

#### Issues Encontradas

| ID | Severidade | Arquivo | Descrição |
|----|-----------|---------|-----------|
| C1 | **HIGH** | `di/NetworkModule.kt` | Dois `@Provides @Singleton` retornando `OkHttpClient` sem `@Qualifier` → Hilt falha na compilação com "OkHttpClient is bound multiple times". `@JvmSuppressWildcards` **NÃO é** um qualifier Dagger. Fix: criar `@Qualifier @Retention(BINARY) annotation class BaseClient` e `AuthenticatedClient`; anotar os providers e injection points correspondentes. |
| C2 | MEDIUM | `ui/dashboard/DashboardScreen.kt` | AC4 exige exibição de nome do usuário, role e armazém no Dashboard. `DashboardScreen` não exibe essas informações — apenas estoque e cards de operações. Opções: (a) `DashboardViewModel` expõe `currentUser`, (b) informações exibidas em header acima do LazyColumn. |
| C3 | MEDIUM | `di/NetworkModule.kt:39` | Certificate Pinning configura pin placeholder `"sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA="`. Qualquer build `release` falhará em todas as conexões HTTPS até o pin real ser configurado. Adicionar comentário `// TODO: substituir pelo pin real — openssl x509...` e/ou criar test build type sem pinning. |
| C4 | MEDIUM | `data/remote/AuthInterceptor.kt` | Interceptor injeta o token proativamente mas não faz retry com token renovado quando recebe HTTP 401. Se o token expirar entre a verificação e a chamada (race condition) ou for invalidado pelo Keycloak, o usuário recebe erro genérico em vez de um refresh transparente. Fix: adicionar `Authenticator` (OkHttp) ou lógica de retry no interceptor. |
| C5 | MEDIUM | `ui/scanner/BarcodeAnalyzer.kt` | `BarcodeScanning.getClient()` retorna um `BarcodeScanner` que deve ser fechado (`scanner.close()`) quando o `ImageAnalysis` é destruído. `BarcodeAnalyzer` não expõe `close()` nem registra no lifecycle da câmera. Fix: implementar `AutoCloseable` ou expor `fun close()` e chamar em `DisposableEffect` na composable. |
| C6 | LOW | `security/TokenProviderTest.kt` | Testes mockam a própria classe sob teste — o `answers {}` replica a lógica real da classe, testando efetivamente a configuração do mock e não o comportamento real. Limitação conhecida (requer Android Context). Aceitável para fundação; cobertura real via testes instrumentados em stories futuras. |
| C7 | LOW | `di/NetworkModule.kt:30` | `HttpLoggingInterceptor.Level.BODY` em debug loga tokens JWT completos nos logs do sistema (adb logcat). Aceitável para desenvolvimento, mas em ambientes de QA com dispositivos físicos pode expor credenciais. Considerar `Level.HEADERS` ou redaction do header Authorization. |
