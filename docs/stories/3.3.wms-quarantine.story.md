# Story 3.3: Controle de Qualidade e Quarentena

## Status

Ready for Review

## Story

**As a** supervisor de armazém (WMS Supervisor),
**I want** receber um fluxo de trabalho estruturado para os `ReceivingNoteItem`s marcados como `FLAGGED` em notas de recebimento com status `COMPLETED_WITH_DIVERGENCE` — com movimentação automática para zona de quarentena, revisão de qualidade, e decisão de destino (liberação para estoque, devolução ao fornecedor ou descarte) — rastreando cada etapa via `QuarantineTask`,
**so that** itens suspeitos nunca entrem no estoque operacional sem validação de qualidade, haja rastreabilidade completa do fluxo de quarentena (movimentos `QUARANTINE_IN` e `QUARANTINE_OUT` registrados), e o SCM seja notificado via Kafka quando há devolução ao fornecedor — garantindo integridade do inventário e compliance com processos de QC.

## Complexity

**Estimativa:** M (Medium) · ~3-4 dias de desenvolvimento

## Business Value

**Por que esta story é crítica na Wave 3:**

- **Completa o ciclo inbound:** Sem a quarentena, itens FLAGGED ficam presos na doca de recebimento indefinidamente, bloqueando novos recebimentos.
- **Integridade do inventário:** Nenhum item com qualidade duvidosa deve entrar no saldo operacional (STORAGE) sem aprovação de supervisor.
- **Rastreabilidade de devolução:** O `StockMovement(QUARANTINE_OUT)` + evento Kafka cria uma trilha auditável de devolução ao fornecedor, essencial para disputas e relatórios regulatórios.
- **Compliance regulatório:** Em setores como alimentos, farmácia e eletrônicos, o fluxo de QC e quarentena é legalmente obrigatório.
- **Desbloqueio da Wave 4:** Story 4.1 (Estoque em Tempo Real) precisa que todos os itens inbound tenham destino definido — seja STORAGE (via putaway) ou QUARANTINE (via esta story).

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Nenhuma location QUARANTINE no warehouse | Média | Alto | Retornar 422 com mensagem clara — operador deve cadastrar ao menos 1 location QUARANTINE antes de gerar tasks |
| Kafka producer não configurado no wms-service | Alta | Médio | Adicionar spring-kafka + KafkaTemplate; tornar producer mockável; lançar evento assíncrono com tratamento de erro |
| RELEASE_TO_STOCK sem location STORAGE disponível | Baixa | Alto | Replicar lógica de capacidade de PutawayService; 422 com mensagem clara se warehouse lotado |
| Supervisor decide task já decidida (concorrência) | Baixa | Médio | `@Version` na entidade + verificação de status `IN_REVIEW` antes de decidir; 409 se não IN_REVIEW |
| Geração duplicada para mesma nota (idempotência) | Baixa | Médio | `existsByTenantIdAndReceivingNoteId` → 409 se já gerado |
| Item FLAGGED sem StockItem associado | Baixa | Médio | Se `item.stockItem == null`, ignorar na geração (mesmo tratamento de Story 3.2) |

## Acceptance Criteria

1. Dado uma `ReceivingNote` com status `COMPLETED_WITH_DIVERGENCE`, ao chamar `POST /api/v1/receiving-notes/{id}/quarantine-tasks`, o sistema cria um `QuarantineTask` com status `PENDING` para cada `ReceivingNoteItem` cujo `status == FLAGGED` e `stockItem != null`. Para cada task criada, cria também um `StockMovement(type=QUARANTINE_IN, sourceLocation=dockLocation, destinationLocation=quarantineLocation)` e atualiza `stockItem.location = quarantineLocation`. Retorna `201 Created` com a lista de tasks geradas.

2. A `quarantineLocation` selecionada no AC 1 é determinada como a primeira localização ativa do tipo `QUARANTINE` no mesmo warehouse da `dockLocation`. Se não existir nenhuma location `QUARANTINE` ativa no warehouse, retorna `422 Unprocessable Entity` com mensagem descritiva.

3. Se a `ReceivingNote` não estiver em status `COMPLETED_WITH_DIVERGENCE`, `POST /api/v1/receiving-notes/{id}/quarantine-tasks` retorna `422 Unprocessable Entity`.

4. Se já existirem `QuarantineTask`s geradas para a mesma `ReceivingNote`, o endpoint retorna `409 Conflict` (idempotência por `receivingNoteId`). **Nota:** notas com status `COMPLETED` (sem divergência) não possuem itens FLAGGED — AC 3 cobre esse caso.

5. Ao chamar `PATCH /api/v1/quarantine-tasks/{id}/start`, o status da task muda de `PENDING` para `IN_REVIEW`; retorna `200` com a task atualizada. Requer role `WMS_SUPERVISOR` ou `WMS_ADMIN`.

6. Ao chamar `PATCH /api/v1/quarantine-tasks/{id}/decide` com `{ "decision": "RELEASE_TO_STOCK", "qualityNotes": "..." }`:
   - Task deve estar `IN_REVIEW` → 409 caso contrário
   - Sistema sugere a primeira location `STORAGE` ou `PICKING` disponível no warehouse (mesma lógica de capacidade de Story 3.2)
   - Se nenhuma location disponível → `422 Unprocessable Entity`
   - `stockItem.location` é atualizado para a `storageLocation` selecionada
   - `StockMovement(type=PUTAWAY, sourceLocation=quarantineLocation, destinationLocation=storageLocation)` é criado
   - `AuditLog(action=MOVEMENT, entityType="QUARANTINE_TASK")` é registrado com o supervisor e a decisão
   - Task recebe `status=APPROVED`, `decision=RELEASE_TO_STOCK`, `decidedAt`, `supervisorId`
   - Retorna `200` com a task atualizada

7. Ao chamar `PATCH /api/v1/quarantine-tasks/{id}/decide` com `{ "decision": "RETURN_TO_SUPPLIER", "qualityNotes": "..." }`:
   - Task deve estar `IN_REVIEW` → 409 caso contrário
   - `StockMovement(type=QUARANTINE_OUT, sourceLocation=quarantineLocation, destinationLocation=null)` é criado
   - `AuditLog(action=MOVEMENT, entityType="QUARANTINE_TASK")` é registrado
   - Evento Kafka é publicado no tópico `wms.quarantine.return-supplier` com payload contendo `tenantId`, `receivingNoteId`, `stockItemId`, `productId`, `quantity`, `qualityNotes`
   - `stockItem.quantityAvailable` permanece inalterado (SCM deve processar a devolução físicamente)
   - Task recebe `status=REJECTED`, `decision=RETURN_TO_SUPPLIER`, `decidedAt`, `supervisorId`
   - Retorna `200` com a task atualizada

8. Ao chamar `PATCH /api/v1/quarantine-tasks/{id}/decide` com `{ "decision": "SCRAP", "qualityNotes": "..." }`:
   - Task deve estar `IN_REVIEW` → 409 caso contrário
   - `StockMovement(type=QUARANTINE_OUT, sourceLocation=quarantineLocation, destinationLocation=null)` é criado com quantidade negativa (ajuste de estoque)
   - `AuditLog(action=MOVEMENT, entityType="QUARANTINE_TASK")` é registrado
   - `stockItem.quantityAvailable = 0` (item descartado)
   - Task recebe `status=REJECTED`, `decision=SCRAP`, `decidedAt`, `supervisorId`
   - Retorna `200` com a task atualizada

9. Ao chamar `PATCH /api/v1/quarantine-tasks/{id}/cancel`, o status muda de `PENDING` ou `IN_REVIEW` para `CANCELLED`; `stockItem.location` permanece na `quarantineLocation` (nenhum movimento adicional); retorna `204 No Content`. Requer role `WMS_SUPERVISOR` ou `WMS_ADMIN`. Se task estiver `APPROVED` ou `REJECTED`, retorna `409 Conflict`.

10. `GET /api/v1/quarantine-tasks` lista todas as tasks do tenant com filtros opcionais por `status` e `warehouseId`. `GET /api/v1/quarantine-tasks/{id}` retorna a task pelo ID.

11. Todos os endpoints aplicam isolamento multi-tenant via `TenantContextHolder.getTenantId()`; tentativa de acesso a task de outro tenant retorna `404 Not Found`.

12. Roles exigidas: `generate` requer `WMS_OPERATOR`, `WMS_SUPERVISOR` ou `WMS_ADMIN`; `start`, `decide` e `cancel` requerem `WMS_SUPERVISOR` ou `WMS_ADMIN`; `GET` (list e by ID) requerem `WMS_OPERATOR` ou acima.

### Lógica de `generateTasks()` — Resumo

```
generateTasks(noteId, tenantId):
  1. Buscar ReceivingNote (404 se não existe)
  2. Validar status == COMPLETED_WITH_DIVERGENCE → 422 se não
  3. existsByTenantIdAndReceivingNoteId → 409 se já gerado
  4. warehouseId = note.getDockLocation().getShelf()...getWarehouse().getId()
  5. quarantineLocations = locationRepository.findQuarantineByWarehouse(warehouseId, tenantId)
     → 422 se vazio ("Nenhuma location QUARANTINE ativa no warehouse X")
  6. quarantineLocation = quarantineLocations.get(0)  // primeira disponível
  7. Para cada item onde item.status==FLAGGED && item.stockItem!=null:
     a. stockItem.location = quarantineLocation
     b. stockItemRepository.save(stockItem)
     c. Criar StockMovement(QUARANTINE_IN, source=dockLocation, dest=quarantineLocation, ref=QUARANTINE_TASK)
     d. Criar QuarantineTask(status=PENDING, source=dockLocation, quarantine=quarantineLocation)
  8. if tasks.isEmpty() → BusinessException("Nenhum item FLAGGED com StockItem encontrado")
  9. return tasks.stream().map(toResponse).toList()
```
