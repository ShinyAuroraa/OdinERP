# Story 10.1: Gestão de Armazéns — CRUD + Hierarquia + Mapa Visual

## Status

InReview

## Story

**As a** administrador de armazém (WMS_ADMIN),
**I want** gerenciar a estrutura física completa do armazém via interface web — cadastrando armazéns, zonas, corredores, prateleiras e localizações com visualização hierárquica e mapa de grid,
**so that** operadores possam navegar pela estrutura do armazém, visualizar status de localizações e administradores possam manter a estrutura física atualizada.

## Complexity

**Estimativa:** L (Large) · ~4-6 dias de desenvolvimento
[Hierarquia 5 níveis (Warehouse→Zone→Aisle→Shelf→Location) + mapa visual grid]

## Dependencies

**Pré-requisitos:**
- Story 9.1 concluída ✅ — Next.js 15 + Keycloak PKCE + Layout Shell + API client
- Story 9.2 concluída ✅ — Design System (DataTable, WmsFormField, PageHeader, StatusBadge, ConfirmDialog, LoadingState, ErrorBoundary)
- Backend STORY-WMS-2.1 Done ✅ — APIs `/api/v1/warehouses`, `/api/v1/zones`, `/api/v1/aisles`, `/api/v1/shelves`, `/api/v1/locations`

## Out of Scope

- ❌ Importação em lote CSV/Excel de estrutura física
- ❌ Drag-and-drop no mapa visual para reordenar localizações
- ❌ Histórico de mudanças de localização
- ❌ Capacidade vs. estoque validado (Wave 4)
- ❌ Reativação de entidades desativadas

## Acceptance Criteria

1. **Página `/warehouses`** — lista de armazéns com DataTable:
   - Colunas: Nome, Código, Localização, Status (StatusBadge), Zonas (count), Ações
   - Busca global + filtro por status (ACTIVE/INACTIVE)
   - Botão "Novo Armazém" (apenas WMS_ADMIN)
   - Loading skeleton + empty state "Nenhum armazém cadastrado"

2. **Formulário de Armazém** (modal ConfirmDialog ou page `/warehouses/new`):
   - Campos: nome*, código*, localização, capacidade (m²), description
   - Validação Zod: código único por tenant, nome obrigatório
   - Submit → `POST /api/v1/warehouses` → toast sucesso/erro (sonner)
   - Editar → `PUT /api/v1/warehouses/{id}` → atualizar DataTable (React Query invalidate)

3. **Drill-down hierárquico** — `/warehouses/{id}`:
   - PageHeader com nome do armazém + breadcrumb
   - Tabs: Zonas | Visão Geral
   - Tab Zonas: DataTable de zonas com CRUD (Add Zona, Editar, Deletar com ConfirmDialog)
   - Drill-down Zona → DataTable de Corredores (Aisles)
   - Drill-down Corredor → DataTable de Prateleiras (Shelves)
   - Drill-down Prateleira → Grid de Localizações (mapa visual)

4. **Mapa visual de Localizações** (grid):
   - Grid de localizações organizadas por coluna/nível
   - Cor por status: AVAILABLE (verde), OCCUPIED (amarelo), BLOCKED (vermelho), QUARANTINE (laranja)
   - Tooltip com: código, tipo (LocationType), fullAddress, capacidade
   - Filtro por LocationType: PICKING, RECEIVING_DOCK, QUARANTINE, GENERAL_STORAGE, etc.
   - Botão "Adicionar Localização" abre formulário com tipo, capacidade, andar

5. **React Query hooks** — `lib/api/warehouses.ts`:
   - `useWarehouses()` — GET /api/v1/warehouses, staleTime 5min
   - `useWarehouse(id)` — GET /api/v1/warehouses/{id}, staleTime 5min
   - `useZones(warehouseId)` — GET /api/v1/warehouses/{id}/zones
   - `useAisles(zoneId)` — GET /api/v1/zones/{id}/aisles
   - `useShelves(aisleId)` — GET /api/v1/aisles/{id}/shelves
   - `useLocations(shelfId)` — GET /api/v1/shelves/{id}/locations
   - Mutations: `useCreateWarehouse`, `useUpdateWarehouse`, `useDeleteWarehouse`, e equivalentes para cada nível
   - Invalidate correto após mutations (cache por nível hierárquico)

6. **TypeScript types** — `types/warehouse.ts`:
   - `Warehouse`, `Zone`, `Aisle`, `Shelf`, `Location` interfaces espelhando DTOs do backend
   - `LocationType` enum: PICKING, RECEIVING_DOCK, QUARANTINE, GENERAL_STORAGE, CROSS_DOCK, STAGING
   - `WarehouseStatus`, `ZoneStatus`, etc.

7. **Proteção por role**:
   - Listagem: WMS_ADMIN, WMS_SUPERVISOR, WMS_OPERATOR, WMS_VIEWER (todos)
   - CRUD (criar/editar/deletar): apenas WMS_ADMIN
   - Botões de ação ocultos para roles sem permissão

8. **Testes unitários** (≥ 12 novos):
   - WarehouseList: render, loading, empty, search
   - WarehouseForm: submit válido, erro de validação
   - LocationGrid: render localizações, filtra por tipo, cores corretas por status
   - hooks: useWarehouses mock, useCreateWarehouse mutation

### API Endpoints Backend (STORY-WMS-2.1)

```
Warehouses:
GET    /api/v1/warehouses              → List<WarehouseResponse>
POST   /api/v1/warehouses              → WarehouseResponse (201)
GET    /api/v1/warehouses/{id}         → WarehouseResponse
PUT    /api/v1/warehouses/{id}         → WarehouseResponse
DELETE /api/v1/warehouses/{id}         → 204

Zones:
GET    /api/v1/warehouses/{wId}/zones  → List<ZoneResponse>
POST   /api/v1/warehouses/{wId}/zones  → ZoneResponse (201)
PUT    /api/v1/zones/{id}              → ZoneResponse
DELETE /api/v1/zones/{id}              → 204

Aisles:
GET    /api/v1/zones/{zId}/aisles      → List<AisleResponse>
POST   /api/v1/zones/{zId}/aisles      → AisleResponse (201)
PUT    /api/v1/aisles/{id}             → AisleResponse
DELETE /api/v1/aisles/{id}             → 204

Shelves:
GET    /api/v1/aisles/{aId}/shelves    → List<ShelfResponse>
POST   /api/v1/aisles/{aId}/shelves    → ShelfResponse (201)
PUT    /api/v1/shelves/{id}            → ShelfResponse
DELETE /api/v1/shelves/{id}            → 204

Locations:
GET    /api/v1/shelves/{sId}/locations → List<LocationResponse>
GET    /api/v1/locations?type=PICKING  → List<LocationResponse> (filtro por tipo)
POST   /api/v1/shelves/{sId}/locations → LocationResponse (201)
PUT    /api/v1/locations/{id}          → LocationResponse
DELETE /api/v1/locations/{id}          → 204
```
