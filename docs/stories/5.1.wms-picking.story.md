# Story 5.1: Picking — Separação de Pedidos (Integração CRM)

## Status

Ready for Review

## Story

**As a** operador de armazém (WMS Operator),
**I want** receber ordens de picking geradas automaticamente a partir de pedidos de venda confirmados pelo CRM — com rota otimizada por algoritmo S-shape, seleção automática de lote por FIFO/FEFO, confirmação item a item via scanner e suporte a picking parcial —
**so that** a separação seja precisa, rastreável e eficiente, integrando o fluxo outbound do WMS ao CRM sem intervenção manual e gerando evento Kafka de conclusão para atualização do status de entrega.

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| CRM publica evento duplicado (retry Kafka) | Alta | Alto | `UNIQUE (tenant_id, crm_order_id)` na tabela + idempotency check em `KafkaListener` |
| Concorrência: dois operadores atribuídos ao mesmo item simultaneamente | Média | Alto | `@Version` em `StockItem` → HTTP 409 Conflict; `@Version` em `PickingOrder` → impede duplo assign |
| Estoque insuficiente detectado apenas no assign (não no create) | Média | Médio | AC5 valida disponibilidade no assign; PENDING apenas cria a intenção sem tocar estoque |
| Kafka `crm.orders.confirmed` indisponível durante receive | Baixa | Alto | Retry automático Kafka (`@KafkaListener` com `errorHandler`); Dead Letter Topic opcional |
| FEFO com `expiryDate = null` (lotes sem validade) | Alta | Baixo | Fallback para FIFO quando `expiryDate` é null — ordena por `receivedAt ASC` |
| Picking parcial com reserva que não é totalmente consumida | Média | Médio | `completeOrder()` libera o residual reservado: `quantityReserved -= (requested - picked)` + restaura `quantityAvailable += (requested - picked)` |

### AC3 — Kafka Consumer: `crm.orders.confirmed` → criação de PickingOrder com FIFO/FEFO

**Given** o tópico Kafka `crm.orders.confirmed` recebe um evento com:
```json
{
  "eventType": "SALES_ORDER_CONFIRMED",
  "tenantId": "uuid-tenant",
  "crmOrderId": "uuid-crm-order",
  "warehouseId": "uuid-warehouse",
  "priority": 2,
  "items": [
    { "productId": "uuid-prod", "quantity": 5 }
  ]
}
```
**When** `CrmOrderEventConsumer` processa o evento,
**Then**:
- Verifica idempotência: `pickingOrderRepository.findByTenantIdAndCrmOrderId(tenantId, crmOrderId)` — se já existe, loga e descarta o evento
- Para cada item do evento, seleciona `StockItem` por FIFO/FEFO:
  - Se produto tem lote com `expiryDate != null` → ordena por `lot.expiryDate ASC` (FEFO)
  - Caso contrário → ordena por `stockItem.receivedAt ASC` (FIFO)
  - Seleciona apenas `StockItems` com `quantityAvailable > 0` do `warehouseId` informado
- Cria `PickingOrder` com `status = PENDING`, `pickingType = SINGLE`, `routingAlgorithm = S_SHAPE`, `crmOrderId` preenchido
- Cria `PickingItem` para cada item resolvido, com `locationId` do `StockItem` selecionado, `lotId` do lote (se houver)
- `sort_order` dos `PickingItem`s é calculado pelo algoritmo S-shape (AC5) na criação
- Se estoque insuficiente para algum item: cria o `PickingItem` com `quantity_requested` e `location_id` do melhor StockItem disponível (mesmo que `quantityAvailable` < `quantity_requested`) — a insuficiência é detectada no assign (AC5)
- Erros no processamento são logados; o Kafka consumer **não propaga exceções** para evitar reprocessamento infinito — utilizar `@KafkaListener(errorHandler = "pickingKafkaErrorHandler")`

### AC5 — `PUT /picking-orders/{id}/assign` — atribuir operador + reservar estoque

**Given** um usuário autenticado com role `WMS_OPERATOR` ou superior,
**When** chama `PUT /picking-orders/{id}/assign` com:
```json
{
  "operatorId": "uuid-operador",
  "zoneId": "uuid-zone-or-null"
}
```
**Then**:
- Ordem em status `PENDING` → processada; outros status → `400 Bad Request` com estado atual
- Ordem de outro tenant → `403 Forbidden`
- Ordem inexistente → `404 Not Found`
- Para cada `PickingItem` em `PENDING`:
  - Verifica `stockItem.quantityAvailable >= pickingItem.quantityRequested` → `400 Bad Request` com `"Estoque insuficiente para produto {productId}: disponível={x}, solicitado={y}"` se insuficiente
  - Dentro de `@Transactional`: `stockItem.quantityAvailable -= quantityRequested` e `stockItem.quantityReserved += quantityRequested` (optimistic locking → HTTP 409 Conflict se concorrência)
- Calcula `sort_order` por algoritmo S-shape:
  - Agrupa `PickingItem`s por `aisle` (via `location.aisle.code`)
  - Corredores com código par → itens ordenados por prateleira ASC
  - Corredores com código ímpar → itens ordenados por prateleira DESC
  - Resultado: lista de itens numerados sequencialmente por `sort_order` (1, 2, 3...)
- `PickingOrder.status` → `IN_PROGRESS`, `operatorId` preenchido
- Retorna `200 OK` com `PickingOrderResponse` atualizado (itens com `sort_order` definido)
- Chama `stockBalanceService.evictAll()` após todas as reservas confirmadas

### AC6 — `PUT /picking-orders/{id}/pick-item/{itemId}` — confirmar coleta de item (scanner)

**Given** um usuário autenticado com role `WMS_OPERATOR` ou superior,
**When** chama `PUT /picking-orders/{id}/pick-item/{itemId}` com:
```json
{
  "quantityPicked": 10,
  "pickedBy": "uuid-operador"
}
```
**Then**:
- Item em status `PENDING` → processado
- Item já `PICKED` ou `SKIPPED` → `400 Bad Request`
- Ordem não `IN_PROGRESS` → `400 Bad Request`
- `quantityPicked > pickingItem.quantityRequested` → `400 Bad Request` com `"Quantidade coletada excede a solicitada"`
- `quantityPicked == quantityRequested` → `PickingItem.status = PICKED`
- `quantityPicked < quantityRequested` → `PickingItem.status = PARTIAL`
- `quantityPicked == 0` → `PickingItem.status = SKIPPED`
- `picked_by` e `picked_at` (UTC) preenchidos
- Retorna `200 OK` com `PickingItemResponse` atualizado

### AC7 — Picking parcial e substituição de localização

**Given** um `PickingItem` em status `PARTIAL` ou `SKIPPED` (estoque insuficiente na localização sugerida),
**When** o operador tenta coletar de localização alternativa via:
`PUT /picking-orders/{id}/pick-item/{itemId}/substitute-location`
```json
{
  "alternativeLocationId": "uuid-location-alt",
  "alternativeLotId": "uuid-lot-alt-or-null",
  "quantityPicked": 7,
  "pickedBy": "uuid-operador"
}
```
**Then**:
- Valida que `alternativeLocationId` pertence ao mesmo `warehouseId` da ordem e ao mesmo tenant
- Verifica `StockItem` na localização alternativa: `quantityAvailable >= quantityPicked` → `400 Bad Request` se insuficiente
- Dentro de `@Transactional`:
  - Decrementa `quantityAvailable` na localização alternativa e incrementa `quantityReserved`
  - Restaura `quantityAvailable` e decrementa `quantityReserved` na localização original (liberando a reserva da localização original)
  - Atualiza `PickingItem.locationId`, `PickingItem.lotId`, `PickingItem.quantityPicked`
  - `PickingItem.status = PICKED` se `quantityPicked == quantityRequested`, senão `PARTIAL`
- Retorna `200 OK` com `PickingItemResponse` atualizado

### AC8 — `PUT /picking-orders/{id}/complete` — completar ordem + dedução de estoque + evento Kafka

**Given** um usuário autenticado com role `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `PUT /picking-orders/{id}/complete`,
**Then**:
- Ordem em status `IN_PROGRESS` com pelo menos um item `PICKED` ou `PARTIAL` → processada; outros status → `400 Bad Request`
- Dentro de `@Transactional` para cada `PickingItem`:
  - `stockItem.quantityReserved -= pickingItem.quantityRequested` *(libera toda a reserva)*
  - Se `quantityPicked > 0`: cria `StockMovement` do tipo `MovementType.OUTBOUND` com `referenceType = ReferenceType.PICKING_ORDER`, `referenceId = pickingOrder.getId()`; *(StockItem.quantityAvailable já foi ajustado no assign — não decrementar novamente)*
  - Se resíduo de reserva (PARTIAL/SKIPPED): `stockItem.quantityAvailable += (quantityRequested - quantityPicked)` *(restaura a diferença)*
- Status da ordem:
  - Todos os itens `PICKED` → `PickingOrder.status = COMPLETED`
  - Algum item `PARTIAL` ou `SKIPPED` → `PickingOrder.status = PARTIAL`
- `completed_at` preenchido com timestamp UTC
- `auditLogRepository.save()` com `entityType("PICKING_ORDER")`, `action(AuditAction.MOVEMENT)`, `newValue(JSON com resumo da coleta)`
- `auditLogIndexer.indexAuditLogAsync()` chamado assincronamente
- `stockBalanceService.evictAll()` para invalidar cache Redis
- Retorna `200 OK` com `PickingOrderResponse` atualizado

### AC9 — `PUT /picking-orders/{id}/cancel` — cancelar ordem + liberar reserva

**Given** um usuário autenticado com role `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `PUT /picking-orders/{id}/cancel` com `{ "reason": "motivo" }`,
**Then**:
- Ordem `PENDING` (sem reserva) → `status = CANCELLED` sem tocar estoque
- Ordem `IN_PROGRESS` (com reserva) → dentro de `@Transactional`: para cada `PickingItem` com status `PENDING`, `PARTIAL` ou `PICKED` *(todos têm reserva ativa — SKIPPED nunca foi reservado)*:
  - `stockItem.quantityReserved -= pickingItem.quantityRequested`
  - `stockItem.quantityAvailable += pickingItem.quantityRequested`
  - `stockBalanceService.evictAll()`
- Ordem `COMPLETED` ou `PARTIAL` → `400 Bad Request` com `"Ordem já concluída não pode ser cancelada"`
- Ordem já `CANCELLED` → `400 Bad Request` com `"Ordem já cancelada"`
- `cancelled_at` e `cancellation_reason` preenchidos
- `WMS_OPERATOR` tentando cancelar → `403 Forbidden`
- Retorna `200 OK` com `PickingOrderResponse`

## Dev Notes

### Contexto Arquitetural

**`MovementType.OUTBOUND`** — verificar se já existe no enum `MovementType.java`. Se não, adicionar. Picking gera saída de estoque da localização de picking.

**`ReferenceType.PICKING_ORDER`** — adicionar ao enum `ReferenceType.java` (mesmo padrão do `INTERNAL_TRANSFER` na Story 4.5).

**`BaseEntity`** é a base correta para ambas as entidades — status e quantidades mudam após criação.

**StockItem — Fluxo de Estoque no Picking:**
```
assign:    quantityAvailable -= requested   quantityReserved += requested
complete:  quantityReserved  -= requested
           (se PARTIAL/SKIPPED): quantityAvailable += (requested - picked)
cancel:    quantityReserved  -= requested   quantityAvailable += requested
```
*Nota: No `complete`, o `quantityAvailable` já foi decrementado no `assign`. O `StockMovement.OUTBOUND` documenta a saída; não há segundo decremento de `quantityAvailable`.*

**FIFO/FEFO — seleção de StockItem:**
```java
// Em PickingService.selectStockItemsForOrder():
List<StockItem> candidates = stockItemRepository
    .findByTenantIdAndProductIdAndWarehouseIdAndQuantityAvailableGreaterThan(
        tenantId, productId, warehouseId, 0);

// FEFO: se produto tem lote com expiryDate
candidates.sort(Comparator.comparing(
    si -> si.getLot() != null && si.getLot().getExpiryDate() != null
        ? si.getLot().getExpiryDate().atStartOfDay(ZoneOffset.UTC).toInstant()
        : si.getReceivedAt() // fallback FIFO
```
