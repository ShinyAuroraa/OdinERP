# Story 6.1: Integração WMS ↔ MRP/MRP II — Matéria-Prima & Produto Acabado

## Status

Draft

## Story

**As a** gerente de produção / sistema MRP,
**I want** que o WMS receba ordens de produção do MRP automaticamente — reservando matéria-prima, gerando ordens de picking para a linha de produção, confirmando a entrega de materiais e registrando o recebimento de produtos acabados —
**so that** a cadeia WMS ↔ MRP seja bidirecional e consistente, com rastreabilidade completa de materiais desde o armazém até a produção e de produtos acabados desde a produção até o estoque, usando Saga pattern para garantir consistência distribuída com Dead Letter Queue para falhas.

## Complexity

**Estimativa:** XL (Extra Large) · ~5-7 dias de desenvolvimento

[Fonte: EPIC-WMS-001 — Story 6.1 é a integração bidirecional WMS↔MRP. Consome eventos `mrp.production.order.released` e `mrp.production.order.cancelled`. Publica eventos `wms.materials.delivered`, `wms.finished.goods.received`, `wms.stock.shortage`. Requer: extensão de `PickingOrder` com `orderType` e `productionOrderId` (V13 migration), novas tabelas `production_material_requests` + `production_material_request_items`, Saga pattern com compensação (liberar reservas em caso de falha), DLQ. Depende de: `StockItem.quantityReserved` (Story 4.1), `PickingService` com fluxo de picking (Story 5.1), `AuditLogRepository`/`AuditLogIndexer` (Story 4.4), `ReferenceType` extensível (Stories 4.5, 5.1, 5.2, 5.3).]

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Estoque insuficiente de MP no momento da ordem | Alta | Alto | AC4: status `STOCK_SHORTAGE` + evento `wms.stock.shortage` → MRP decide aguardar/cancelar |
| Duplicata de `mrp.production.order.released` (retry Kafka) | Alta | Médio | `UNIQUE (tenant_id, production_order_id)` + `DataIntegrityViolationException` catch |
| Saga compensation falha após reserva parcial | Baixa | Alto | AC6: transação explícita com rollback — se criação do PickingOrder falha, release de TODAS as reservas via `StockService.releaseReservation()` |
| Cancelamento de OP com picking em andamento | Média | Médio | AC10: status `PICKING_IN_PROGRESS` → loga `WARN`, não cancela automaticamente; supervisor deve intervir |
| Concorrência: duas OPs reservam o mesmo estoque | Média | Alto | AC3: `StockItem.@Version` (optimistic locking) já implementado em Story 4.1 |
| `mrp.production.order.released` sem componentes | Baixa | Baixo | Consumer valida `event.components().isEmpty()` → status `ERROR` + log warning |

### AC4 — Kafka Consumer: `mrp.production.order.released` → Saga de reserva e picking

**Given** o tópico `mrp.production.order.released` recebe um `MrpProductionOrderReleasedEvent`,
**When** `MrpProductionOrderConsumer.handleProductionOrderReleased()` processa o evento,
**Then** o fluxo Saga é executado em `ProductionIntegrationService.processProductionOrderReleased()`:

**Passo 1 — Idempotência:**
- `existsByTenantIdAndProductionOrderId(tenantId, productionOrderId)` → se já existe, loga INFO e retorna (sem exceção)
- Guard secundário: captura `DataIntegrityViolationException` (UNIQUE constraint) e retorna sem propagar

**Passo 2 — Criar `ProductionMaterialRequest`:**
- `status = PENDING` → salva
- Cria `ProductionMaterialRequestItem` para cada componente do evento
- Após salvar itens: `status = RESERVING` → salva

**Passo 3 — Reservar estoque (por componente):**
- Para cada `ProductionMaterialRequestItem`:
  - `StockItem` encontrado com saldo disponível (`quantityAvailable >= quantityRequired`):
    - `stockItem.quantityReserved += quantityRequired`
    - `stockItem.quantityAvailable -= quantityRequired` *(fórmula: `totalQty - reserved - quarantine`)*
    - Salva `StockItem`
    - `item.quantityReserved = quantityRequired`, `item.shortage = false`
  - `StockItem` com saldo insuficiente (`quantityAvailable < quantityRequired`):
    - `item.quantityReserved = quantityAvailable` *(reserva o que tem)*
    - `item.shortage = true`
    - `request.shortageComponents++`
- Salva todos os itens atualizados

**Passo 4A — Sem shortage → Criar PickingOrder PRODUCTION_ORDER:**
```
IF request.shortageComponents == 0:
    PickingOrder picking = PickingOrder.builder()
        .tenantId(tenantId)
        .warehouseId(request.warehouseId)
        .orderType(PickingOrderType.PRODUCTION_ORDER)
        .productionOrderId(productionOrderId)
        .status(PickingStatus.PENDING)
        .build();
    pickingOrderRepository.save(picking);

    // Criar PickingItems para cada componente
    for each item in requestItems:
        PickingItem.builder()
            .pickingOrderId(picking.getId())
            .productId(item.productId)
            .quantityRequested(item.quantityReserved)
            .status(PickingItemStatus.PENDING)
            .build();

    request.pickingOrderId = picking.getId();
    request.status = PICKING_PENDING;
```

**Passo 4B — Com shortage → Publicar evento de shortage:**
```
IF request.shortageComponents > 0:
    request.status = STOCK_SHORTAGE;
    stockShortageEventPublisher.publishStockShortage(buildShortageEvent(...));
    // NÃO cria PickingOrder — MRP decide aguardar ou cancelar
```

**Compensação (Saga rollback) — se criação do PickingOrder falha:**
```
ON Exception during PickingOrder creation:
    // Libera todas as reservas
    for each item in requestItems WHERE item.quantityReserved > 0:
        stockItem.quantityReserved -= item.quantityReserved;
        stockItem.quantityAvailable += item.quantityReserved;
        stockItem.save();
    request.status = ERROR;
    request.errorMessage = exception.getMessage();
    request.save();
    // Loga ERROR mas NÃO propaga — consumer não deve bloquear offset
```

**AuditLog:** registra `AuditAction.CREATE` com `entityType = "PRODUCTION_MATERIAL_REQUEST"` após criação bem-sucedida.

### AC5 — Kafka Consumer: `mrp.production.order.cancelled` → cancelamento e release de reservas

**Given** o tópico `mrp.production.order.cancelled` recebe um `MrpProductionOrderCancelledEvent`,
**When** `MrpProductionOrderConsumer.handleProductionOrderCancelled()` processa o evento,
**Then**:
- Busca `ProductionMaterialRequest` por `(tenantId, productionOrderId)`
- Se não encontrado → loga WARN e retorna (idempotência / mensagem fora de ordem)
- **Se status for `PICKING_IN_PROGRESS`:**
  - Loga `WARN`: `"OP {} cancelada enquanto picking em andamento (pickingOrder={}). Intervenção manual necessária."`
  - Atualiza `request.status = CANCELLED`, `request.cancellationReason = event.cancellationReason()`, `request.cancelledAt = Instant.now()`
  - **Não** cancela o PickingOrder automaticamente (operador deve concluir/cancelar manualmente)
  - AuditLog registra o cancelamento com `WARNING` no `newValue`
- **Se status for qualquer outro (`PENDING`, `RESERVING`, `STOCK_SHORTAGE`, `PICKING_PENDING`):**
  - Libera todas as reservas: para cada item com `quantityReserved > 0`, incrementa `stockItem.quantityAvailable` e decrementa `stockItem.quantityReserved`
  - Se `pickingOrderId != null` e status `PICKING_PENDING`: cancela o PickingOrder (status → CANCELLED)
  - `request.status = CANCELLED`, `request.cancellationReason`, `request.cancelledAt`
- **Se status for `DELIVERED` ou `FINISHED_GOODS_RECEIVED`:**
  - Loga INFO: `"OP {} já concluída, cancelamento ignorado"` — retorna sem alteração
- AuditLog registra `AuditAction.UPDATE` com `oldValue/newValue`

### AC6 — `POST /api/v1/production-material-requests/{id}/confirm-delivery` — confirmar entrega de MP na produção

**Given** um usuário autenticado com role `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `POST /api/v1/production-material-requests/{id}/confirm-delivery`,
**Then**:
- Request em status `PICKING_PENDING` ou `PICKING_IN_PROGRESS` → processado; outros → `409 Conflict`
- Request de outro tenant → `403 Forbidden`; inexistente → `404 Not Found`
- Para cada `ProductionMaterialRequestItem`:
  - Cria `StockMovement` com `type = OUTBOUND`, `referenceType = PRODUCTION_ORDER`, `referenceId = request.id`
  - Decrementa `StockItem.quantityReserved -= item.quantityReserved`
  - `item.quantityDelivered = item.quantityReserved`
- `request.status = DELIVERED`, `request.confirmedDeliveryAt = Instant.now()`, `request.confirmedBy` = UUID do usuário autenticado
- Publica `MaterialsDeliveredEvent` via `MaterialsDeliveredEventPublisher`
- AuditLog registra `AuditAction.MOVEMENT` com detalhes da entrega
- Retorna `200 OK` com `ProductionMaterialRequestResponse`

### AC7 — `POST /api/v1/production-material-requests/{id}/receive-finished-goods` — receber produto acabado no WMS

### AC10 — Dead Letter Queue: `mrp.production.order.released.dlq`

**Given** o consumer `mrp.production.order.released` falha após 3 retentativas,
**When** o Spring Kafka DefaultErrorHandler esgota as tentativas,
**Then**:
- Mensagem enviada automaticamente para tópico `mrp.production.order.released.dlq`
- Consumer **não bloqueia** o offset — próximas mensagens continuam sendo processadas
- Loga `ERROR` com `productionOrderId`, `tenantId` e stacktrace do erro original
- Mesmo comportamento para `mrp.production.order.cancelled.dlq`
- Configurar `DefaultErrorHandler` em `KafkaConfig.java` com `FixedBackOff(1000L, 3L)` (3 tentativas, 1s entre elas)

### AC12 — Rastreabilidade e AuditLog

**Given** qualquer operação de criação, atualização ou movimentação de materiais,
**When** a operação é executada com sucesso,
**Then**:
- `AuditLog` salvo via `auditLogRepository.save()` com:
  - `entityType = "PRODUCTION_MATERIAL_REQUEST"`
  - `entityId = request.getId()`
  - `action` = `CREATE` (criação), `UPDATE` (cancelamento/atualização), `MOVEMENT` (entrega/recebimento)

### Arquitetura do Saga Pattern

```
Consumer recebe MrpProductionOrderReleasedEvent
    ↓
[1] Cria ProductionMaterialRequest (PENDING)
    ↓
[2] status → RESERVING, reserva StockItems
    ↓
[3A] Sem shortage → Cria PickingOrder (PRODUCTION_ORDER)
    ├── Sucesso → status → PICKING_PENDING
    └── Falha → COMPENSAÇÃO: libera reservas → status → ERROR
         ↓
[3B] Com shortage → status → STOCK_SHORTAGE + evento wms.stock.shortage
```
