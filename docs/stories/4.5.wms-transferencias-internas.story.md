# Story 4.5: Transferências Internas entre Posições

## Status

Ready for Review

## Story

**As a** operador de armazém (WMS Operator),
**I want** mover estoque entre posições, zonas e armazéns de forma controlada — manual ou automática (reabastecimento de picking) — com confirmação via scanner, registro imutável no audit log e publicação de evento Kafka,
**so that** o sistema sempre reflita a localização real do estoque em tempo real, com rastreabilidade completa de cada movimentação interna.

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Concorrência em movimentação de estoque (dois operadores transferem o mesmo StockItem simultaneamente) | Alta | Alto | `@Version` em `StockItem` lança `OptimisticLockingFailureException` → HTTP 409 Conflict |
| Cache Redis com saldo desatualizado após transferência confirmada | Média | Médio | `stockBalanceService.evictAll()` após confirmação (padrão da Story 4.3) |
| Kafka indisponível não deve bloquear confirmação de transferência | Baixa | Médio | Mesmo padrão de `QuarantineEventPublisher`: erros logados mas não propagados |
| Confirmar transferência já CONFIRMED (duplo clique no scanner) | Média | Médio | Verificação de estado: `PENDING` → rejeita com `400 Bad Request` se já CONFIRMED/CANCELLED |
| Stock disponível insuficiente na origem | Média | Alto | Validação em `confirmTransfer()`: `quantityAvailable >= quantity` antes de decrementar |

## Acceptance Criteria

### AC1 — V9 migration: tabela `internal_transfers`

**Given** o banco PostgreSQL com o schema WMS existente,
**When** a migration `V9__internal_transfers.sql` é aplicada,
**Then**:
- `CREATE TABLE internal_transfers` com colunas:
  - `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
  - `tenant_id UUID NOT NULL`
  - `transfer_type VARCHAR(20) NOT NULL CHECK (transfer_type IN ('MANUAL', 'REPLENISHMENT'))`
  - `status VARCHAR(20) NOT NULL DEFAULT 'PENDING' CHECK (status IN ('PENDING', 'CONFIRMED', 'CANCELLED'))`
  - `source_location_id UUID NOT NULL REFERENCES locations(id)`
  - `destination_location_id UUID NOT NULL REFERENCES locations(id)`
  - `product_id UUID NOT NULL REFERENCES products_wms(id)`
  - `lot_id UUID REFERENCES lots(id)` *(nullable — produtos sem controle de lote)*
  - `quantity INT NOT NULL CHECK (quantity > 0)`
  - `requested_by UUID NOT NULL` *(UUID Keycloak — sem FK cross-service)*
  - `confirmed_by UUID` *(nullable — preenchido na confirmação)*
  - `cancelled_by UUID` *(nullable — preenchido no cancelamento)*
  - `reason VARCHAR(500)` *(nullable — motivo da transferência)*
  - `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`
  - `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`
  - `version BIGINT NOT NULL DEFAULT 0`
- `CONSTRAINT chk_diff_locations CHECK (source_location_id != destination_location_id)`
- Índice: `idx_internal_transfers_tenant_status ON internal_transfers (tenant_id, status, created_at DESC)`
- Índice: `idx_internal_transfers_tenant_product ON internal_transfers (tenant_id, product_id, created_at DESC)`

### AC2 — `InternalTransfer` entity + `InternalTransferRepository`

**Given** a migration V9 aplicada,
**When** o @dev implementa a camada JPA,
**Then**:
- `InternalTransfer` entity: `@SuperBuilder @NoArgsConstructor @Entity @Table("internal_transfers")` — estende `BaseEntity` (tem `updatedAt`, status é mutável)
- `@Version Long version` — optimistic locking no próprio `InternalTransfer` (independente do `StockItem`)
- Enums internos:
  ```java
  public enum TransferType { MANUAL, REPLENISHMENT }
  public enum TransferStatus { PENDING, CONFIRMED, CANCELLED }
  ```
- `InternalTransferRepository extends JpaRepository<InternalTransfer, UUID>` com:
  - `Page<InternalTransfer> findByTenantIdAndStatus(UUID tenantId, TransferStatus status, Pageable pageable)`
  - `Page<InternalTransfer> findByTenantId(UUID tenantId, Pageable pageable)`
  - `Optional<InternalTransfer> findByIdAndTenantId(UUID id, UUID tenantId)`

### AC3 — POST /transfers — criar transferência manual

**Given** um usuário autenticado com role `WMS_OPERATOR`,
**When** chama `POST /transfers` com:
```json
{
  "sourceLocationId": "uuid",
  "destinationLocationId": "uuid",
  "productId": "uuid",
  "lotId": "uuid-or-null",
  "quantity": 10,
  "reason": "Reabastecimento setor A"
}
```
**Then** retorna `201 Created` com:
```json
{
  "id": "uuid",
  "transferType": "MANUAL",
  "status": "PENDING",
  "sourceLocationId": "uuid",
  "destinationLocationId": "uuid",
  "productId": "uuid",
  "lotId": "uuid-or-null",
  "quantity": 10,
  "reason": "Reabastecimento setor A",
  "requestedBy": "uuid-operador",
  "createdAt": "2026-02-23T10:00:00Z"
}
```
- Validação: `sourceLocationId != destinationLocationId` → `400 Bad Request` se iguais
- Validação: `quantity > 0` → `400 Bad Request` se zero ou negativo
- Validação: `sourceLocation` e `destinationLocation` pertencem ao mesmo `tenantId` do JWT → `400 Bad Request` se não
- Validação: `productId` existe no tenant → `404 Not Found` se não
- A transferência é criada em status `PENDING` — o estoque NÃO é movido neste momento
- `transferType = MANUAL` para chamadas via API REST por operador

### AC4 — PUT /transfers/{id}/confirm — confirmar transferência via scanner

**Given** um usuário autenticado com role `WMS_OPERATOR`,
**When** chama `PUT /transfers/{id}/confirm` com:
```json
{
  "confirmedBy": "uuid-operador"
}
```
**Then**:
- Transferência no status `PENDING` → processada e retornada com status `CONFIRMED`
- Transferência já `CONFIRMED` ou `CANCELLED` → `400 Bad Request` com mensagem descrevendo o estado atual
- Transferência de outro tenant → `403 Forbidden`
- `id` inexistente → `404 Not Found`

### AC5 — Atualização de estoque na confirmação (com optimistic locking)

**Given** uma transferência `PENDING` sendo confirmada,
**When** `InternalTransferService.confirmTransfer()` é executado,
**Then** dentro de uma única `@Transactional`:
- Busca `StockItem` na `sourceLocation` para o `product` + `lot` → `404` se não encontrado
- Verifica `stockItem.quantityAvailable >= transfer.quantity` → `400 Bad Request` com mensagem `"Estoque insuficiente: disponível={x}, solicitado={y}"` se insuficiente
- Decrementa `sourceStockItem.quantityAvailable -= quantity` e salva (lança `OptimisticLockingFailureException` → HTTP `409 Conflict` se concorrência)
- Busca ou cria `StockItem` na `destinationLocation` para o mesmo `product` + `lot`
- Incrementa `destinationStockItem.quantityAvailable += quantity` e salva
- Cria `StockMovement` do tipo `MovementType.TRANSFER` com `sourceLocation`, `destinationLocation`, `referenceType = ReferenceType.INTERNAL_TRANSFER`, `referenceId = transfer.getId()`
- Chama `stockBalanceService.evictAll()` para invalidar cache Redis

### AC6 — Registro no audit log e indexação ES

**Given** uma transferência sendo confirmada,
**When** o `StockMovement` é salvo na mesma transação de AC5,
**Then**:
- `auditLogRepository.save(AuditLog.builder()...)` com:
  - `entityType("INTERNAL_TRANSFER")`, `entityId(transfer.getId())`
  - `action(AuditAction.MOVEMENT)`, `actorId(confirmedBy)`
  - `newValue(JSON: {"movementType":"TRANSFER","quantity":{q},"sourceLocationId":"{srcId}","destinationLocationId":"{dstId}"})`
- `auditLogIndexer.indexAuditLogAsync(savedAuditLog)` chamado após o save (assíncrono, não-crítico)
- Tudo dentro da mesma `@Transactional` — falha no `auditLogRepository.save()` faz rollback de todo o bloco

### AC7 — Publicação evento Kafka após confirmação

**Given** uma transferência confirmada com sucesso (transação de AC5 commitada),
**When** o serviço publica o evento,
**Then**:
- `StockMovementEventPublisher.publishTransferEvent(transfer, movement)` envia para tópico `wms.stock.movements`
- Payload do evento:
  ```json
  {
    "eventType": "INTERNAL_TRANSFER_CONFIRMED",
    "tenantId": "uuid",
    "transferId": "uuid",
    "movementId": "uuid",
    "productId": "uuid",
    "lotId": "uuid-or-null",
    "quantity": 10,
    "sourceLocationId": "uuid",
    "destinationLocationId": "uuid",
    "confirmedBy": "uuid",
    "occurredAt": "2026-02-23T10:05:00Z"
  }
  ```
- Key Kafka: `tenantId.toString()` (garante ordenação por tenant)
- Erros de publicação são logados como `ERROR` mas **não propagados** — a transferência já está confirmada no banco (mesmo padrão de `QuarantineEventPublisher`)

### AC8 — PUT /transfers/{id}/cancel — cancelar transferência

**Given** um usuário autenticado com role `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `PUT /transfers/{id}/cancel` com `{ "reason": "motivo" }`,
**Then**:
- Transferência `PENDING` → status `CANCELLED`, `cancelled_by` preenchido, retorna `200 OK`
- Transferência `CONFIRMED` → `400 Bad Request` com `"Transferência já confirmada não pode ser cancelada"`
- Transferência `CANCELLED` → `400 Bad Request` com `"Transferência já cancelada"`
- `WMS_OPERATOR` tentando cancelar → `403 Forbidden`
- Transferência de outro tenant → `403 Forbidden`
- Cancelamento NÃO movimenta estoque (nada foi movido ainda no status PENDING)

### AC9 — GET /transfers/{id} e GET /transfers (listagem paginada)

**Given** um usuário autenticado com role `WMS_OPERATOR` ou superior,
**When** chama `GET /transfers/{id}`,
**Then** retorna `200 OK` com `InternalTransferResponse` completo incluindo `confirmedBy`, `cancelledBy`, timestamps.
- Transferência de outro tenant → `403 Forbidden`

**When** chama `GET /transfers?status=PENDING&page=0&size=20`,
**Then** retorna `200 OK` com página de transferências do tenant filtradas por `status` (opcional).

### AC11 — Segurança: roles por operação

**Given** os endpoints de transferências,
**Then** as regras de autorização são:

- `PUT /transfers/{id}/confirm` → `WMS_OPERATOR`, `WMS_SUPERVISOR`, `WMS_ADMIN`
- `PUT /transfers/{id}/cancel` → `WMS_SUPERVISOR`, `WMS_ADMIN` (operador NÃO pode cancelar)
