# Story 16.5: Health Endpoints + Observabilidade

## Status

Done

## Story

**As a** engenheiro DevOps,
**I want** endpoints de health check e métricas Prometheus no `wms-web`,
**so that** o Kubernetes possa verificar a saúde do pod e o Prometheus scrape métricas do processo Node.js.

## Complexity

**Estimativa:** S (Small) · ~0.5 dia

## Dependencies

- Story 16.4 ✅ (K8s manifests — probes aguardando `/api/health`)

## Out of Scope

- ❌ Métricas de negócio (requisições WMS, etc.)
- ❌ Grafana dashboard
- ❌ Alertmanager rules

## Acceptance Criteria

1. **`app/api/health/route.ts`** — GET `/api/health`:
   - Retorna `{ status: "ok", service: "wms-web", timestamp: ISO, uptime: seconds }`
   - HTTP 200
2. **`app/api/metrics/route.ts`** — GET `/api/metrics`:
   - Retorna métricas em formato Prometheus text (Content-Type: `text/plain; version=0.0.4`)
   - Inclui: `process_uptime_seconds`, `process_heap_bytes{used,total}`, `process_rss_bytes`
3. **`infra/k8s/wms-web/deployment.yaml`** atualizado:
   - probes HTTP GET atualizadas de `/` para `/api/health`
   - anotações Prometheus scrape no pod template
4. **Testes unitários** (≥ 4): health 200, body fields, metrics format, Content-Type
5. **169+ testes passando** (sem regressões)

## Dev Notes

### Estrutura

```
app/api/
```

### Sem dependências extras

Métricas implementadas sem `prom-client` — usa `process.memoryUsage()` e `process.uptime()` nativas do Node.js.

### K8s Prometheus Scrape Annotations

```yaml
annotations:
  prometheus.io/scrape: "true"
  prometheus.io/port: "3000"
  prometheus.io/path: "/api/metrics"
```
