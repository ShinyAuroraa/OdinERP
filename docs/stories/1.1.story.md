## Story

**As a** desenvolvedor,
**I want** um microserviço Spring Boot estruturado e funcional,
**so that** eu tenha a base técnica do backend do ODIN CRM pronta para receber código de negócio.

## Acceptance Criteria

1. Projeto Spring Boot 3 + Java 21 criado com estrutura de módulos Gradle multi-projeto: `crm-domain`, `crm-application`, `crm-infrastructure`, `crm-web`
2. Estrutura DDD interna de cada módulo respeitada: `domain → application → infrastructure → interfaces` (sem dependências inversas)
3. Dependências configuradas no módulo correto:
   - `crm-domain`: zero dependências de framework (Java puro)
   - `crm-web`: Spring Web, Spring Security, Spring Actuator, Micrometer, Lombok, MapStruct
   - `crm-infrastructure`: Spring Data JPA, Hibernate, Resilience4j, Kafka, gRPC stubs
4. Endpoint `GET /actuator/health` retorna `{"status": "UP"}` com HTTP 200
5. Dockerfile multi-stage criado — imagem final baseada em `eclipse-temurin:21-jre-alpine`, executando como usuário não-root, tamanho < 200MB
6. `docker/compose.yml` local com PostgreSQL 17, Redis 7, Kafka 3.7 (KRaft), Keycloak 24 e OpenSearch 2 para desenvolvimento local
7. Logging estruturado em JSON configurado via Logback com padrão: `%d{ISO8601} [%X{traceId},%X{spanId}] %-5level [%X{userId}] %logger{36} — %msg%n`
8. Testes unitários de smoke passando com JUnit 5 (ao menos um teste por módulo confirma que o contexto Spring carrega sem erros)
9. `application.yml`, `application-local.yml` e `application-test.yml` criados com externalização de todas as credenciais via variáveis de ambiente (zero hardcode)
10. `gradle/libs.versions.toml` criado como version catalog único — todas as versões de dependências centralizadas neste arquivo

## Dev Notes

### Contexto da História

Esta é a **primeira story do projeto** — sem story anterior para referência. Este skeleton estabelece a fundação arquitetural sobre a qual todos os épicos seguintes serão construídos.

### Estrutura de Módulos (Source: architecture.md#Section-10.1)

```
odin-crm-backend/
├── settings.gradle.kts               ← Declara 4 submodules
├── build.gradle.kts                  ← Root: shared config
├── gradle/libs.versions.toml         ← Version catalog único
├── crm-domain/                       ← Java puro, ZERO Spring
├── crm-application/                  ← Use cases, depende só de crm-domain
├── crm-infrastructure/               ← Adapters: DB, Kafka, gRPC, Redis, OpenSearch
└── crm-web/                          ← Spring Boot main app: controllers, security, Flyway
```

**REGRA CRÍTICA de dependências entre módulos:**
```
crm-web → crm-infrastructure → crm-application → crm-domain
```
Dependências reversas são proibidas. `crm-domain` não pode importar nada de Spring ou de outros módulos.

### Versões Exatas (Source: architecture.md#Section-3)

```
Java:          21 LTS (eclipse-temurin)
Spring Boot:   3.3.x
Gradle:        8.x (Kotlin DSL)
PostgreSQL:    17 (Alpine)
Redis:         7 (Alpine)
Kafka:         3.7.0 (KRaft mode — sem ZooKeeper)
Keycloak:      24.0
OpenSearch:    2.13.0
```

### Dockerfile — Detalhamento (Source: architecture.md#Section-13.6)

```dockerfile
FROM eclipse-temurin:21-jdk-alpine AS builder
WORKDIR /build
# Copiar Gradle wrapper e arquivos de build primeiro (cache de layers)
COPY gradlew settings.gradle.kts build.gradle.kts gradle/ ./
COPY crm-domain/ crm-domain/
COPY crm-application/ crm-application/
COPY crm-infrastructure/ crm-infrastructure/
COPY crm-web/ crm-web/
RUN ./gradlew :crm-web:bootJar -x test --no-daemon

FROM eclipse-temurin:21-jre-alpine AS runtime
RUN addgroup -S odin && adduser -S odin -G odin
WORKDIR /app
COPY --from=builder /build/crm-web/build/libs/*.jar app.jar
USER odin
EXPOSE 8080 9090
ENTRYPOINT ["java", \
  "-XX:+UseContainerSupport", \
  "-XX:MaxRAMPercentage=75.0", \
  "-Djava.security.egd=file:/dev/./urandom", \
  "-jar", "app.jar"]
```

### application.yml — Configurações obrigatórias (Source: architecture.md#Section-10.10)

```yaml
spring:
  application:
    name: odin-crm-backend
  jpa:
    open-in-view: false       # OBRIGATÓRIO — previne N+1 via lazy loading
    hibernate:
      ddl-auto: validate      # Flyway gerencia schema, JPA só valida
  flyway:
    enabled: true
    locations: classpath:db/migration

management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus,metrics
  endpoint:
    health:
      probes:
        enabled: true
      group:
        liveness:
          include: livenessState
        readiness:
          include: readinessState,db,redis,kafka
```

### Testing

**Frameworks:** JUnit 5 (`junit-jupiter`), Spring Boot Test (`@SpringBootTest`), AssertJ
**Localização dos testes:** `src/test/java/` espelhando estrutura de `src/main/java/`
**Perfil de teste:** `@ActiveProfiles("test")` — usa `application-test.yml`
**Cobertura mínima:** 80% (verificado via Jacoco em CI — não obrigatório nesta story, apenas smoke tests)
**Padrão de nomenclatura:** `{ClasseTestada}Test.java` para unitários, `{ClasseTestada}IntegrationTest.java` para integração

**Smoke test mínimo para esta story:**
```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@ActiveProfiles("test")
class CrmApplicationSmokeTest {
    @Autowired
    private ApplicationContext context;

    @Test
    void contextLoads() {
        assertThat(context).isNotNull();
    }
}
```

> **Note:** Testcontainers será configurado em Story 1.2. Para esta story, o perfil `test` deve funcionar com H2 em memória OU desabilitar AutoConfigurations de DB para o smoke test (`@SpringBootTest` com `excludeAutoConfiguration`).
