# Story 4.1: Controle de Estoque em Tempo Real

## Status

Ready

## Story

**As a** supervisor ou operador de armazém (WMS Supervisor / WMS Operator),
**I want** consultar o saldo de estoque em tempo real por produto, localização, lote e status — com cache Redis para baixa latência — e visualizar a taxa de ocupação por zona e armazém,
**so that** decisões operacionais (putaway, picking, transferência) sejam baseadas em dados precisos e atualizados, sobrecarga de localizações seja evitada, e o sistema responda com latência aceitável mesmo sob alta carga de consultas simultâneas.

## Complexity

**Estimativa:** M (Medium) · ~2-3 dias de desenvolvimento
[Fonte: EPIC-WMS-001 — sem nova entidade; leitura sobre `StockItem` existente + Redis cache layer + JPQL aggregations]

## Dependencies

**Pré-requisitos para iniciar esta story:**
- Story 3.1 concluída ✅ — `StockItem` criado com `quantityAvailable`, `quantityReserved`, `quantityQuarantine`, `quantityDamaged` e `@Version` (optimistic locking)
- Story 3.2 concluída ✅ — `PutawayService.confirm()` popula `StockItem.quantityAvailable`
- Story 3.3 concluída ✅ — `QuarantineService.decide()` move qty entre `quantityQuarantine` ↔ `quantityAvailable`

**Stories desbloqueadas após esta:**
- Story 4.2 — Rastreabilidade (consulta de saldo por lote/série/validade)
- Story 4.3 — Inventário Físico (reconciliação compara saldo atual com contagem)
- Story 5.1 — Picking (reserva de estoque com `quantityReserved`)

## Out of Scope

> Itens explicitamente **não implementados** nesta story:

- ❌ Endpoint de escrita de estoque (ajuste, transferência) — Story 4.5
- ❌ Kafka consumer para atualizar saldo a partir de eventos externos — Enhancement futuro (saldo já é mantido pelos services existentes)
- ❌ Relatórios históricos de saldo por data — Story 4.4 (audit log)
- ❌ Reserva de estoque para picking — Story 5.1 (PickingService gerencia `quantityReserved`)
- ❌ Dashboard gráfico (UI React) — Wave 8 / UX stories

- ❌ Cache warm-up na inicialização do serviço — Enhancement futuro

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Redis indisponível em produção | Baixa | Médio | Degradação graciosa: fallback para query direta ao PostgreSQL sem exception |
| Cache eviction global (allEntries=true) causa thundering herd | Média | Médio | TTL curto (5min) + eviction por tenant-prefix via `RedisCacheWriter`; avaliar `allEntries = true` apenas se implementação simples justificar |
| `ObjectOptimisticLockingFailureException` não tratada em callers existentes | Alta | Alto | AC7 formaliza o tratamento obrigatório em PutawayService e QuarantineService — incluído na Task 9 |
| JPQL com navegação profunda (shelf→aisle→zone→warehouse) gera cartesian joins | Média | Médio | Monitorar explain plan nos testes de integração; índices em V7 mitigam |
| `countByTenantIdAndShelfAisleZoneWarehouseId` nome inválido para Spring Data | Baixa | Alto | Usar `@Query` JPQL explícita ao invés de derived query name — clarificar na Dev Note |

## Acceptance Criteria

### AC1 — GET /stock/balance (listagem com filtros)
**Given** um usuário autenticado com qualquer role WMS,
**When** chama `GET /stock/balance` com filtros opcionais `productId`, `locationId`, `lotId`, `warehouseId`,
**Then** retorna `200 OK` com lista de `StockBalanceResponse` filtrada pelo `tenantId` do JWT.
- Sem filtros retorna todos os `StockItem` do tenant
- Filtros são cumulativos (AND)
- Lista vazia retorna `[]` com `200 OK` (não `404`)

### AC2 — StockBalanceResponse contém todos os campos necessários
**Given** um `StockItem` com produto, lote, localização e quantidades,
**When** o response é serializado,
**Then** `StockBalanceResponse` contém:
```json
{
  "id": "uuid-do-stock-item",
  "productId": "uuid",
  "productCode": "SKU-001",
  "productName": "Nome do Produto",
  "locationId": "uuid",
  "locationCode": "A-01-B-02",
  "locationType": "STORAGE",
  "lotId": "uuid | null",
  "lotNumber": "LOT-2026-001 | null",
  "quantityAvailable": 100,
  "quantityReserved": 10,
  "quantityQuarantine": 5,
  "quantityDamaged": 2,
  "receivedAt": "2026-02-22T10:00:00Z"
}
```

### AC3 — GET /stock/balance/location/{locationId}
**Given** um `locationId` válido pertencente ao tenant,
**When** chama `GET /stock/balance/location/{locationId}`,
**Then** retorna `200 OK` com lista de `StockBalanceResponse` da localização.
**And** se `locationId` não existe ou pertence a outro tenant → retorna `404 Not Found`.

### AC4 — GET /stock/occupation?warehouseId={id} (ocupação por armazém)
**Given** um `warehouseId` válido pertencente ao tenant,
**When** chama `GET /stock/occupation?warehouseId={id}`,
**Then** retorna `200 OK` com `WarehouseOccupationResponse`:
```json
{
  "warehouseId": "uuid",
  "warehouseName": "Armazém Central",
  "totalLocations": 500,
  "occupiedLocations": 320,
  "occupancyRate": 64.0,
  "zones": [
    {
      "zoneId": "uuid",
      "zoneName": "Zona A",
      "totalLocations": 100,
      "occupiedLocations": 75,
      "occupancyRate": 75.0,
      "totalCapacityUnits": 1000,
      "usedCapacityUnits": 750
    }
  ]
}
```
**And** se `warehouseId` não existe ou pertence a outro tenant → `404 Not Found`.
**And** `occupancyRate` = (`occupiedLocations` / `totalLocations`) × 100, arredondado em 1 casa decimal.

### AC5 — Redis Cache em consultas de saldo
**Given** que `spring-boot-starter-data-redis` está configurado e Redis acessível,
**When** `GET /stock/balance` é chamado com os mesmos parâmetros duas vezes consecutivas,
**Then** a segunda chamada retorna o resultado do cache Redis (sem novo acesso ao PostgreSQL).
**And** TTL padrão: 5 minutos (configurável via `wms.cache.stock-balance.ttl-minutes`).
**And** cache key pattern: `stock:balance:{tenantId}:{paramHash}`.

### AC6 — Cache invalidation em operações de escrita
**Given** que existe um cache de saldo para um produto/localização,
**When** `PutawayService.confirm()` altera `StockItem.quantityAvailable`,
**Then** o cache daquele tenant é evictado (`@CacheEvict(cacheNames = "stockBalance", allEntries = false)`).
**And** mesma invalidação ocorre em `QuarantineService.decide()` e `QuarantineService.cancel()`.
**And** na ausência de Redis (ex.: testes), o sistema degrada graciosamente sem exception (fallback para query direta).

### AC7 — Bloqueio otimista: tratamento de ObjectOptimisticLockingFailureException
**Given** que `StockItem` usa `@Version` para optimistic locking,
**When** dois processos concorrentes tentam modificar o mesmo `StockItem` simultaneamente,
**Then** o segundo processo recebe `ObjectOptimisticLockingFailureException` do JPA.
**And** os serviços que modificam `StockItem` (`PutawayService`, `QuarantineService`) capturam essa exceção e lançam `BusinessException("Conflito de concorrência — tente novamente")` com status `409 Conflict`.
**And** os endpoints de escrita existentes (`POST /quarantine-tasks/{id}/start`, `POST /quarantine-tasks/{id}/decide`) retornam `409` nesse cenário.

> **Nota:** `StockItem.@Version` já existe. Esta AC formaliza o tratamento nos callers.

### AC8 — Isolamento multi-tenant
**Given** dois tenants A e B com StockItems distintos,
**When** tenant A chama `GET /stock/balance`,
**Then** apenas `StockItem`s com `tenantId` do JWT do tenant A são retornados.
**And** `GET /stock/balance/location/{locationId}` retorna `404` se a location pertence ao tenant B.
**And** `GET /stock/occupation?warehouseId={id}` retorna `404` se o warehouse pertence ao tenant B.

### AC9 — RBAC: roles autorizadas
**Given** endpoints de consulta de saldo são read-only,
**When** usuário com role `wms-operator`, `wms-supervisor`, ou `wms-admin` chama qualquer endpoint de stock balance,
**Then** retorna `200 OK` com dados do tenant.
**And** usuário sem nenhuma role WMS (ex.: token de outro microserviço sem claim wms) → `403 Forbidden`.
**And** token ausente ou inválido → `401 Unauthorized`.

### AC10 — Consulta por produto com soma de quantidades
**Given** um produto presente em múltiplas localizações (ex.: 3 localizações com 50 unidades cada),
**When** chama `GET /stock/balance?productId={id}`,
**Then** retorna lista com 3 entradas (uma por StockItem) — **não** um total agregado.
**And** o caller (frontend/other service) é responsável por somar se necessário.

## Dev Notes

### Arquitetura

**Padrão:** leitura direta do `StockItem` (tabela existente) — sem nova entidade, sem migration de schema.

```
GET /stock/balance
  → StockBalanceController
    → StockBalanceService (@Cacheable)
      → StockItemRepository (JPQL query com filtros opcionais)
        → Redis cache hit: retorna sem DB
        → Cache miss: query PostgreSQL, popula cache, retorna

GET /stock/occupation
  → StockBalanceController
    → StockBalanceService (@Cacheable)
      → LocationRepository (aggregate por zone/warehouse)
      → StockItemRepository (count por location)
```

### Redis Cache Configuration

```java
// StockCacheConfig.java
@Configuration
@EnableCaching
public class StockCacheConfig {
    @Bean
    public RedisCacheConfiguration stockBalanceCacheConfig(
            @Value("${wms.cache.stock-balance.ttl-minutes:5}") int ttl) {
        return RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(ttl))
                .serializeValuesWith(RedisSerializationContext.SerializationPair
                        .fromSerializer(new GenericJackson2JsonRedisSerializer()));
    }
}
```

**Cache eviction nos serviços existentes:**

#### ⚠️ Concerns Documentados

**[CONCERN-1] Severity: LOW — AC5 cache key pattern diverge da spec**
- Spec: `stock:balance:{tenantId}:{paramHash}`
- Implementação: `{tenantId}:bal:{productId}:{locationId}:{lotId}:{warehouseId}`
- Impacto funcional: nenhum — cache funciona corretamente. Divergência é apenas cosmética.
- Recomendação: alinhar spec com implementação em próxima revisão de story (sem bloqueio).

**[CONCERN-2] Severity: LOW — AC6 spec `allEntries = false` vs implementação `allEntries = true`**
- Spec original usa `allEntries = false`, mas @po validou explicitamente no Change Log v1.1.0: "OBS: @CacheEvict key wildcard não funciona em Spring SpEL — @dev deve usar `allEntries=true`".
- Decision log do @dev documenta: "Spring SpEL wildcard de tenant não funciona — `allEntries=true` necessário".
- Impacto: eviction mais agressiva (todos os entries do cache), porém tecnicamente correto dado o constraint do Spring SpEL.
- Status: **Aceito** — decisão validada pelo @po e documentada.

**[CONCERN-3] Severity: MEDIUM — AC7 implementação via GlobalExceptionHandler, não nos services**
- Spec AC7: "os serviços (`PutawayService`, `QuarantineService`) capturam essa exceção e lançam `BusinessException`"
- Implementação: `GlobalExceptionHandler.handleOptimisticLocking()` captura `ObjectOptimisticLockingFailureException` globalmente e retorna `409 Conflict` com mensagem traduzida.
- Comportamento externo: **idêntico** — o caller recebe `409 Conflict` com a mensagem correta.
- Ausência de teste de integração cobrindo o cenário de concorrência real (apenas comportamento via GlobalExceptionHandler verificado indiretamente).
- Recomendação: atualizar spec AC7 para refletir a abordagem global (mais correta por DRY principle) em Story 4.2 ou revisão de spec.

**[CONCERN-4] Severity: LOW — Cobertura de edge cases em RBAC**
- Testes cobrem: `WMS_ADMIN` (admin), `WMS_OPERATOR` (operator), `401` (sem token).
- Não há teste para: `WMS_SUPERVISOR` role explícita; `403 Forbidden` (token válido sem role WMS).
- AC9 especifica comportamento para ambos os casos, mas apenas `401` e autorizado testados formalmente.
- Recomendação: adicionar em Story 4.2 ou como enhancement de test coverage.
