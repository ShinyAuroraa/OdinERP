# Story 8.2: App Android WMS — Recebimento & Conferência

## Status

Ready for Review

## Story

**As a** operador de armazém,
**I want** um fluxo completo de recebimento e conferência de mercadorias no aplicativo Android WMS, com leitura de código de barras GS1, confirmação de itens, registro de divergências com foto de avaria e assinatura digital,
**so that** eu possa executar o recebimento de pedidos de compra diretamente do dispositivo móvel — com ou sem conexão de rede — garantindo rastreabilidade total de cada item recebido e sincronização automática ao recuperar conectividade.

## Complexity

**Estimativa:** L (Large) · ~4-5 dias de desenvolvimento

[Fonte: EPIC-WMS-001 — Story 8.2 é o primeiro fluxo operacional real do Wave 8, construído sobre a fundação da Story 8.1. Reusa BarcodeAnalyzer, Hilt DI, Room, WorkManager e a estrutura MVVM + Clean Architecture existentes. Novas entidades Room: `receiving_orders_cache`, `receiving_items_cache`, `receiving_sync_queue`. Novo módulo: `ReceivingSyncWorker`. APIs backend: 8 endpoints REST da Story 3.1 já disponíveis.]

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Captura de foto (câmera) conflita com ML Kit CameraX já em uso | Média | Alto | Usar `ImageCapture` Use Case do CameraX separado do `ImageAnalysis`; configurar múltiplos use cases na mesma `CameraProvider` conforme documentação CameraX |
| GS1-128 com AIs complexos (AI(10) Lote + AI(17) Validade na mesma string) pode falhar parsing | Média | Médio | Implementar `Gs1Parser` com regex e fallback manual; casos de borda cobertos por testes U5–U6 |
| Assinatura digital (canvas touch) pode ter performance ruim em dispositivos low-end | Baixa | Médio | Usar `Canvas` nativo do Compose com `Pointer Input`; throttle de 16ms para pontos do path |
| Offline sync com conflito: confirmação local vs cancelamento remoto da ordem | Média | Alto | Política "server wins" para estado da ordem; log de conflito visível no histórico local |
| Armazenamento de fotos de avaria pode esgotar espaço em armazéns com muitas divergências | Baixa | Médio | Fotos comprimidas a 512px / 60% JPEG antes de armazenar; limpeza automática pós-sync |
| `ReceivingSyncWorker` competindo com `SyncWorker` base em janelas de conectividade curtas | Baixa | Baixo | Workers com tags distintas; `ExistingWorkPolicy.KEEP` para evitar duplicação |

## Acceptance Criteria

### AC1 — Lista de Ordens de Recebimento Pendentes

**Given** operador autenticado com role `WMS_OPERATOR` (ou superior),
**When** acessa a tela "Recebimento" via menu de Operações,
**Then**:
- Lista exibe ordens com status `PENDING` ou `IN_PROGRESS` via `GET /api/v1/receiving/orders?status=PENDING&tenantId={tenantId}`
- Cada card de ordem exibe: número da ordem, fornecedor, data esperada, total de itens, status
- Ordens `IN_PROGRESS` exibem badge de progresso (ex: "3/8 itens confirmados")
- Pull-to-refresh atualiza a lista
- Em modo offline: lista vem do Room (`receiving_orders_cache`) com banner "Dados offline"
- Estado vazio (sem ordens): mensagem "Nenhuma ordem de recebimento pendente" com ícone
- Erro de rede: mensagem de erro com botão "Tentar novamente"

---

### AC2 — Detalhe da Ordem e Início do Recebimento

**Given** operador na lista de ordens,
**When** seleciona uma ordem,
**Then**:
- Tela de detalhe exibe: cabeçalho da ordem (número, fornecedor, data), lista de itens esperados
- Cada item exibe: código do produto, descrição, quantidade esperada, quantidade confirmada (inicialmente 0), status (PENDING / CONFIRMED / DIVERGENT)
- Botão "Iniciar Recebimento" disponível para ordens `PENDING`; "Continuar" para `IN_PROGRESS`
- Botão "Scan Item" abre `BarcodeScannerScreen` (reuso do componente 8.1) no modo recebimento
- Dados do detalhe carregados via `GET /api/v1/receiving/orders/{id}`
- Detalhe cacheado em Room (`receiving_items_cache`) para uso offline

---

### AC3 — Leitura e Parsing de Código de Barras GS1

**Given** operador na tela de conferência de item,
**When** escaneia código de barras do produto,
**Then**:
- `BarcodeAnalyzer` existente (Story 8.1) detecta o código sem modificação
- `Gs1Parser` (novo componente) extrai Application Identifiers da string GS1-128:
  - AI(01) = GTIN-14 (identificador do produto)
  - AI(10) = Número de lote
  - AI(17) = Data de validade (formato YYMMDD → convertido para `LocalDate`)
  - AI(21) = Número de série
- Para EAN-13: converte para GTIN-14 com prefixo "0" (compatível com AC backend Story 3.1)
- Para QR Code: parseia JSON ou string GS1 composta
- Para SSCC: extrai identificador de container de pallet
- Código inválido ou produto não pertencente à ordem: exibe alerta com opção de scan manual
- Campo de entrada manual disponível como fallback (herda do AC6 da Story 8.1)

---

### AC4 — Confirmação de Item Recebido

**Given** operador com código de barras do item escaneado/inserido,
**When** confirma o recebimento do item,
**Then**:
- Tela de confirmação exibe: produto identificado (nome, GTIN), lote extraído do scan, validade extraída, quantidade esperada
- Campo de quantidade recebida (numérico, pré-preenchido com quantidade esperada — editável)
- Botão "Confirmar" chama `PUT /api/v1/receiving/orders/{id}/items/{itemId}/confirm` com `{ quantity, lotNumber, expiryDate, serialNumber }`
- Resposta de sucesso: item marcado como CONFIRMED na lista; contagem de progresso atualizada
- Em offline: confirmação enfileirada em `receiving_sync_queue` com status `PENDING`; item marcado localmente como CONFIRMED_OFFLINE
- Feedback visual: checkmark verde + vibração háptica ao confirmar
- Após confirmação: retorna automaticamente para lista de itens da ordem

---

### AC5 — Registro de Divergência (Falta, Excesso, Avaria)

**Given** operador identificando discrepância entre esperado e recebido,
**When** registra uma divergência,
**Then**:
- Botão "Registrar Divergência" disponível por item na lista de detalhe
- Tipos de divergência selecionáveis via `DivergenceType` enum: `SHORTAGE` (falta), `EXCESS` (excesso), `DAMAGE` (avaria), `WRONG_PRODUCT` (produto errado)
- Para SHORTAGE/EXCESS: campo de quantidade real recebida (obrigatório)
- Para DAMAGE/WRONG_PRODUCT: campo de observação (obrigatório, mínimo 10 caracteres)
- Para qualquer tipo: botão "Adicionar Foto" abre câmera via CameraX `ImageCapture` (máx. 3 fotos por divergência)
- Foto capturada: comprimida a 512px / 60% JPEG, armazenada em app-private storage (`context.filesDir/receiving/`)
- Confirmação da divergência: chama `PUT /api/v1/receiving/orders/{id}/items/{itemId}/divergence` com tipo, quantidade, observação e fotos em Base64
- Em offline: divergência + fotos enfileiradas no `receiving_sync_queue`
- Item marcado como DIVERGENT com badge visual (cor laranja) na lista da ordem

---

### AC6 — Assinatura Digital do Conferente

**Given** operador com todos os itens confirmados ou divergentes na ordem,
**When** finaliza o recebimento,
**Then**:
- Botão "Finalizar Recebimento" habilitado somente quando todos os itens tiverem status CONFIRMED ou DIVERGENT (nenhum PENDING)
- Tela de assinatura exibe canvas interativo para desenho com o dedo (Compose Canvas + `detectDragGestures`)
- Botão "Limpar" redefine o canvas
- Botão "Confirmar Assinatura" valida que o canvas não está vazio (mínimo de pontos desenhados > 50)
- Assinatura renderizada como PNG em memória → convertida para Base64
- Base64 da assinatura submetido via `POST /api/v1/receiving/orders/{id}/signature`
- Em offline: Base64 da assinatura armazenado em Room (campo `signatureBase64` em `ReceivingSyncQueueEntity`) e submetido pelo `ReceivingSyncWorker`
- Após assinatura aceita: chama `POST /api/v1/receiving/orders/{id}/complete`; ordem marcada como COMPLETED
- Tela de sucesso: "Recebimento concluído! Ordem #{numero}" com botão "Voltar para lista"

---

### AC7 — Sincronização Offline (`ReceivingSyncWorker`)

**Given** dispositivo sem conectividade com operações de recebimento enfileiradas,
**When** conectividade é restaurada,
**Then**:
- `ReceivingSyncWorker` (extends `CoroutineWorker`) ativado por `Constraints.Builder().setRequiredNetworkType(NetworkType.CONNECTED)`
- Worker processa `receiving_sync_queue` em ordem FIFO por `createdAt`
- Para cada item da fila com status `PENDING`: executa chamada REST correspondente (confirm / divergence / signature / complete)
- Sucesso: item da fila marcado como `SYNCED`; cache Room atualizado
- Falha transitória (5xx / timeout): retry com backoff exponencial (máx. 3 tentativas)
- Falha permanente (4xx): item marcado como `SYNC_FAILED`; notificação local ao usuário via `NotificationCompat`
- Conflito (ordem cancelada remotamente): operação local descartada; log de conflito gravado; usuário notificado
- Status de sync visível na tela de Perfil: "X itens pendentes de sincronização"
- Worker registrado com tag `"receiving_sync"` (diferente de `"general_sync"` da Story 8.1)

---

### AC8 — Segurança e Multi-tenant no Fluxo de Recebimento

**Given** qualquer operação de recebimento no app,
**Then**:
- Todas as chamadas REST incluem `Authorization: Bearer {jwt}` via `AuthInterceptor` (existente em 8.1)
- `tenant_id` extraído do JWT aplicado em todas as queries Room (`WHERE tenant_id = ?`)
- Fotos de avaria armazenadas exclusivamente em `context.filesDir/receiving/{tenantId}/` (não acessíveis por outras apps)
- Assinatura armazenada encriptada no Room via `@ColumnInfo(typeAffinity = ColumnInfo.BLOB)` com AES-256
- Limpeza automática de fotos sincronizadas com sucesso (deletar arquivo local após `SYNCED`)
- Dados de ordens de recebimento no Room isolados por `tenant_id` (constraint de query)
- Operador com role `WMS_OPERATOR` não pode cancelar ordem (apenas `WMS_SUPERVISOR`/`WMS_ADMIN`)
- Tentativa de acesso a ordem de outro tenant → erro 403 do backend; app exibe "Acesso negado" e retorna para lista

### GS1 Parser — Lógica de Referência

```kotlin
// Gs1Parser.kt
object Gs1Parser {
    // GS1-128 Application Identifiers
    private val AI_PATTERNS = mapOf(
        "01" to Regex("01(\\d{14})"),   // GTIN-14
        "10" to Regex("10([^\\x1D]{1,20})"),  // Lot/Batch (até separador GS1 = \x1D)
        "17" to Regex("17(\\d{6})"),    // Expiry YYMMDD
        "21" to Regex("21([^\\x1D]{1,20})")   // Serial Number
    )

    data class Gs1Data(
        val gtin: String? = null,
        val lotNumber: String? = null,
        val expiryDate: LocalDate? = null,
        val serialNumber: String? = null
    )

    fun parse(rawValue: String): Gs1Data {
        // 1. EAN-13 → GTIN-14 (prefixo "0")
        if (rawValue.length == 13 && rawValue.all { it.isDigit() }) {
            return Gs1Data(gtin = "0$rawValue")
        }
        // 2. GS1-128 com AIs concatenados
        val gtin = AI_PATTERNS["01"]?.find(rawValue)?.groupValues?.get(1)
        val lot = AI_PATTERNS["10"]?.find(rawValue)?.groupValues?.get(1)
        val expiryStr = AI_PATTERNS["17"]?.find(rawValue)?.groupValues?.get(1)
        val serial = AI_PATTERNS["21"]?.find(rawValue)?.groupValues?.get(1)
        val expiry = expiryStr?.let {
            LocalDate.of(2000 + it.substring(0,2).toInt(), it.substring(2,4).toInt(), it.substring(4,6).toInt())
        }
        return Gs1Data(gtin = gtin, lotNumber = lot, expiryDate = expiry, serialNumber = serial)
    }
}
```

### Estratégia Offline Detalhada

```
Operação online:
  ReceivingRepository.confirmItem()
    → WmsApiService.confirmReceivingItem() OK
    → ReceivingItemDao.updateLocalStatus(CONFIRMED)
    → retorna Result.Success

Operação offline:
  ReceivingRepository.confirmItem()
    → WmsApiService.confirmReceivingItem() throws NetworkException
    → ReceivingSyncQueueDao.insert(ReceivingSyncQueueEntity(type=CONFIRM, payload=json))
    → ReceivingItemDao.updateLocalStatus(CONFIRMED_OFFLINE)
    → retorna Result.QueuedForSync

ReceivingSyncWorker (ao recuperar rede):
  → query PENDING itens da fila para este tenantId (FIFO por createdAt)
  → para cada item: executar chamada REST
    → 2xx: updateStatus(SYNCED), deletar arquivo de foto local
    → 5xx/timeout: incrementar retryCount; se retryCount >= 3 → updateStatus(SYNC_FAILED) + notificação
    → 4xx (404 ordem não existe): conflito → updateStatus(SYNC_CONFLICT) + notificação
```
