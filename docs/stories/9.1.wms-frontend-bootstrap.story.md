# Story 9.1: Bootstrap Next.js 15 + Keycloak PKCE + Layout Shell

## Status

Ready for Review

## Story

**As a** engenheiro frontend,
**I want** criar o projeto base Next.js 15 com autenticação Keycloak PKCE, middleware de proteção de rotas multi-tenant, e o layout shell do WMS (sidebar + header + breadcrumbs),
**so that** todas as stories subsequentes do frontend WMS possam ser implementadas sobre uma fundação segura, tipada e com design system pronto — sem precisar re-implementar auth, routing ou layout.

## Complexity

**Estimativa:** M (Medium) · ~3-5 dias de desenvolvimento
[Fonte: EPIC-FE-001 Wave 1]

## Dependencies

**Pré-requisitos para iniciar esta story:**
- EPIC-WMS-001 concluído — APIs REST disponíveis em `/api/v1/`
- Keycloak realm `odin` com client `wms-web` configurado (PKCE, redirect URIs)
- `apps/` directory disponível no monorepo Odin ERP

**Stories desbloqueadas após esta:**
- Story 9.2 — Design System & Componentes Base (depende do projeto criado aqui)
- Todas as stories de feature (Waves 2–7) dependem desta fundação

## Out of Scope

- ❌ Componentes de domínio WMS (armazéns, produtos, etc.) → Stories Wave 2+
- ❌ Configuração do Keycloak realm (assume configurado externamente)
- ❌ Deploy Kubernetes / ArgoCD → @devops
- ❌ Storybook → opcional, Stories posteriores
- ❌ Internacionalização completa (i18n EN) → PT-BR apenas nesta story
- ❌ Modo escuro → Story 8.1 (Dashboard)
- ❌ PWA / Service Workers → fora do escopo MVP

## Business Value

- **Zero-config auth:** Qualquer desenvolvedor que criar uma nova page Next.js já tem autenticação, tenant extraction e RBAC prontos — elimina risco de rota exposta acidentalmente
- **Design system desde o início:** Layout shell consistente evita retrabalho de UI em cada story
- **Multi-tenant desde a fundação:** `TenantContext` disponível globalmente — qualquer API call já inclui `tenant_id` automaticamente

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Keycloak PKCE com Next.js 15 App Router — sessão em Server Components | Alta | Alto | Usar `keycloak-js` no lado cliente com `use client`; Server Components recebem token via cookie httpOnly |
| Next.js 15 não disponível no npm | Baixa | Alto | Verificar versão atual; usar `next@15` ou `next@canary` conforme disponibilidade |
| CORS em desenvolvimento local | Alta | Baixo | Configurar `rewrites` no `next.config.ts` para proxy da API WMS |

## Acceptance Criteria

1. **Projeto Next.js 15 criado** — `apps/wms-web/` inicializado com `create-next-app@latest` (App Router, TypeScript, Tailwind CSS, ESLint). `package.json` com scripts: `dev`, `build`, `start`, `lint`, `typecheck`, `test`. `tsconfig.json` com `strict: true`. Compila e roda em `localhost:3000`.

2. **Keycloak PKCE Auth implementado** — `keycloak-js` instalado. `KeycloakProvider` (client component) em `app/layout.tsx` inicializa o adapter com `flow: 'standard'` (PKCE default no keycloak-js). Configuração via env vars: `NEXT_PUBLIC_KEYCLOAK_URL`, `NEXT_PUBLIC_KEYCLOAK_REALM`, `NEXT_PUBLIC_KEYCLOAK_CLIENT_ID`. Usuário não autenticado é redirecionado para login Keycloak. Após login, token JWT armazenado em memória (não localStorage).

3. **Middleware de proteção de rotas** — `middleware.ts` na raiz de `apps/wms-web/`. Rotas sob `/(auth)/` exigem autenticação válida. Rotas públicas: `/login`, `/api/auth/`. Middleware extrai `tenant_id` do JWT e injeta em cookie `x-tenant-id` para uso em Server Components. Role insuficiente redireciona para `/unauthorized`.

4. **TenantContext e AuthContext criados** — `TenantContext` disponibiliza `tenantId: string` e `userRoles: WmsRole[]` para toda a árvore de componentes. `useAuth()` hook retorna `{ user, token, roles, tenantId, isAuthenticated, logout }`. `useHasRole(role: WmsRole)` retorna `boolean`.

5. **API Client configurado** — `lib/api/client.ts` com `fetch` wrapper que:
   - Adiciona `Authorization: Bearer {token}` em toda requisição
   - Adiciona `X-Tenant-Id: {tenantId}` header
   - Base URL: `NEXT_PUBLIC_API_URL` (default: `/api/proxy` → rewrites para o backend WMS)
   - `next.config.ts` com rewrites: `/api/proxy/:path*` → `${WMS_API_URL}/api/v1/:path*`
   - Erros 401 disparam logout automático, erros 403 redirecionam para `/unauthorized`

6. **Layout Shell completo** — `app/(auth)/layout.tsx` com:
   - **Sidebar** fixa (240px): logo Odin ERP, menu de navegação com ícones (Lucide), collapse para mobile, indicador de item ativo
   - **Header** (64px): breadcrumbs dinâmicos, nome do usuário + avatar, botão logout, nome do tenant
   - **Conteúdo principal:** área scrollável com padding consistente
   - **Responsividade:** sidebar vira drawer (hamburger) em mobile (< 768px)
   - Menu items por módulo (todos os itens do epic, desabilitados os não implementados com badge "Em breve"):
     - Dashboard, Armazéns, Produtos, Recebimento, Putaway, Estoque, Rastreabilidade, Inventário, Auditoria, Transferências, Picking, Packing, Shipping, MRP, Relatórios

7. **Página `/unauthorized` e tratamento de erros** — Página 403 com mensagem clara e botão voltar. `error.tsx` global para erros inesperados. `not-found.tsx` para 404. `loading.tsx` com skeleton global.

8. **Testes passando** — Suite de testes com ≥ 8 testes:
   - `middleware.test.ts`: rotas protegidas redirecionam sem token, rotas públicas passam
   - `useAuth.test.tsx`: hook retorna valores corretos com mock Keycloak
   - `useHasRole.test.tsx`: retorna true/false por role
   - `TenantContext.test.tsx`: tenantId extraído do token mock
   - `apiClient.test.ts`: headers JWT e tenant presentes, 401 dispara logout
   - `Layout.test.tsx`: sidebar renderiza com menu items, header renderiza user info
   - `middleware-role.test.ts`: role insuficiente redireciona para /unauthorized
   - Pelo menos 1 teste e2e Playwright: login flow completo (mock Keycloak)

9. **Linting e TypeScript passando** — `npm run lint` sem erros. `npm run typecheck` sem erros. ESLint com `@typescript-eslint/no-explicit-any` habilitado.

### Estratégia de Auth com Next.js 15

**Problema central:** Next.js App Router mistura Server e Client Components. Keycloak JS é um biblioteca cliente (browser).

**Solução recomendada:**
```
- KeycloakProvider (Client Component) inicializa auth no browser
- Token armazenado em React state + cookie httpOnly (via Route Handler)
- Middleware.ts lê o cookie para proteger rotas no edge
- Server Components leem o cookie para obter token
- Client Components usam useAuth() hook
```

**IMPORTANTE:** NÃO usar `localStorage` para o token — vulnerável a XSS. Usar cookie `httpOnly` ou memória React.
