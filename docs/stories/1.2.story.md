## Story

**As a** desenvolvedor,
**I want** banco de dados configurado com pipeline de migrations versionadas e testes de repositório com PostgreSQL real,
**so that** o schema evolua de forma controlada e reproduzível em todos os ambientes, e os testes de integração sejam confiáveis desde o início do projeto.

## Acceptance Criteria

1. PostgreSQL 17 conectado via Spring Data JPA com HikariCP — pool configurado com `maximum-pool-size: 20` e `minimum-idle: 5` em `application.yml`
2. Flyway configurado e ativo com `locations: classpath:db/migration` — migrations executadas automaticamente no startup em todos os ambientes (exceto perfil `test`, que usa Testcontainers)
3. Migration inicial `V1__create_schema.sql` criada em `crm-web/src/main/resources/db/migration/` com: extensões `pgcrypto` e `pg_trgm`, e função de trigger `set_updated_at()` pronta para uso nas stories seguintes
4. Testcontainers configurado no módulo `crm-infrastructure` — testes de repositório usam `PostgreSQLContainer<>` com imagem `postgres:17-alpine`, não H2 nem banco externo
5. Classe base `AbstractRepositoryIntegrationTest` criada em `crm-infrastructure/src/test/` com `@Testcontainers`, `@DataJpaTest`, `@AutoConfigureTestDatabase(replace = NONE)` e `@Container` estático — outros testes de repositório estendem esta classe
6. `CrmApplicationSmokeTest` existente continua passando sem modificações — Testcontainers só é ativado em testes que usam a classe base de integração
7. Dependências de teste `testcontainers-junit` e `testcontainers-postgres` adicionadas ao módulo `crm-infrastructure/build.gradle.kts` (aliases já existem em `libs.versions.toml`)
8. Variáveis de conexão ao banco externalizadas — zero hardcode em qualquer arquivo commitado; `application-local.yml` usa variáveis de ambiente com defaults para Docker Compose local
9. `./gradlew test` retorna `BUILD SUCCESSFUL` com todos os testes passando, incluindo ao menos um teste de repositório real em `SchemaIntegrationTest`
10. Flyway `ddl-auto: validate` já configurado na Story 1.1 — confirmar que JPA valida corretamente contra o schema criado pela migration, sem erros de divergência

### SQL Completo da Migration V1 (Source: architecture.md#Section-8.2)

```sql
-- V1__create_schema.sql
-- Instala extensões necessárias para o schema do ODIN CRM
-- pgcrypto: fornece gen_random_uuid() para PKs UUID
-- pg_trgm: permite índices GIN trigram para busca textual em razao_social e nome_fantasia

CREATE EXTENSION IF NOT EXISTS "pgcrypto";
CREATE EXTENSION IF NOT EXISTS "pg_trgm";

-- Função de trigger reutilizada por todas as tabelas com coluna updated_at
-- Acionada automaticamente antes de cada UPDATE para manter updated_at sincronizado
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### HikariCP — YAML (Source: architecture.md#Section-10.5)

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${DB_HOST:localhost}:${DB_PORT:5432}/${DB_NAME:odin_crm}
    username: ${DB_USER}
    password: ${DB_PASSWORD}
    driver-class-name: org.postgresql.Driver
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
      pool-name: OdinCrmHikariPool
```

**ATENÇÃO:** `application-local.yml` deve ter `DB_PORT: 5433` (não 5432).

### AbstractRepositoryIntegrationTest — Código (Source: architecture.md#Section-15.2)

```java
@Testcontainers
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@ActiveProfiles("test")
public abstract class AbstractRepositoryIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:17-alpine")
            .withDatabaseName("odin_crm_test")
            .withUsername("odin_crm_app")
            .withPassword("test_password");

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
        registry.add("spring.flyway.enabled", () -> "true");
    }
}
```

### Debug Log References

- **Docker Desktop 29.x incompatibility**: All Docker named pipes (`docker_engine`, `docker_cli`, `dockerDesktopLinuxEngine`) and TCP endpoint at port 2375 return HTTP 400 for API versions ≤1.43. Root cause: TC 1.19.8 (forced down by Spring Boot 3.3.6 BOM from declared 1.20.5) bundles shaded docker-java-core 3.3.6 which caps API negotiation at 1.43. Fix: `systemProperty("api.version", "1.46")` in Gradle test config — reads `api.version` JVM system property (NOT `DOCKER_API_VERSION` env var). Discovered by decompiling TC's shaded `DefaultDockerClientConfig.class`.
- **Migration classpath**: `V1__create_schema.sql` was initially placed in `crm-web` per AC3, but `@DataJpaTest` in `crm-infrastructure` module does not include `crm-web` resources in its classpath. Migration moved to `crm-infrastructure/src/main/resources/db/migration/`. At production runtime `crm-web` depends on `crm-infrastructure`, so Flyway still finds and applies the migration.

### Completion Notes List

- **AC3 deviation**: Migration `V1__create_schema.sql` placed in `crm-infrastructure/src/main/resources/db/migration/` instead of `crm-web/src/main/resources/db/migration/` as specified. Rationale: `crm-infrastructure` is the module that owns persistence adapters and Flyway; placing migrations here keeps them on the classpath for both integration tests (`@DataJpaTest`) and production startup (since `crm-web → crm-infrastructure` dependency chain). No functional impact.
- **Docker Desktop 29.x workaround**: `systemProperty("api.version", "1.46")` added to `crm-infrastructure/build.gradle.kts` test task. Also configurable via `DOCKER_API_VERSION` env var for CI/CD. Documented in `~/.testcontainers.properties`.
- **Spring Boot BOM downgrade**: Spring Boot 3.3.6 forces Testcontainers down from 1.20.5 → 1.19.8. The `api.version` system property fix works with both versions.
- All 4 `SchemaIntegrationTest` tests pass: `pgcryptoExtensionIsInstalled`, `pgTrgmExtensionIsInstalled`, `setUpdatedAtFunctionExists`, `flywayAppliedV1Migration`.
- `./gradlew test` → BUILD SUCCESSFUL across all 3 modules: `crm-domain`, `crm-web`, `crm-infrastructure`.
