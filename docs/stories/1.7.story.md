## Story

**As a** administrador,
**I want** gerenciar usuários e seus perfis de acesso no CRM,
**so that** cada colaborador tenha apenas as permissões necessárias para sua função.

## Acceptance Criteria

1. Tela `/admin/users` lista todos os usuários do realm `odin-crm` via Keycloak Admin REST API — exibindo nome, e-mail, cargo, roles atribuídas e status (ativo/inativo)
2. Admin atribui e revoga roles individuais por usuário (`crm-vendedor`, `crm-gerente`, `crm-sac`, `crm-financeiro`, `crm-admin`) — mudanças refletem imediatamente no Keycloak
3. Admin ativa/inativa usuários via toggle — operação é soft no Keycloak (`enabled: false`), preservando histórico de dados em `user_profiles`
4. Toda ação (atribuição/revogação de role, ativação/inativação) é realizada pelo backend via Keycloak Admin REST API com service account autenticado — frontend não tem acesso direto ao Keycloak Admin
5. Endpoint `GET /api/v1/admin/users` e endpoints de mutação retornam HTTP 403 para qualquer role diferente de `crm-admin`
6. Cada alteração de role ou status grava um registro em `audit_logs` com: `entity_type=USER`, `entity_id` (keycloak_id), `action` (ROLE_ASSIGNED/ROLE_REVOKED/USER_ENABLED/USER_DISABLED), `actor_id`, `old_values`, `new_values`

## Dev Notes

### Keycloak Admin REST API

O backend usa service account para autenticar no Keycloak Admin. O service account precisa das client roles `manage-users` e `view-users` do client `realm-management` no realm `master` (ou no realm `odin-crm`).

Endpoints Keycloak Admin usados:
```
GET    {KEYCLOAK_URL}/admin/realms/odin-crm/users
GET    {KEYCLOAK_URL}/admin/realms/odin-crm/users/{id}/role-mappings/realm
POST   {KEYCLOAK_URL}/admin/realms/odin-crm/users/{id}/role-mappings/realm   # body: [{id, name}]
DELETE {KEYCLOAK_URL}/admin/realms/odin-crm/users/{id}/role-mappings/realm   # body: [{id, name}]
PUT    {KEYCLOAK_URL}/admin/realms/odin-crm/users/{id}                       # body: {enabled: false}
POST   {KEYCLOAK_URL}/realms/master/protocol/openid-connect/token            # client_credentials
```

Token do service account deve ser renovado automaticamente (TTL típico 300s). Usar `RestClient` com interceptor de autenticação.

### Dependências Backend

Usar `RestClient` nativo do Spring Boot 3.2+ (não adicionar `keycloak-admin-client` para evitar conflito de versão com Spring Security).

```kotlin
// crm-infrastructure/build.gradle.kts — nenhuma dependência nova necessária
// RestClient já disponível via spring-boot-starter-web
```

**Propriedades de configuração** (`application.yml`):
```yaml
keycloak:
  admin:
    url: ${KEYCLOAK_URL:http://localhost:8180}
    realm: odin-crm
    client-id: ${KEYCLOAK_ADMIN_CLIENT_ID:odin-crm-backend}
    client-secret: ${KEYCLOAK_ADMIN_CLIENT_SECRET}
```

**ExternalSecret adicional** no Helm chart backend: `KEYCLOAK_ADMIN_CLIENT_SECRET` → `odin-crm/keycloak/admin-client-secret`

### Frontend — Guard de Role

A página `/admin/users` deve verificar `crm-admin` server-side:

```typescript
// app/(dashboard)/admin/users/page.tsx
import { auth } from '@/lib/auth'
import { redirect } from 'next/navigation'

export default async function AdminUsersPage() {
  const session = await auth()
  const roles = (session?.user as any)?.roles as string[] ?? []
  if (!roles.includes('crm-admin')) redirect('/dashboard')
  // ...
}
```

### Audit Log

Gravar antes da ação no Keycloak (garantia de não perder o registro em caso de falha na resposta):

```java
// Padrão de gravação em UserAdminService
auditLogRepository.save(buildAuditLog(actorId, "USER", keycloakId, "ROLE_ASSIGNED", oldRoles, newRoles));
keycloakAdminRestClient.addRoles(keycloakId, rolesToAdd);
```

`actor_id` extraído do JWT Bearer token via `SecurityContextHolder`.

## API Spec

| Method | Path | Description | Role |
|--------|------|-------------|------|
| `GET` | `/api/v1/admin/users` | Lista usuários com roles e status | `crm-admin` |
| `PATCH` | `/api/v1/admin/users/{keycloakId}/roles` | Atribui/revoga roles | `crm-admin` |
| `PATCH` | `/api/v1/admin/users/{keycloakId}/status` | Ativa/inativa usuário | `crm-admin` |

**GET /api/v1/admin/users — Response:**
```json
[
  {
    "keycloakId": "uuid",
    "nome": "João Silva",
    "email": "joao@empresa.com",
    "cargo": "Vendedor",
    "enabled": true,
    "roles": ["crm-vendedor"],
    "lastLogin": "2026-02-24T10:00:00Z"
  }
]
```

**PATCH /api/v1/admin/users/{id}/roles — Request:**
```json
{ "roles": ["crm-vendedor", "crm-gerente"] }
```

**PATCH /api/v1/admin/users/{id}/status — Request:**
```json
{ "enabled": false }
```
