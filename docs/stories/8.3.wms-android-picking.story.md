# Story 8.3: App Android WMS — Picking Guiado & Expedição

## Status

Ready for Review

## Story

**As a** operador de armazém,
**I want** um fluxo completo de picking guiado e expedição no aplicativo Android WMS, com listagem de tarefas de picking pendentes, navegação por corredor, confirmação de itens via scanner com alertas de FEFO/FIFO, e confirmação de carregamento de volumes no caminhão com encerramento do manifesto,
**so that** eu possa executar separação de pedidos e expedição diretamente do dispositivo móvel — com ou sem conexão de rede — garantindo conformidade FEFO/FIFO, rastreabilidade total de cada item separado e sincronização automática ao recuperar conectividade.

## Complexity

**Estimativa:** L (Large) · ~4-5 dias de desenvolvimento

[Fonte: EPIC-WMS-001 — Story 8.3 é o segundo fluxo operacional do Wave 8, construído sobre a fundação de recebimento da Story 8.2. Implementa dois fluxos completos: (1) Picking Guiado — listagem de tarefas PICKING_PENDING, detalhe com posições e itens esperados, confirmação de item pickado via scanner GS1 com alerta FEFO/FIFO, conclusão da separação completa; (2) Expedição — listagem de ordens SHIPPING_PENDING, confirmação de carregamento de pacotes (scan + placa), encerramento do manifesto. Reusa BarcodeAnalyzer, Gs1Parser, Hilt DI, Room, WorkManager e padrão MVVM + Clean Architecture da Story 8.2. Novas entidades Room: `picking_tasks_cache`, `picking_items_cache`, `picking_sync_queue`, `shipping_orders_cache`, `shipping_packages_cache`. Novo worker: `PickingSyncWorker`. APIs backend: 9 endpoints REST das Stories 5.1 e 5.3 já disponíveis. Corrige padrão QA-8.2-003 (cache retorna dados reais) e QA-8.2-004 (AtomicInteger para NOTIFICATION_ID).]

## Acceptance Criteria

### AC1 — Lista de Tarefas de Picking Pendentes

**Given** operador autenticado com role `WMS_OPERATOR` (ou superior),
**When** acessa a tela "Picking" via menu de Operações,
**Then**:
- Lista exibe tarefas com status `PICKING_PENDING` via `GET /api/v1/picking/orders?status=PICKING_PENDING&tenantId={tenantId}`
- Cada card de tarefa exibe: número da tarefa, corredor, total de itens, itens pickados (ex: "2/5 itens"), prioridade (badge: ALTA/NORMAL), status
- Pull-to-refresh atualiza a lista
- Em modo offline: lista vem do Room (`picking_tasks_cache`) com banner "Dados offline — mostrando cache local"
- Estado vazio (sem tarefas): mensagem "Nenhuma tarefa de picking pendente" com ícone de caixa vazia
- Erro de rede: mensagem de erro com botão "Tentar novamente"
- Dados do Room exibidos corretamente quando offline (corrige padrão QA-8.2-003)

---

### AC2 — Detalhe da Tarefa e Início do Picking

**Given** operador na lista de tarefas de picking,
**When** seleciona uma tarefa,
**Then**:
- Tela de detalhe exibe: cabeçalho da tarefa (número, corredor, prioridade), lista de itens esperados ordenados por posição
- Cada item exibe: código do produto, descrição, posição no armazém, quantidade esperada, quantidade pickada (inicialmente 0), status (PENDING / PICKED / SKIPPED)
- Indicação textual de rota: "Corredor {corridor} — Posição {position}" por item (sem mapa gráfico)
- Botão "Iniciar Picking" disponível para tarefas `PICKING_PENDING`
- Botão "Scan Item" abre `BarcodeScannerScreen` (reuso do componente 8.1/8.2) no modo picking
- Dados do detalhe carregados via `GET /api/v1/picking/orders/{id}`
- Detalhe cacheado em Room (`picking_items_cache`) para uso offline
- Itens com `expiryDate` < hoje+30 dias exibem badge de aviso FEFO (cor amarela)

---

### AC3 — Confirmação de Item Pickado (Scan + Validação)

**Given** operador na tela de detalhe da tarefa com item PENDING,
**When** escaneia código de barras do item ou da posição,
**Then**:
- `BarcodeAnalyzer` existente detecta o código sem modificação
- `Gs1Parser` extrai: GTIN-14 (AI 01), número de lote (AI 10), data de validade (AI 17), número de série (AI 21)
- Para código de posição (SSCC / GS1-128): `Gs1Parser` extrai identificador de posição; `PickingConfirmScreen` pré-preenche campo posição
- Tela de confirmação exibe: produto identificado (nome, GTIN), lote extraído do scan, validade extraída, posição, quantidade esperada
- Campo de quantidade pickada (numérico, pré-preenchido com `expectedQty` — editável pelo operador)
- Campos de lote e posição editáveis como fallback manual
- Botão "Confirmar Pick" chama `PUT /api/v1/picking/orders/{id}/items/{itemId}/pick` com `{ quantity, lotNumber, position, serialNumber }`
- Resposta de sucesso: item marcado como PICKED na lista; contagem de progresso atualizada
- Em offline: confirmação enfileirada em `picking_sync_queue` com status `PENDING`; item marcado localmente como `PICKED_OFFLINE`
- Feedback visual: checkmark verde + vibração háptica (100ms) ao confirmar

---

### AC4 — Alerta de FEFO/FIFO

**Given** operador confirmando item pickado em `PickingConfirmScreen`,
**When** o lote sendo confirmado não é o mais antigo disponível para aquele produto (verificação por `expiryDate` no `picking_items_cache`),
**Then**:
- Antes de confirmar, `PickingViewModel.confirmItemPicked()` verifica se existe item do mesmo produto com `expiryDate` anterior ao lote sendo confirmado
- Estado `FEFOWarning` emitido com dados do lote mais antigo disponível (lotNumber, expiryDate)
- `PickingConfirmScreen` exibe Snackbar: "⚠️ FEFO: Existe lote mais antigo ({lotNumber} / vence {expiryDate}). Confirme para continuar."
- Alerta é **não bloqueante** — operador pode prosseguir com a confirmação mesmo com o aviso exibido
- Botão "Confirmar Pick" permanece habilitado durante o aviso
- Se não há violação FEFO, o estado `FEFOWarning` não é emitido; confirmação prossegue normalmente
- `PickingItemDao.getPendingByProductSortedByExpiry(productCode, taskId)` retorna itens ordenados por `expiry_date ASC NULLS LAST`

---

### AC5 — Conclusão da Separação Completa

**Given** operador com todos os itens pickados na tarefa de picking,
**When** finaliza a separação,
**Then**:
- Botão "Concluir Separação" habilitado somente quando todos os itens tiverem status PICKED ou SKIPPED (nenhum PENDING)
- `PickingCompleteScreen` exibe resumo: total de itens pickados, itens skipped, corredor, duração da operação
- Botão "Confirmar Conclusão" chama `PUT /api/v1/picking/orders/{id}/complete`
- Em offline: operação enfileirada em `picking_sync_queue` com `operationType=COMPLETE`
- Resposta de sucesso: tarefa marcada como COMPLETED no Room; tela de sucesso exibe "Separação concluída! Tarefa #{numero}"
- Botão "Voltar para lista" retorna para `PickingListScreen`
- Tarefa concluída removida da lista de tarefas pendentes na próxima atualização

---

### AC6 — Lista de Ordens de Expedição

**Given** operador autenticado com role `WMS_OPERATOR` (ou superior),
**When** acessa a tela "Expedição" via menu de Operações ou via navegação de picking concluído,
**Then**:
- Lista exibe ordens com status `SHIPPING_PENDING` via `GET /api/v1/shipping/orders?status=SHIPPING_PENDING&tenantId={tenantId}`
- Cada card de ordem exibe: número da ordem, transportadora, placa do veículo (ou "—" se ausente), total de volumes, volumes carregados (ex: "1/3 volumes"), status
- Pull-to-refresh atualiza a lista
- Em modo offline: lista vem do Room (`shipping_orders_cache`) com banner "Dados offline"
- Estado vazio: mensagem "Nenhuma ordem de expedição pendente" com ícone
- Erro de rede: mensagem com botão "Tentar novamente"

---

### AC7 — Carregamento de Volumes e Confirmação de Manifesto

**Given** operador na tela de detalhe de uma ordem de expedição,
**When** confirma carregamento de volumes e fecha o manifesto,
**Then**:
- Tela de detalhe exibe: cabeçalho da ordem (número, transportadora, placa), lista de volumes/pacotes
- Cada pacote exibe: `trackingCode`, peso (se disponível), status (PENDING / LOADED)
- Botão "Scan Volume" abre `BarcodeScannerScreen` para ler `trackingCode` do pacote
- Após scan, `ShippingConfirmScreen` exibe: código de rastreio identificado, campo editável de placa do veículo (pré-preenchido se disponível na ordem)
- Botão "Confirmar Carregamento" chama `PUT /api/v1/shipping/orders/{id}/packages/{packageId}/load` com `{ trackingCode, vehiclePlate }`
- Pacote marcado como LOADED na lista; contagem de volumes carregados atualizada
- Quando todos os volumes estão LOADED: botão "Fechar Manifesto" habilitado
- Botão "Fechar Manifesto" chama `POST /api/v1/shipping/orders/{id}/complete`
- Resposta de sucesso: tela "Manifesto encerrado! Ordem #{numero}" com botão "Voltar para lista"
- Em offline: operações enfileiradas em `picking_sync_queue` com `operationType=LOAD_PACKAGE` ou `COMPLETE_SHIPPING`

---

### AC8 — Sincronização Offline (`PickingSyncWorker`)

**Given** dispositivo sem conectividade com operações de picking ou expedição enfileiradas,
**When** conectividade é restaurada,
**Then**:
- `PickingSyncWorker` (extends `CoroutineWorker`) ativado por `Constraints.Builder().setRequiredNetworkType(NetworkType.CONNECTED)`
- Worker processa `picking_sync_queue` em ordem FIFO por `createdAt`
- Para cada item da fila com status `PENDING`: executa chamada REST correspondente por `operationType` (CONFIRM_PICK / COMPLETE_TASK / LOAD_PACKAGE / COMPLETE_SHIPPING)
- Sucesso: item da fila marcado como `SYNCED`; cache Room atualizado com novo status
- Falha transitória (5xx / timeout): retry com backoff exponencial (máx. 3 tentativas); após: `SYNC_FAILED` + notificação local via `NotificationCompat`
- Falha permanente (4xx — 404 tarefa não existe): item marcado como `SYNC_CONFLICT`; log gravado; usuário notificado
- `NOTIFICATION_ID` gerenciado via `AtomicInteger` (corrige QA-8.2-004) para evitar IDs duplicados em contextos concorrentes
- Worker registrado com tag `"picking_sync"` (diferente de `"receiving_sync"` da Story 8.2 e `"general_sync"` da Story 8.1)
- Status de sync visível na tela de Perfil: "X itens de picking/expedição pendentes de sincronização"

### FEFO/FIFO Alert — Lógica de Referência

```kotlin
// PickingViewModel.kt — confirmItemPicked() com verificação FEFO:
fun confirmItemPicked(taskId: String, itemId: String, qty: Int, lotNumber: String?,
                      position: String, serialNumber: String?, expiryDate: LocalDate?) {
    viewModelScope.launch {
        // 1. FEFO check: buscar item mais antigo do mesmo produto na tarefa
        val item = pickingItemDao.getById(itemId) ?: return@launch
        val oldestLot = pickingItemDao.getPendingByProductSortedByExpiry(
            productCode = item.productCode,
            taskId = taskId
        ).firstOrNull()

        if (oldestLot != null && oldestLot.id != itemId &&
            oldestLot.expiryDate != null && expiryDate != null &&
            oldestLot.expiryDate.isBefore(expiryDate)) {
            // Emite aviso não-bloqueante; UI exibe Snackbar
            _uiState.value = PickingUiState.FEFOWarning(
                currentItem = item,
                olderLot = oldestLot.lotNumber ?: "?",
                olderExpiry = oldestLot.expiryDate
            )
            // Não retorna — confirmação prossegue após aviso ser exibido na UI
        }

        // 2. Confirmar via API (online) ou enfileirar (offline)
        val request = ConfirmPickItemRequestDto(
            quantity = qty, lotNumber = lotNumber,
            position = position, serialNumber = serialNumber
        )
        when (val result = pickingRepository.confirmItemPicked(taskId, itemId, request)) {
            is ApiResult.Success -> {
                pickingItemDao.updateLocalStatus(itemId, "PICKED")
                _uiState.value = PickingUiState.ConfirmSuccess
            }
            is ApiResult.NetworkError -> {
                pickingSyncQueueDao.insert(
                    PickingSyncQueueEntity(
                        tenantId = tokenProvider.getTenantId(),
                        operationType = "CONFIRM_PICK",
                        taskId = taskId,
                        itemId = itemId,
                        payload = json.encodeToString(request)
                    )
                )
                pickingItemDao.updateLocalStatus(itemId, "PICKED_OFFLINE")
                _uiState.value = PickingUiState.SyncQueued
            }
            is ApiResult.Error -> _uiState.value = PickingUiState.Error(result.message)
        }
    }
}
```

### PickingSyncWorker — NOTIFICATION_ID AtomicInteger (Fix QA-8.2-004)

```kotlin
// PickingSyncWorker.kt
@HiltWorker
class PickingSyncWorker @AssistedInject constructor(
    @Assisted context: Context,
    @Assisted workerParams: WorkerParameters,
    private val pickingSyncQueueDao: PickingSyncQueueDao,
    private val wmsApiService: WmsApiService
) : CoroutineWorker(context, workerParams) {

    companion object {
        const val TAG = "picking_sync"
        // FIX QA-8.2-004: AtomicInteger para NOTIFICATION_ID — evita IDs duplicados em contextos concorrentes
        private val NOTIFICATION_ID = AtomicInteger(2000)
    }

    override suspend fun doWork(): Result = withContext(Dispatchers.IO) {
        val tenantId = inputData.getString("tenantId") ?: return@withContext Result.failure()
        val pendingItems = pickingSyncQueueDao.getPendingByTenant(tenantId) // FIFO por createdAt

        for (item in pendingItems) {
            try {
                val response = when (item.operationType) {
                    "CONFIRM_PICK" -> {
                        val req = json.decodeFromString<ConfirmPickItemRequestDto>(item.payload)
                        wmsApiService.confirmItemPicked(item.taskId, item.itemId!!, req)
                    }
                    "COMPLETE_TASK" -> wmsApiService.completePickingOrder(item.taskId)
                    "LOAD_PACKAGE"  -> {
                        val req = json.decodeFromString<LoadPackageRequestDto>(item.payload)
                        wmsApiService.loadPackage(item.taskId, item.itemId!!, req)
                    }
                    "COMPLETE_SHIPPING" -> wmsApiService.completeShippingOrder(item.taskId)
                    else -> null
                }

                if (response?.isSuccessful == true) {
                    pickingSyncQueueDao.updateStatus(item.id, "SYNCED")
                } else if (response?.code() == 404) {
                    // Conflito permanente — não retenta
                    pickingSyncQueueDao.updateStatus(item.id, "SYNC_CONFLICT")
                    notifyUser("Conflito de sincronização", "Operação ${item.operationType} não encontrada remotamente.")
                } else {
                    handleRetry(item)
                }
            } catch (e: IOException) {
                handleRetry(item)
            }
        }
        Result.success()
    }

    private suspend fun handleRetry(item: PickingSyncQueueEntity) {
        if (item.retryCount >= 2) {
            pickingSyncQueueDao.updateStatus(item.id, "SYNC_FAILED")
            notifyUser("Falha na sincronização", "Operação ${item.operationType} falhou após 3 tentativas.")
        } else {
            pickingSyncQueueDao.incrementRetryCount(item.id)
        }
    }

    private fun notifyUser(title: String, message: String) {
        val notification = NotificationCompat.Builder(applicationContext, "picking_sync_channel")
            .setSmallIcon(android.R.drawable.ic_dialog_alert)
            .setContentTitle(title)
            .setContentText(message)
            .setPriority(NotificationCompat.PRIORITY_DEFAULT)
            .build()
        NotificationManagerCompat.from(applicationContext)
```

```kotlin
override suspend fun getTasks(tenantId: String): ApiResult<List<PickingTask>> {
    return try {
        val response = wmsApiService.getPickingOrders(tenantId = tenantId)
        if (response.isSuccessful) {
            val tasks = response.body()?.map { it.toDomain() } ?: emptyList()
            // Salvar no Room para uso offline
            pickingTaskDao.insertOrUpdateAll(tasks.map { it.toCacheEntity() })
            ApiResult.Success(tasks, isOffline = false)
        } else {
            loadTasksFromCache(tenantId)
        }
    } catch (e: IOException) {
        // FIX QA-8.2-003: retornar dados reais do Room, não lista vazia
        loadTasksFromCache(tenantId)
    }
}

private suspend fun loadTasksFromCache(tenantId: String): ApiResult<List<PickingTask>> {
    // FIX QA-8.2-003: consultar Room e retornar dados cacheados (não emptyList())
    val cached = pickingTaskDao.getByTenantId(tenantId)
    val tasks = cached.map { it.toDomain() }
    return ApiResult.Success(tasks, isOffline = true)
}
```

## Testing

### Estratégia de Testes

**Unit Tests (13 testes — JUnit 5 + MockK):**
- `@ExtendWith(MockKExtension::class)` + `kotlinx-coroutines-test` (`StandardTestDispatcher`)
- Mocks: `IPickingRepository`, `IShippingRepository`, `PickingTaskDao`, `PickingItemDao`, `PickingSyncQueueDao`, `ShippingOrderDao`, `ShippingPackageDao`, `WmsApiService`, `TokenProvider`
- Foco: `Gs1Parser` (SSCC parse), `PickingViewModel` (FEFO warning + offline + states), `PickingSyncWorker` (retry + conflict + failure + AtomicInteger), `PickingRepository` (API-first + cache fix QA-8.2-003), `ShippingViewModel` (loadPackage state)
- `TestCoroutineRule` para controle de coroutines

**Instrumented Tests (6 testes — Espresso + Compose Testing):**
- Execução em emulador Android API 26+ (não executável localmente no Windows — AGP limitation com path não-ASCII)
- `@get:Rule val composeTestRule = createAndroidComposeRule<MainActivity>()`
- Foco: UI rendering das 8 telas novas, badge FEFO, campo pré-preenchido, botão condicional, navegação
- Dados mockados via `@HiltAndroidTest` + `TestModule` substituindo `PickingModule`

**Ferramentas:**
- JUnit 5 + MockK 1.13.x para unit tests
- `kotlinx-coroutines-test` 1.8.x para coroutines
- `androidx.compose.ui:ui-test-junit4` para Compose testing
- `androidx.work:work-testing` para `PickingSyncWorker` unit tests

**Limitações conhecidas:**
- Testes não executam localmente no Windows (path não-ASCII + AGP). Executar em CI/CD Linux ou emulador via Android Studio.
- Vibração háptica não testável em unit tests; validada em smoke test manual no dispositivo.
