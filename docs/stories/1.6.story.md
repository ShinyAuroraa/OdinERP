## Story

**As a** desenvolvedor,
**I want** pipeline automatizado que valide, construa e faça deploy do CRM no Kubernetes,
**so that** cada merge na branch principal resulte em deploy confiável sem intervenção manual.

## Acceptance Criteria

1. GitHub Actions com stages: `lint` → `test` → `build` → `docker-push` → `deploy` — backend e frontend com workflows distintos em `.github/workflows/backend-ci.yml` e `.github/workflows/frontend-ci.yml`
2. Build backend (Gradle) e frontend (pnpm/Next.js) em jobs paralelos dentro do mesmo pipeline — jobs independentes rodando em `ubuntu-latest`
3. Imagem Docker publicada no AWS ECR com tag do commit SHA: `{ECR_REGISTRY}/odin-crm-{backend|frontend}:{GITHUB_SHA}` — apenas em push para `master`
4. Helm charts com templates Kubernetes: `Deployment`, `Service`, `Ingress`, `ConfigMap`, `ExternalSecret` (AWS Secrets Manager), `HPA`, `PDB` — para `odin-crm-backend` e `odin-crm-frontend`
5. ArgoCD `Application` manifests configurados para sync automático do namespace `odin-crm-staging` ao detectar mudança em `values-staging.yaml`
6. Pod do backend e do frontend em estado `Running` e `Ready` após deploy no namespace `odin-crm-staging`
7. URL pública `https://crm-staging.odin.internal` acessível com certificado TLS válido via AWS ALB + ACM

## Scope

### IN (esta story)
- `.github/workflows/backend-ci.yml` — pipeline completo do backend (test, build, docker-push, deploy)
- `.github/workflows/frontend-ci.yml` — pipeline completo do frontend (lint, build, docker-push, deploy)
- Helm chart completo para `odin-crm-backend` com todos os templates Kubernetes
- Helm chart completo para `odin-crm-frontend` com todos os templates Kubernetes
- ArgoCD `Application` manifests para namespace `odin-crm-staging`

### Monorepo vs Polyrepo — Desvio Crítico da Arquitetura
[Source: architecture.md#Section-11 — desvio documentado]

A arquitetura descreve **polyrepo** (dois repositórios separados). A implementação real é um **monorepo** (`OdinERP` em `ShinyAuroraa/OdinERP`). Consequências:

1. `.github/workflows/` fica na raiz do monorepo (`meu-projeto/.github/workflows/`), não em cada subdiretório
2. Path filters obrigatórios nos triggers para evitar que mudança no frontend dispare CI do backend e vice-versa
3. ArgoCD `source.path` deve incluir o subdiretório: `odin-crm-backend/helm/odin-crm-backend`
4. Job `deploy` que comita `values-staging.yaml` deve usar `[skip ci]` para evitar loop infinito de CI

### AWS Security
[Source: architecture.md#Section-14.1]

- Istio mTLS `STRICT` em todos os namespaces `odin-*`
- GitHub Actions → AWS via **OIDC** (sem access keys hardcoded no repositório)
- GitHub Actions OIDC IAM role trust policy:
  ```json
  {
    "Effect": "Allow",
    "Principal": { "Federated": "arn:aws:iam::{ACCOUNT}:oidc-provider/token.actions.githubusercontent.com" },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com",
        "token.actions.githubusercontent.com:sub": "repo:ShinyAuroraa/OdinERP:ref:refs/heads/master"
      }
    }
  }
  ```
- IAM permissions necessárias: `ecr:GetAuthorizationToken`, `ecr:BatchCheckLayerAvailability`, `ecr:PutImage`, `ecr:InitiateLayerUpload`, `ecr:UploadLayerPart`, `ecr:CompleteLayerUpload`

### Riscos Conhecidos

| Risco | Mitigação |
|-------|-----------|
| CI loop: job `deploy` commita `values-staging.yaml` → dispara novo CI | Usar `[skip ci]` no commit message do job deploy |
| OIDC role trust policy incorreta → AWS auth falha no CI | Configurar IAM OIDC provider `token.actions.githubusercontent.com` antes de executar o workflow |
| External Secrets Operator não instalado → ExternalSecret CRD ausente | `kubectl get crd externalsecrets.external-secrets.io` antes de aplicar manifests; instalar via Helm se ausente |
| ArgoCD source.path aponta para diretório errado no monorepo | Testar `argocd app create` manualmente antes de commitar o manifest |
| `values-staging.yaml` commit pelo CI: "Author identity unknown" | Configurar `git config user.email github-actions@github.com` e `user.name "GitHub Actions"` no job |
| Frontend ConfigMap: `KEYCLOAK_ISSUER` aponta para `localhost:8180` (dev) | Atualizar para URL real do Keycloak no EKS (`https://keycloak.odin.internal/realms/odin`) antes do deploy |
| `NEXTAUTH_URL` no ConfigMap deve corresponder à URL real do frontend | Verificar que `crm-staging.odin.internal` resolve via Route 53 antes do deploy |
