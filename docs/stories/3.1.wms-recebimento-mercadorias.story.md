# Story 3.1: Recebimento de Mercadorias (Inbound — Integração SCM)

## Status

Ready for Review

## Story

**As a** operador de armazém (WMS Operator),
**I want** registrar e confirmar o recebimento de mercadorias vindas de pedidos de compra do SCM — conferindo itens por quantidade, lote, validade e número de série, e sinalizando divergências para revisão do supervisor,
**so that** cada mercadoria recebida gere um `StockItem` rastreável na doca de recebimento, um `Movement(INBOUND)` no histórico e um registro no `AuditLog`, habilitando o fluxo de putaway (Story 3.2) e tornando o estoque visível em tempo real (Story 4.1).

## Business Value

**Por que esta story é o coração da Wave 3:**

- **Inbound é o ponto zero do estoque:** Sem recebimento, nenhum StockItem existe. Todas as operações de armazém (picking, inventário, expedição) dependem de estoque previamente recebido.
- **Rastreabilidade desde a origem:** Lote e validade capturados no recebimento são a base para FEFO no putaway (Story 3.2) e para callbacks de recall (Story 7.1).
- **Integração SCM↔WMS:** O Kafka consumer desta story fecha o loop de pedidos de compra — o comprador cria o PO no SCM, e o WMS automaticamente prepara a nota de recebimento, eliminando entrada manual duplicada.
- **Compliance obrigatório:** Indústrias farmacêutica e alimentícia exigem rastreabilidade por lote e validade desde o momento de entrada no armazém — esta story é o gate.

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Kafka consumer processa evento duplicado (retry do SCM) | Alta | Médio | Idempotência via `purchaseOrderRef` único por tenant — se nota já existe, ignorar evento (upsert) |
| `StockItem.location` obrigatório mas dock não cadastrada | Média | Alto | Validar no `complete()`: warehouse deve ter ao menos 1 location com `type=RECEIVING_DOCK`; retornar 422 claro se ausente |
| `ProductWms.controlsLot=true` mas lote não informado | Alta | Alto | Validação no `confirmItem()`: se `product.controlsLot && lotNumber == null` → 422 com mensagem explícita |
| `ProductWms.controlsExpiry=true` com data no passado | Média | Alto | Validar `expiryDate > LocalDate.now()` em `confirmItem()`; bloquear recebimento (produto já vencido) |
| Concorrência: dois operadores confirmando o mesmo item | Baixa | Alto | `@Version` em `ReceivingNoteItem` via `BaseEntity`; otimistic locking resolve |
| Kafka offline / SCM indisponível | Média | Baixo | Fallback: criação manual via `POST /api/v1/receiving-notes`; Kafka é canal adicional, não único |

## Acceptance Criteria

1. **Flyway V4 — Novas tabelas de recebimento:**
   - `receiving_notes`: `id` UUID PK, `tenant_id` UUID NOT NULL, `warehouse_id` UUID NOT NULL FK→warehouses, `dock_location_id` UUID NOT NULL FK→locations, `purchase_order_ref` VARCHAR(100) NULLABLE, `supplier_id` UUID NULLABLE, `status` VARCHAR(30) NOT NULL DEFAULT `'PENDING'`, `created_at` TIMESTAMP, `updated_at` TIMESTAMP
   - `receiving_note_items`: `id` UUID PK, `tenant_id` UUID NOT NULL, `receiving_note_id` UUID NOT NULL FK→receiving_notes, `product_id` UUID NOT NULL FK→products_wms, `expected_quantity` INT NOT NULL CHECK(>0), `received_quantity` INT NULLABLE, `divergence_type` VARCHAR(20) DEFAULT `'NONE'`, `lot_number` VARCHAR(100) NULLABLE, `manufacturing_date` DATE NULLABLE, `expiry_date` DATE NULLABLE, `gs1_code` VARCHAR(200) NULLABLE, `item_status` VARCHAR(20) NOT NULL DEFAULT `'PENDING'`, `lot_id` UUID NULLABLE FK→lots, `stock_item_id` UUID NULLABLE FK→stock_items, `created_at` TIMESTAMP, `updated_at` TIMESTAMP
   - Índices: `idx_receiving_notes_tenant_status`, `idx_receiving_notes_purchase_order_ref`, `idx_receiving_note_items_note_id`

2. **Criação manual de nota de recebimento** — `POST /api/v1/receiving-notes`:
   - Recebe `CreateReceivingNoteRequest`: `warehouseId` (UUID, obrigatório), `dockLocationId` (UUID, obrigatório — deve ser `LocationType.RECEIVING_DOCK` do mesmo warehouse e tenant), `supplierId` (UUID, opcional), `purchaseOrderRef` (String, opcional), `items` (lista com `productId`, `expectedQuantity`)
   - Retorna 201 com `ReceivingNoteResponse` completo (status=`PENDING`)
   - `dockLocationId` de outro warehouse/tenant → 422 `"Dock location não pertence ao warehouse informado"`
   - `productId` inexistente no tenant → 422 `"Produto não encontrado: {productId}"`
   - Role requerida: `WMS_OPERATOR`, `WMS_SUPERVISOR` ou `WMS_ADMIN`

3. **Kafka consumer — criação automática via SCM:**
   - Consumer escutando tópico `scm.purchase-orders.confirmed` (grupo `wms-receiving`)
   - Mensagem JSON: `{ "purchaseOrderId": "...", "tenantId": "...", "warehouseId": "...", "supplierId": "...", "items": [{"productId": "...", "expectedQuantity": N}] }`
   - Idempotente: se `purchaseOrderRef` já existe para o tenant → log de aviso, ignorar (não criar duplicata)
   - Se `warehouseId` não encontrado no tenant → log de erro, **não** lançar exception (não envenenar a fila); publicar no Dead Letter Topic `wms.receiving.dlq`
   - Se `productId` não encontrado no tenant → mesma política (DLQ), não parar o consumer
   - `dockLocationId` default: primeira location do warehouse com `type = RECEIVING_DOCK`; se nenhuma → DLQ com motivo

4. **Início da conferência** — `POST /api/v1/receiving-notes/{id}/start`:
   - Muda status da nota: `PENDING` → `IN_PROGRESS`
   - Apenas notas com status `PENDING` podem ser iniciadas → 409 Conflict se outro status
   - Nota de outro tenant → 404
   - Role requerida: `WMS_OPERATOR+`

5. **Confirmação de item** — `POST /api/v1/receiving-notes/{id}/items/{itemId}/confirm`:
   - Recebe `ConfirmReceivingItemRequest`: `receivedQuantity` (Int, @Min(0)), `lotNumber` (String, opcional), `manufacturingDate` (LocalDate, opcional), `expiryDate` (LocalDate, opcional), `gs1Code` (String, opcional), `serialNumbers` (List<String>, opcional)
   - Validações obrigatórias por configuração de produto:
     - `ProductWms.controlsLot = true` → `lotNumber` obrigatório; se ausente → 422 `"Produto requer lote obrigatório"`
     - `ProductWms.controlsExpiry = true` → `expiryDate` obrigatório E `expiryDate > LocalDate.now()`; se vencido → 422 `"Produto com validade expirada não pode ser recebido"`
     - `ProductWms.controlsSerial = true` → `serialNumbers` não nulo e `serialNumbers.size() == receivedQuantity`
   - `receivedQuantity != expectedQuantity` → `divergenceType` calculado automaticamente: `SHORT` (recv<exp), `EXCESS` (recv>exp), `DAMAGED` (quando qty=0)
   - `receivedQuantity = 0` → `item_status = FLAGGED`, `divergenceType = DAMAGED`
   - `receivedQuantity > 0 e != expectedQuantity` → `item_status = CONFIRMED`, `divergenceType = SHORT|EXCESS` (divergência registrada mas não bloqueia)
   - `receivedQuantity = expectedQuantity` → `item_status = CONFIRMED`, `divergenceType = NONE`
   - Nota precisa estar com status `IN_PROGRESS`; se outro status → 409
   - Role requerida: `WMS_OPERATOR+`

6. **Conclusão da nota** — `POST /api/v1/receiving-notes/{id}/complete`:
   - Pré-condições verificadas:
     - Todos os `ReceivingNoteItem` com `item_status ∈ {CONFIRMED, FLAGGED}` (nenhum `PENDING`)
     - Se algum item `FLAGGED`: status da nota → `COMPLETED_WITH_DIVERGENCE` (requer flag de supervisão); itens FLAGGED **não** geram StockItem
     - Se todos `CONFIRMED (NONE)`: status → `COMPLETED`
   - Para cada item com `item_status = CONFIRMED` e `receivedQuantity > 0`:
     - Se `product.controlsLot`: buscar `Lot` por (`product`, `lotNumber`, `tenantId`) — criar se não existir (com `expiryDate`, `manufacturingDate`, `supplierId`)
     - Criar `StockItem` com: `location = dockLocation`, `product`, `lot` (ou null), `quantityAvailable = receivedQuantity`, `receivedAt = Instant.now()`
     - Criar `StockMovement` com: `type = INBOUND`, `toLocation = dockLocation`, `quantity = receivedQuantity`, `referenceType = RECEIVING_NOTE`, `referenceId = note.id`
     - Criar `AuditLog` com: `action = CREATE`, `entityType = STOCK_ITEM`, `entityId = stockItem.id`, `userId = JWT.sub`, `tenantId`
     - Se `product.controlsSerial`: criar `SerialNumber` para cada serial em `serialNumbers`
     - Linkar: `ReceivingNoteItem.stockItemId = stockItem.id`, `ReceivingNoteItem.lotId = lot.id`
   - Retorna 200 com `ReceivingNoteResponse` completo
   - Role requerida: `WMS_OPERATOR+`

7. **Aprovação de divergências** — `POST /api/v1/receiving-notes/{id}/approve-divergences`:
   - Disponível apenas para notas com status `COMPLETED_WITH_DIVERGENCE`
   - Para cada item `FLAGGED`: cria `StockItem` com `quantityDamaged = receivedQuantity` (qty=0 → nenhum StockItem)
   - Status da nota → `COMPLETED`
   - Role requerida: `WMS_SUPERVISOR` ou `WMS_ADMIN` (OPERATOR → 403)

8. **Listagem e consulta:**
   - `GET /api/v1/receiving-notes` — lista notas do tenant com filtros `?status=PENDING|IN_PROGRESS|COMPLETED|COMPLETED_WITH_DIVERGENCE&warehouseId={id}` — Role: `WMS_OPERATOR+`
   - `GET /api/v1/receiving-notes/{id}` — detalhe completo com items — Role: `WMS_OPERATOR+`
   - Nota de outro tenant → 404

9. **Segurança e isolamento multi-tenant:**
   - Toda operação usa `TenantContextHolder.getTenantId()` para filtrar e validar dados
   - JWT ausente → 401; Role insuficiente → 403
   - `warehouseId`, `dockLocationId`, `productId` de outro tenant → 422 ou 404 (não vazar informação)
   - Kafka consumer: `tenantId` lido do payload JSON (não do JWT — consumer roda sem contexto HTTP)

10. **Testes obrigatórios:**
    - Unit tests em `ReceivingNoteServiceTest`:
      - `createNote_success()`, `createNote_invalidDockLocation_throws422()`
      - `confirmItem_success_noLot()`, `confirmItem_requiresLot_lotMissing_throws422()`, `confirmItem_expiredProduct_throws422()`
      - `completeNote_allConfirmed_createsStockItems()`, `completeNote_withFlagged_statusDivergence()`
      - `approveDivergences_operatorRole_throws403()`
    - Integration tests em `ReceivingNoteControllerIntegrationTest`:
      - `createNote_returns201()`, `startConference_returns200()`, `confirmItem_returns200()`
      - `confirmItem_missingLotForTrackedProduct_returns422()`
      - `completeNote_allConfirmed_createsStockItems_returns200()`
      - `approveWithOperatorRole_returns403()`, `approveWithSupervisorRole_returns200()`
    - Kafka test em `ReceivingNoteKafkaConsumerTest` com `@EmbeddedKafka`:

### Padrão de Isolamento Multi-tenant no Consumer Kafka

O Kafka consumer não tem contexto HTTP, portanto não há JWT disponível. O `tenantId` deve ser lido **do payload JSON** do evento, não do `TenantContextHolder`. Dentro do consumer, setar manualmente:

```java
@KafkaListener(topics = "scm.purchase-orders.confirmed", groupId = "wms-receiving")
public void consume(PurchaseOrderConfirmedEvent event) {
    try {
        TenantContextHolder.setTenantId(event.tenantId()); // setar manualmente
        receivingNoteService.createFromEvent(event);
    } catch (Exception e) {
        log.error("Erro processando PO {}: {}", event.purchaseOrderId(), e.getMessage());
        deadLetterKafkaTemplate.send("wms.receiving.dlq", event);
    } finally {
        TenantContextHolder.clear(); // CRÍTICO: limpar após processamento
    }
}
```

### Concorrência e Optimistic Locking

`ReceivingNoteItem` herda `@Version` de `BaseEntity`. Se dois operadores tentarem confirmar o mesmo item simultaneamente, o segundo receberá `OptimisticLockException` — tratar com 409 Conflict no `@ExceptionHandler`.
