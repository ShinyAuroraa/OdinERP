# Story 10.2: Gestão de Produtos WMS — CRUD + Controles Lote/Serial

## Status

Done

## Story

**As a** administrador de armazém (WMS_ADMIN),
**I want** gerenciar produtos no contexto WMS via interface web — com SKU, atributos de armazenagem, configuração de controles (lote, serial, validade) e informações GS1,
**so that** operadores possam ver os produtos cadastrados, seus controles de rastreabilidade e configurações de armazenagem antes das operações de recebimento e picking.

## Complexity

**Estimativa:** M (Medium) · ~3-4 dias de desenvolvimento
[CRUD + formulário rico com controles condicionais + StorageType + GS1 fields]

## Dependencies

**Pré-requisitos:**
- Story 9.1 concluída ✅ — Next.js 15 + Keycloak PKCE + Layout Shell + API client
- Story 9.2 concluída ✅ — Design System completo
- Backend STORY-WMS-2.2 Done ✅ — API `/api/v1/products`

## Out of Scope

- ❌ Importação em lote CSV/Excel de produtos
- ❌ Geração automática de código GS1-128 / QR Code
- ❌ Sincronização com cadastro master de produtos (ERP)
- ❌ Gestão de imagens de produto
- ❌ Conversão de unidades (caixa → unidade)

## Acceptance Criteria

1. **Página `/products`** — lista de produtos WMS com DataTable:
   - Colunas: SKU, Nome, StorageType (badge), Controles (ícones: lote/serial/validade), Status, Ações
   - Busca global (SKU, nome) + filtro por StorageType + filtro por status
   - Botão "Novo Produto" (apenas WMS_ADMIN)
   - Loading skeleton + empty state "Nenhum produto cadastrado"

2. **Formulário de Produto** (modal ou page `/products/new`):
   - **Campos obrigatórios:** nome*, SKU*, storageType*
   - **Campos opcionais:** productId (UUID externo), EAN-13, GS1-128, description
   - **Dimensões:** comprimento, largura, altura (m), peso (kg), unitsPerLocation
   - **Controles de rastreabilidade** (checkboxes):
     - `controlsLot` — Controla por lote
     - `controlsSerial` — Controla por número de série
     - `controlsExpiry` — Controla data de validade
     - `requiresSanitaryVigilance` — Vigilância sanitária (ANVISA)
   - **StorageType** select: DRY, REFRIGERATED, FROZEN, HAZARDOUS, OVERSIZED, CONTROLLED
   - Validação Zod: SKU único por tenant, dimensões ≥ 0
   - Submit → `POST /api/v1/products` → toast sucesso/erro
   - Editar → `PUT /api/v1/products/{id}` → invalidate lista

3. **Página de detalhe `/products/{id}`**:
   - PageHeader: SKU + nome do produto + status badge
   - Card "Informações Gerais": storageType, dimensões, peso, unitsPerLocation
   - Card "Rastreabilidade": badges para controles ativos (lote, serial, validade, ANVISA)
   - Card "Códigos GS1": EAN-13, GS1-128 (se configurados)
   - Botão "Editar" (WMS_ADMIN) + Botão "Desativar" com ConfirmDialog destructive

4. **React Query hooks** — `lib/api/products.ts`:
   - `useProducts(filters?)` — GET /api/v1/products, staleTime 5min
   - `useProduct(id)` — GET /api/v1/products/{id}
   - `useCreateProduct` — POST /api/v1/products
   - `useUpdateProduct` — PUT /api/v1/products/{id}
   - `useDeleteProduct` — DELETE /api/v1/products/{id}

5. **TypeScript types** — `types/product.ts`:
   - `ProductWms` interface com todos os campos do backend
   - `StorageType` enum: DRY, REFRIGERATED, FROZEN, HAZARDOUS, OVERSIZED, CONTROLLED
   - `CreateProductRequest`, `UpdateProductRequest` (omit id)

6. **Proteção por role**:
   - Listagem + Detalhe: todos os roles
   - CRUD: apenas WMS_ADMIN
   - Botões de ação ocultos para roles sem permissão

7. **Testes unitários** (≥ 10 novos):
   - ProductList: render, loading, empty, busca por SKU, filtro StorageType
   - ProductForm: submit válido, erro SKU vazio, controles condicionais visíveis
   - ProductDetail: render campos, badge controles ativos
   - Hooks: useProducts mock
