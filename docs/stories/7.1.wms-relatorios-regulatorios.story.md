# Story 7.1: Relatórios Regulatórios — Ficha de Estoque, ANVISA, Rastreabilidade & Movimentações

## Status

Ready for Review

## Story

**As a** gerente de compliance / gestor de armazém,
**I want** gerar relatórios regulatórios sob demanda e de forma agendada — Ficha de Estoque (Receita Federal), Controle de Vigilância Sanitária (ANVISA), Rastreabilidade por Lote para recalls e Relatório de Movimentações por período — exportáveis em PDF, Excel e XML,
**so that** o Odin ERP atenda aos requisitos legais brasileiros, permita auditorias regulatórias com dados precisos e rastreáveis, e garanta conformidade com Receita Federal e ANVISA sem intervenção manual.

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Performance: Apache POI (`XSSFWorkbook`) em memória para datasets grandes (>50k linhas) | Média | Alto | Usar `SXSSFWorkbook` (streaming) para Excel; limitar PDF a 10k linhas por chamada com aviso ao usuário |
| Scheduler em múltiplas instâncias (K8s): dois pods executam o mesmo schedule simultaneamente | Alta | Médio | Usar `@SchedulerLock` (ShedLock) ou Spring Integration Lock; ou garantir `fixedDelay` + lock pessimista no `report_schedules.next_execution_at` via SQL `UPDATE ... WHERE next_execution_at <= now` |
| Formato regulatório desatualizado (Receita Federal muda layout da Ficha de Estoque) | Baixa | Alto | Isolar lógica de formatação nos generators — alteração de layout não afeta infra; configurável por tenant |
| `CronExpression.parse()` do Spring não suporta todos os padrões Quartz (ex: `L`, `W`, `#`) | Média | Baixo | Documentar que aceita apenas Spring cron (6 campos); retornar `400` com mensagem explicativa |

## Acceptance Criteria

### AC1 — `GET /api/v1/reports/ficha-estoque` — Ficha de Estoque (Receita Federal)

**Given** um usuário autenticado com role `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `GET /api/v1/reports/ficha-estoque?warehouseId={uuid}&dataInicio={date}&dataFim={date}&format=JSON`,
**Then**:
- `warehouseId` obrigatório; `productId` e `lotId` opcionais como filtros adicionais
- `dataInicio` e `dataFim` obrigatórios (formato `yyyy-MM-dd`); período máximo 365 dias
- Retorna: por produto — saldo inicial do período, entradas (recebimentos), saídas (picking/shipping/produção), ajustes de inventário, saldo final
- Todos os valores em unidades inteiras; inclui `productId`, `sku`, `productName`, `warehouseId`, `periodoInicio`, `periodoFim`
- Multi-tenant: filtra por `tenantId` do JWT
- Request de outro tenant → `403 Forbidden`; `warehouseId` inválido → `400 Bad Request`
- `format=JSON` (padrão) → `200 OK` com body JSON; outros formatos → ver AC5

---

### AC2 — `GET /api/v1/reports/anvisa` — Controle de Vigilância Sanitária

**Given** um usuário autenticado com role `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `GET /api/v1/reports/anvisa?warehouseId={uuid}&dataInicio={date}&dataFim={date}`,
**Then**:
- Retorna apenas produtos onde `ProductWms.vigilanciaSanitaria = true`
- Por produto/lote: `sku`, `productName`, `lotNumber`, `expiryDate`, `quantityReceived`, `quantityExpedited`, `currentBalance`, `supplier`, `receivedAt`
- Se nenhum produto com `vigilanciaSanitaria = true` no tenant → retorna lista vazia `[]` com `200 OK`
- Filtros: `lotId` (opcional), `productId` (opcional)
- Multi-tenant isolation obrigatório
- Mesmas regras de formato que AC5

---

### AC3 — `GET /api/v1/reports/rastreabilidade-lote/{lotId}` — Rastreabilidade para Recall

**Given** um usuário autenticado com role `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `GET /api/v1/reports/rastreabilidade-lote/{lotId}`,
**Then**:
- Busca `Lot` por `(id, tenantId)` → se não encontrado → `404 Not Found`; se outro tenant → `403 Forbidden`
- Retorna árvore de rastreabilidade completa do lote:
  - `loteInfo`: lotNumber, productId, sku, productName, expiryDate, supplierName, receivedAt
  - `movimentacoes`: lista cronológica de todos os `StockMovement` relacionados ao lote (entrada, putaway, picking, transferência, expedição, ajuste)
  - `localização atual`: localização atual do estoque remanescente do lote (se houver saldo)
  - `expedições`: lista de ordens de picking/shipping que incluíram o lote (para recall de destino)
- Formato padrão JSON; suporta `?format=PDF|EXCEL|XML` via AC5

---

### AC4 — `GET /api/v1/reports/movimentacoes` — Movimentações por Período

**Given** um usuário autenticado com role `WMS_OPERATOR`, `WMS_SUPERVISOR` ou `WMS_ADMIN`,
**When** chama `GET /api/v1/reports/movimentacoes?warehouseId={uuid}&dataInicio={date}&dataFim={date}`,
**Then**:
- Filtros opcionais: `productId`, `movementType` (RECEIVING/OUTBOUND/PICKING/TRANSFER/ADJUSTMENT/etc.), `locationId`
- Retorna lista paginada (`page`, `size`) de movimentações: `movementId`, `movementType`, `referenceType`, `referenceId`, `productId`, `sku`, `lotId`, `locationId`, `quantity`, `performedBy`, `createdAt`
- Ordenação padrão: `createdAt DESC`
- Multi-tenant isolation obrigatório
- Período máximo 365 dias; se exceder → `400 Bad Request`

---

### AC5 — Formatos de Exportação: PDF, Excel, XML, JSON

**Given** qualquer endpoint de relatório (AC1-AC4),
**When** o parâmetro `?format=` é especificado,
**Then**:
- `format=JSON` (padrão) → `Content-Type: application/json` → body JSON estruturado
- `format=PDF` → `Content-Type: application/pdf` → arquivo PDF com tabela do relatório, cabeçalho (nome do relatório, tenant, período, data de geração), rodapé com paginação
- `format=EXCEL` → `Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` → arquivo .xlsx com aba para o relatório, cabeçalho em negrito, auto-filter ativo
- `format=XML` → `Content-Type: application/xml` → XML com elemento raiz `<relatorio>`, atributos de metadados (tipo, periodo, tenantId, geradoEm), elementos filhos por linha
- `format` inválido → `400 Bad Request` com mensagem clara
- O header `Content-Disposition: attachment; filename="relatorio-{tipo}-{dataGeracao}.{ext}"` deve estar presente para PDF e EXCEL

---

### AC6 — Agendamento de Relatórios (`/api/v1/reports/schedules`)

**Given** um usuário com role `WMS_ADMIN`,
**When** opera os endpoints de agendamento:
**Then**:

**POST /api/v1/reports/schedules** → cria agendamento:
- Body: `{ "reportType": "FICHA_ESTOQUE", "format": "PDF", "cronExpression": "0 0 8 * * MON", "warehouseId": "uuid", "filters": {} }`
- Valida `cronExpression` com Quartz ou Spring CronExpression parser → `400` se inválida
- Salva em `report_schedules` com `tenantId` do JWT
- Retorna `201 Created` com `ReportScheduleResponse`

**GET /api/v1/reports/schedules** → lista agendamentos do tenant:
- Retorna lista paginada dos agendamentos ativos do tenant
- Roles: `WMS_SUPERVISOR` e `WMS_ADMIN`

**DELETE /api/v1/reports/schedules/{id}** → remove agendamento:
- `403` se pertencer a outro tenant; `404` se não existir
- Apenas `WMS_ADMIN`

**Execução automática (Spring @Scheduled a cada minuto):**
- Verifica agendamentos cujo cron está vencido (campo `nextExecutionAt ≤ now`)
- Gera o relatório chamando internamente o `ReportService`
- Atualiza `lastExecutedAt` e calcula `nextExecutionAt`
- Loga `INFO` com: `"Relatório {type} gerado para tenant {id} — {N} linhas"` (v1: sem envio por e-mail)

### Cálculo de `saldoInicial` para Ficha de Estoque

`saldoInicial = quantityAvailable_atual - Σ(entradas no período) + Σ(saídas no período)`

Obtido consultando `StockMovement` com `createdAt BETWEEN dataInicio AND dataFim` agrupado por produto/localização.
