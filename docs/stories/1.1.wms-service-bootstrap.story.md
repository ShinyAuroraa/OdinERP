# Story 1.1: WMS Service Bootstrap

## Status

Done

## Story

**As a** engenheiro de DevOps,
**I want** configurar a estrutura base do microserviço WMS com Dockerfile, manifests Kubernetes, pipeline CI/CD e conexões com PostgreSQL, Redis e Elasticsearch,
**so that** o time de desenvolvimento tenha um ambiente production-ready para implementar todas as funcionalidades do WMS sobre uma base sólida e padronizada.

## Complexity

**Estimativa:** M (Medium) · ~5-8 dias de desenvolvimento
[Fonte: EPIC-WMS-001 Story Summary]

## Dependencies

**Pré-requisitos para iniciar esta story:**
- Repositório Git criado e acessível
- Conta GitHub com permissão para configurar Actions Secrets
- Cluster Kubernetes (staging) acessível via `kubectl`
- Registry de imagens disponível (GitHub Container Registry)
- Credenciais de acesso ao PostgreSQL, Redis e Elasticsearch de staging

**Stories desbloqueadas após esta:**
- Story 1.2 — Schema & Domain Model (depende do projeto Spring Boot base)
- Story 1.3 — Autenticação & RBAC Keycloak (depende da estrutura Spring Security base)
- Story 1.4 — Observabilidade (depende do pipeline CI/CD e K8s manifests)

**Nota:** Story 1.3 (Keycloak) assumirá e substituirá o `SecurityFilterChain` permissivo criado nesta story.

## Out of Scope

> Itens explicitamente **não implementados** nesta story:

- ❌ Domain model, entities JPA ou schema de banco de dados → Story 1.2
- ❌ Autenticação Keycloak ou RBAC → Story 1.3
- ❌ Prometheus metrics customizadas, dashboards Grafana, Jaeger tracing → Story 1.4
- ❌ Qualquer lógica de negócio WMS (armazéns, estoque, etc.) → Waves 2+
- ❌ Frontend / App Android → Waves futuras
- ❌ Istio service mesh configuration → Story 1.4

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Testcontainers Docker indisponível no CI runner | Baixa | Alto | Usar `runs-on: ubuntu-latest` (tem Docker); fallback: `services` do GitHub Actions |
| Imagem Docker exceder 200MB | Média | Médio | Multi-stage build + JRE slim; verificar com `docker images` no CI |
| Secrets K8s não configurados no staging | Média | Alto | Documentar variáveis necessárias no README-WMS.md; usar `secretKeyRef` em vez de valores inline |
| Spring Boot 3 + Java 21 incompatibilidade de deps | Baixa | Médio | Usar Spring Boot BOM para gerenciar versões; testar localmente antes do push |
| Security permissiva esquecida em produção | Média | Crítico | Comentário `// TEMPORÁRIO` + task explícita na Story 1.3 de remoção |

## Acceptance Criteria

1. **Projeto Spring Boot 3 inicializado** — Projeto Java 21 + Spring Boot 3.x criado com estrutura de pacotes padrão (`com.odin.wms`), com dependências base: Spring Web, Spring Data JPA, Spring Security, Spring Actuator, Flyway, Lombok, MapStruct.

2. **Dockerfile multi-stage funcional** — Dockerfile com estágio de build (Maven) e estágio de runtime (JRE 21 slim), imagem final < 200MB, usuário não-root, health check configurado.

3. **Manifests Kubernetes completos** — `Deployment`, `Service`, `ConfigMap`, `Secret` (referências), `HorizontalPodAutoscaler` e `Ingress` criados no diretório `k8s/wms/`.

4. **Pipeline CI/CD funcionando** — GitHub Actions pipeline com stages: `test → build → scan → push → deploy`. ArgoCD Application configurado apontando para `k8s/wms/`.

5. **Conexão PostgreSQL configurada** — DataSource Spring Boot configurado via `application.yml`, Flyway migrations habilitadas (`V1__init.sql` vazio para baseline), connection pool HikariCP configurado.

6. **Conexão Redis configurada** — Spring Data Redis com Lettuce client configurado, `RedisTemplate<String, Object>` bean registrado, health check no Actuator.

7. **Conexão Elasticsearch configurada** — Spring Data Elasticsearch configurado, client `ElasticsearchClient` bean registrado, health check no Actuator.

8. **Health checks operacionais** — Endpoint `/actuator/health` retornando status UP para todos os componentes (DB, Redis, Elasticsearch). Liveness e Readiness probes configurados no K8s.

9. **Graceful shutdown implementado** — Aplicação aguarda requisições em andamento antes de encerrar (timeout: 30s), configurado via `spring.lifecycle.timeout-per-shutdown-phase`.

10. **Testes de integração passando** — Testes com Testcontainers para PostgreSQL, Redis e Elasticsearch. Pipeline CI executa testes antes do build da imagem.

### Padrão de Conexão Redis

```java
// RedisConfig.java
@Configuration
public class RedisConfig {
    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(new Jackson2JsonRedisSerializer<>(Object.class));
        template.setHashKeySerializer(new StringRedisSerializer());
        template.setHashValueSerializer(new Jackson2JsonRedisSerializer<>(Object.class));
        return template;
    }
}
```

### Padrão Testcontainers

```java
// AbstractIntegrationTest.java
@SpringBootTest
@Testcontainers
@ActiveProfiles("test")
public abstract class AbstractIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:17-alpine")
        .withDatabaseName("wms_test")
        .withUsername("wms_test")
        .withPassword("wms_test");

    @Container
    static GenericContainer<?> redis = new GenericContainer<>("redis:7-alpine")
        .withExposedPorts(6379);

    @Container
    static ElasticsearchContainer elasticsearch =
        new ElasticsearchContainer("docker.elastic.co/elasticsearch/elasticsearch:8.11.0")
            .withEnv("discovery.type", "single-node")
            .withEnv("xpack.security.enabled", "false");

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
        registry.add("spring.data.redis.url",
            () -> "redis://localhost:" + redis.getMappedPort(6379));
        registry.add("spring.elasticsearch.uris",
            () -> "http://localhost:" + elasticsearch.getMappedPort(9200));
    }
}
```

### Completion Notes List

- **ElasticsearchConfig:** Usado `ElasticsearchConfiguration` (Spring Data Elasticsearch) em vez de configuração manual — mais idiomático com Spring Boot 3.
- **RedisConfig:** Usado `GenericJackson2JsonRedisSerializer` com `ObjectMapper` customizado (com `JavaTimeModule` e `DefaultTyping`) em vez do deprecated `Jackson2JsonRedisSerializer`.
- **SecurityConfig:** Configuração temporária permissiva conforme especificado — comentário javadoc referencia Story 1.3 explicitamente.
- **Testcontainers:** Containers iniciados estaticamente (`static {}`) para reutilização entre testes da mesma JVM, reduzindo tempo de CI.
- **pom.xml:** Adicionado `flyway-database-postgresql` separado (Spring Boot 3.2+ requer módulo específico para PostgreSQL 17).
- **CI/CD:** Pipeline inclui `paths` filter para executar apenas em mudanças do wms-service, economizando minutos de CI.
- **K8s secret.yaml:** Valores marcados como `MANAGED_BY_VAULT` — nunca commitados com valores reais.
